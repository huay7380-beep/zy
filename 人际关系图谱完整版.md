人际关系图谱（完整版）
一、文档概述
1.1 设计目标
本图谱旨在全面、动态地描述人与人之间的社会关系，为上层的事件图谱和策略引擎提供标准化的数据基础。它结合了中国社会特有的关系文化，并严格遵循“数据与策略分离”的原则。

1.2 核心改进
完整性：新增“自我”及多种中国特色关系类型，总数达77种。

标准化：引入 type_code 和 dimension 字段，支持程序化引用。

动态性：新增 event_interface 字段，明确关系与事件的交互规则。

策略性：恢复 active_signals、default_trust、disclosure_depth，为策略引擎提供精细指令。

演化性：扩展生命周期阶段至12个，补充复合关系场景切换规则。

1.3 字段说明
每个关系类型遵循以下结构：

字段	类型	描述
type_code	string	英文编码，供系统引用（如 spouse）
relationship_type	string	中文显示名称
dimension	integer	所属维度（0-5）
definition	string	关系定义
subtypes	array	可选，子类型提示，用于更精细的策略
base_weight	float	基础权重（0-1）
default_strategy	object	默认交互策略，包含：
• communication_style: 沟通风格
• emotional_support: 情感支持方式
• sensitive_topics: 敏感话题列表
• maintenance_frequency: 维护频率（枚举）
• conflict_tendency: 冲突倾向（枚举）
• active_signals: 主动释放的正向信号类型
• default_trust: 默认信任度（high/medium/low）
• disclosure_depth: 信息分享深度（private/partial/public）
• llm_role: 策略引擎角色提示
• interaction_guideline: 具体交互指南
event_interface	object	与事件图谱的接口，包含：
• typical_events: 常见事件类型列表
• high_impact_positive: 高影响正向事件
• high_impact_negative: 高影响负向事件
• transition_triggers: 关系类型转换触发条件
weight_overrides	array	可选，特殊场景下的权重覆盖规则

1.5 复合关系与动态调整规则
场景切换：当用户处于工作场景时，优先使用维度4（社会与契约）策略；家庭场景优先维度1或2；社交场景优先维度2或3。

季节性事件：春节、中秋、生日等自动触发维护提醒，权重临时提升0.05。

关系转换：转换时，新类型继承旧类型的事件历史和情感趋势，但权重重置为新类型的 base_weight。

三角关系检测：当A与B的关系因C的事件受影响时，记录间接影响链。

自定义关系类型添加方案
为了满足用户对未覆盖关系的扩展需求，同时保证与预设关系类型的数据结构一致、策略引擎无缝集成，我们设计以下自定义关系添加方案。该方案允许用户在个人图谱中新增关系类型，且不影响系统预设的 77 种基础关系。

一、设计原则
数据结构一致：自定义关系类型必须采用与预设关系完全相同的字段结构，确保策略引擎无需区分即可处理。

编码唯一性：自定义关系的 type_code 需自动生成并保证唯一，不与预设及现有自定义编码冲突。

隔离性：自定义关系仅对当前用户生效，不影响全局基础库。

可管理性：用户可增删改自定义关系，系统记录其来源为 user_defined。

二、字段要求
用户在添加自定义关系时，必须提供以下信息（与预设关系字段对应）：

字段	类型	必填	说明
relationship_type	string	是	中文显示名称，如“棋友”
dimension	integer	是	所属维度（0-5），用户需从预定义维度中选择
definition	string	是	关系定义描述
base_weight	float	是	基础权重（0-1）
default_strategy	object	是	包含以下子字段：
• communication_style: 沟通风格（字符串）
• emotional_support: 情感支持方式（字符串）
• sensitive_topics: 敏感话题数组
• maintenance_frequency: 维护频率（从枚举中选择）
• conflict_tendency: 冲突倾向（从枚举中选择）
• active_signals: 主动信号数组
• default_trust: 默认信任度（high/medium/low）
• disclosure_depth: 分享深度（private/partial/public）
• llm_role: 角色提示（字符串）
• interaction_guideline: 交互指南（字符串）
event_interface	object	推荐	包含 typical_events, high_impact_positive, high_impact_negative, transition_triggers，若用户不填可留空或使用默认空值
可选字段：

subtypes: 子类型提示数组

weight_overrides: 权重覆盖规则数组（一般用户可不填）

三、编码生成规则
系统自动为自定义关系生成唯一 type_code，规则如下：

前缀：custom_

核心部分：取用户输入的中文名称的拼音全拼（小写），若名称中包含非中文字符则原样保留。为保证唯一性，可在末尾添加短随机字符串（如 _xxx）或基于时间戳的哈希。

示例：用户添加“棋友”，生成的 type_code 可能为 custom_qiyou，若已存在则自动加序号 custom_qiyou_1。

注意：编码生成逻辑应保证不与预设类型编码（如 spouse, parent）冲突，因预设均为英文单词，而自定义带 custom_ 前缀，天然隔离。

四、存储结构
自定义关系类型应存储在用户个人数据中，与预设基础库分离。推荐两种存储方式：

方式一：用户专属 JSON 文件
在用户个人图谱数据中增加 custom_relationship_types 数组，结构与基础 relationship_types 一致，并额外添加字段 created_at 和 updated_at。

json
{
  "user_id": "u123",
  "custom_relationship_types": [
    {
      "type_code": "custom_qiyou",
      "relationship_type": "棋友",
      "dimension": 2,
      "definition": "经常一起下棋的朋友，棋艺切磋，棋外也有交流",
      "base_weight": 0.45,
      "default_strategy": { ... },
      "event_interface": { ... },
      "created_at": "2026-02-21T10:00:00Z",
      "updated_at": "2026-02-21T10:00:00Z"
    }
  ]
}
方式二：合并存储并标记
将自定义关系与预设关系合并存储在用户的关系类型库中，但增加 source 字段标识 user_defined。这种方式便于统一查询，但需在初始化时确保基础库不被污染。

json
{
  "relationship_types": [
    // ... 预设的 77 种关系（source: "system"）
    {
      "type_code": "custom_qiyou",
      "relationship_type": "棋友",
      "source": "user_defined",
      "created_at": "...",
      "updated_at": "..."
    }
  ]
}
推荐使用方式一，保持基础库的纯净，查询时合并两套数据即可。

五、策略引擎集成
策略引擎在加载关系策略时，应合并基础库与用户自定义库，合并规则：

若自定义关系的 type_code 与基础库冲突（极罕见），以用户自定义为准（覆盖）。

策略引擎无需区分来源，直接使用合并后的关系类型定义。

动态调整规则（如季节性事件、关系转换）同样适用于自定义关系。

六、用户界面建议
为用户提供简单的添加表单，引导填写必要字段，并自动生成编码。建议字段如下：

关系名称（必填）

所属维度（下拉选择：0-自我、1-血缘与亲缘、2-情感与亲密、3-地缘与成长、4-社会与契约、5-负向与博弈）

定义描述（文本域）

基础权重（滑块 0-1）

沟通风格（输入或从预设列表选择）

情感支持方式（输入或从预设列表选择）

敏感话题（输入，支持多个）

维护频率（下拉选择预设枚举）

冲突倾向（下拉选择预设枚举）

主动信号（输入，支持多个）

默认信任度（单选：高/中/低）

分享深度（单选：私密/部分/公开）

LLM角色提示（输入）

交互指南（文本域）

常见事件（输入，支持多个，可选）

对于不熟悉策略细节的用户，可提供“从类似关系复制”功能，例如以“普通朋友”为模板快速生成。

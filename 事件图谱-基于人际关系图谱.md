事件图谱完整构建方案（优化版）

> **数据源对齐**：本方案基于 `方案/人际关系图谱/人际关系图谱完整版.md`（77种关系类型）构建，事件类型枚举、规则引擎及动态更新机制均与人际关系图谱的 `event_interface` 字段保持一致。策略层面以 `策略关系定义.md`（D1-D5五维策略框架）为执行标准。

一、设计目标
动态演化：通过真实互动事件驱动人物关系权重、阶段和情感趋势实时变化。

可视化分层：人物图谱位于平面（Z=0），事件节点悬浮于上层（Z>0），点击事件后相关人物按事件权重动态聚合。

与策略引擎联动：事件更新触发关系变更，策略引擎据此生成更精准的交互建议。

隐私可控：事件数据继承人物关系的可见性控制，支持独立设置。

敏感字段处理策略
默认脱敏：存储前对 context 和 evidence 进行自动脱敏（姓名、电话、地址等）。
本地保留：支持仅本地存储原文，云端仅保留事件结构化元数据。
自动过期：支持设置 context 与 evidence 过期清空（如 30 天）。
加密存储：对敏感字段进行加密，仅用户密钥可解密。
明确授权：采集外部数据需用户明确同意并说明用途。
2. 参与边（PARTICIPATED_IN）
连接人物与事件，记录角色和影响程度。

连接事件与关系边，记录具体影响细节，便于追溯和回放。

  "relationship_id": "string",
  "direction": "enum (from_to / to_from / both)",
  "old_weight": "float",
  "new_weight": "float",
  "delta": "float (new - old)",
  "old_phase": "string",

  "rule_id": "string (触发规则ID)",
  "description": "string (影响原因，如'因帮助事件+0.1')",
  "timestamp": "datetime"
}
4. 关系边的增强
在原有关系边中增加近期事件引用，便于快速获取上下文。

    "event_id": "event_123",
    "timestamp": "datetime",
    "type": "string",
    "sentiment_score": "float"
  }
]
同时增加健康度指标，由策略引擎定期计算：

json
"health_score": "float (0-1，基于近期事件平均情感得分、交互频率等)"
三、关联方式
人物 ↔ 事件：通过 PARTICIPATED_IN 边双向关联。

事件 ↔ 关系：通过 AFFECTS 边连接，一个事件可影响多条关系。

关系 ↔ 事件：通过关系边的 recent_interactions 数组反向索引，或通过 (relationship)<-[:AFFECTS]-(event) 查询历史影响。

图数据库示例（Cypher）：

cypher
// 查询某人参与的所有事件
MATCH (p:Person {person_id: 'user123'})-[r:PARTICIPATED_IN]->(e:Event) RETURN e

// 查询某事件影响的所有关系
MATCH (e:Event {event_id: 'event456'})-[a:AFFECTS]->(rel:RELATES_TO) RETURN rel

// 查询某关系的历史影响事件
MATCH (rel:RELATES_TO {relationship_id: 'rel789'})<-[a:AFFECTS]-(e:Event) RETURN e
四、动态更新机制
1. 事件触发更新流程
当新事件被记录（手动或自动检测），执行以下步骤：

识别参与者：从事件描述提取或用户指定人物ID列表。

确定受影响的关系：对每一对参与者，查询两者之间是否存在关系边。若存在，进入更新；若不存在，生成候选关系并由用户确认后创建。自动识别事件进入待确认队列，确认后写入图谱并触发更新。

计算事件影响：

根据事件类型、情感得分、重要性，结合关系当前权重和阶段，应用预定义规则计算权重调整量 delta。

规则引擎支持配置化（见下文）。

更新关系属性：

weights.from_to（用户对对方）和 weights.to_from（对方对用户）可分别调整（需区分角色）。

phase.current：若累积影响超过阈值，变更阶段（如 stable→conflict）。

sentiment_trend：基于本次事件情感得分和历史趋势重新计算（up/down/stable）。

recent_interactions：添加事件摘要，移除最旧记录。

记录影响边：创建 AFFECTS 边，存储详细影响信息。

触发策略引擎：通知策略引擎相关关系已变更，用于下次交互建议。

2. 事件影响规则引擎
规则可配置化，支持用户自定义。每条规则包含：

适用条件：事件类型、情感得分范围、关系类型、当前阶段等。

权重调整公式：可基于 importance、sentiment_score、role 等计算，并按 event_type 分段。

阶段变更阈值：累积影响达到阈值时切换阶段。

优先级与冲突：用户手动标记优先于规则引擎，规则冲突按优先级处理；优先级相同时采用平均或最大变动法，并记录冲突解决方式。
单次更新上限：每次事件触发权重调整设置绝对值上限（如 ±0.2）。
冷却期机制：同一关系在 24 小时内限制更新次数，超出则延迟或合并处理。
速率限制：限制权重变化速率（如每小时不超过 0.1）。
累积阈值：多次微小同向影响合并，达到阈值后再更新。
手动锁定：用户手动调整的权重在一定时间内不受自动规则影响（如 7 天）。

3. 长期衰减与事件综合
自然衰减：每周无事件时，关系权重减 0.01（可配置），通过定时任务执行。

健康度指标：由策略引擎定期计算，公式示例：

text
health_score = 0.4 * avg_sentiment + 0.3 * (1 - decay_rate) + 0.2 * event_density + 0.1 * user_correction
其中 avg_sentiment 为最近10次事件的平均情感得分，decay_rate = min(1, days_since_last_event / 30)，event_density 为近期事件数量与期望值的比率，user_correction 为用户手动打分。

4. 事件链与复合事件
相关联事件可通过 parent_event_id 归并为复合事件，用于简化可视化与因果展示。

五、可视化支持
1. 分层布局
人物节点：固定于 Z=0 平面，布局采用力导向图或预计算位置。

事件节点：根据 visual.layer 分配 Z 轴高度（如 1.0~3.0），XY 坐标可根据参与者位置取平均，或由力导向图自然分布。

连接线：人物-事件使用垂直或斜线（半透明），事件-关系影响线（AFFECTS）默认隐藏，点击事件后高亮显示。

2. 点击聚合逻辑
用户点击事件节点时，前端获取该事件的所有参与者（PARTICIPATED_IN 边）。

聚合规则：

以事件节点为中心，将相关人物拉近到半径 2.0 的圆环上。

人物在圆环上的角度按 impact_factor 降序排列（影响越大越靠近事件方向）。

同时高亮显示受该事件影响的关系边（AFFECTS 边），并标注权重变化。

若事件设置了 aggregation_key，可按该字段将人物分组显示（如按角色分组）。
支持按角色分组聚合与按影响程度分组的视图切换。

3. 持续事件与简化实现
持续事件可在开始与结束生成节点，中间状态通过连线或动画呈现。
默认只展示 completed 的核心节点，planned/ongoing 可延后展示或折叠。
direction 字段用于权重更新，初期可不参与可视化渲染。

4. 时间轴播放
支持按时间顺序播放事件，人物关系动态演变（权重和颜色变化），可观察关系演化路径。

六、隐私与权限控制
事件节点的 visibility 默认继承自参与人物关系中的最高隐私级别，例如：若事件涉及一个仅自己可见的“敌人”，则整个事件默认 private。

用户可单独修改事件的可见性，覆盖继承规则。

查询时，系统根据当前用户权限过滤事件和关系，确保隐私安全。

七、与策略引擎联动
策略引擎在生成交互建议时，从图谱层获取：

关系最新属性（weights, phase, health_score）

近期事件摘要（recent_interactions）

受影响的历史事件详情（通过 AFFECTS 边查询）

据此调整建议：

若 phase 为 conflict，建议采用“包容”策略，避免敏感话题。

若近期有 help 事件，建议主动表达感谢。

若 health_score 过低，建议增加互动频率。

若 parent_circle 关系中发生 conflict 事件（孩子间矛盾），建议冷静处理，优先以孩子利益为纽带化解家长分歧。

若 property_staff 关系中 property_service 事件积压且情感得分持续为负，建议走正式投诉渠道并保留记录。

若 regular_merchant 关系权重因长期无消费自然衰减至临界值，提醒用户该关系可能转为 inactive。

| gift（店主主动赠品/让利） | +0.5~+0.7 | +0.05 | growing→stable | 人情味建立信号 |
| help（日常小帮忙） | +0.3~+0.5 | +0.04 | — | 超出商业关系的善意 |
| conflict（服务质量投诉） | -0.3~-0.5 | -0.05 | stable→stagnant | 可通过沟通修复 |
| rejection（涨价/拒绝优惠） | -0.1~-0.2 | -0.02 | — | 轻度负面，属商业正常行为 |

特殊规则：
- 连续30天未消费，权重自然衰减 0.01/周（比通用衰减略快，因关系基础为商业行为）
- 店铺搬迁/关店时，关系自动进入 `inactive`，90天后转为 `terminated`

九、扩展性考虑
1. 自动事件检测
> 自动事件检测的完整逻辑详见「十二、事件处理管道（EPP）」章节。
> 硬件采集能力随通用模型迭代持续提升，本文件只定义图谱层的接收规范，不绑定具体采集方案。

2. 事件类型自定义
允许用户新增事件类型，并配置影响规则（通过规则引擎界面）。

3. 批量导入
支持从日历、社交媒体导入事件，自动匹配人物（通过名称、时间等模糊匹配）。

九、性能优化建议
索引：为 Event.timestamp、Event.event_type、PARTICIPATED_IN.person_id 创建索引。
复合索引：建议 (AFFECTS.relationship_id, AFFECTS.timestamp)。

限流：recent_interactions 限制长度（如 20 条），避免属性过大。

缓存：策略引擎可缓存常用关系属性，减少实时查询压力。
事件分页：默认仅加载最近 30 天事件，支持分页。
归档策略：超过 1 年事件与影响边移入冷存储，保留聚合统计。
分区存储：按时间分区存储事件节点，AFFECTS 边支持 TTL 自动清理。
查询优化：关系历史事件默认返回最近 N 条，支持分页。
缓存策略：健康度异步计算，每小时更新一次。

## 十二、事件处理管道（EPP）

> **核心前提**：事件的本质是「现实世界中发生的、对人际关系有影响的事实单元」。
> 对话、消息、语音记录都只是事件的**信号载体**，不等于事件本身。
> 系统的职责是从每天海量杂乱信号中**筛选**出值得记录的事件，而非将所有内容强行并入图谱。

### 12.1 事件判定的第一性原理

在任何技术处理之前，先明确"值得记录的事件"的判定标准：

| 判定维度 | 说明 | 不入库的反例 |
|---------|------|------------|
| **有真实参与者** | 必须涉及关系图谱中已记录或可识别的人 | "明天天气真好" |
| **有可感知的影响** | 事件结束后，至少一条关系边权重/阶段/情感应当改变，或存在改变可能 | 日常打招呼（除非累积触发） |
| **有时间锚点** | 可确认发生时间或时间范围，而非泛指 | "以前我们吵过架"（无时间） |
| **有因果指向** | 能对应到至少一个 `event_type_code` | 无法分类的噪音 |

**满足以上全部4条 → 候选事件。不满足 → 忽略或归入原始日志备查。**

---

### 12.2 管道总览

```
[原始信号流]  文字对话 / 语音转写 / 手动输入 / 日历导入
       │
       ▼
【Stage 0：入口过滤】
  去噪 + 参与者识别 + 时间标注
  → 输出：带时间戳的「对话片段」序列
       │
       ▼
【Stage 1：场景切分】
  一天的对话 → 多个「对话场景」（同参与者 + 连续时间段）
  → 输出：场景列表
       │
       ▼
【Stage 2：事件候选提取】
  每个场景 → 0至多个「候选事件」（含置信度）
  → 宁可漏报，不可滥报
       │
       ▼
【Stage 3：事件分级与归并】
  候选事件 → 主事件 P1 / 次事件 P2 / 线索 P3
  堆叠消解 + 嵌套归并 + 线索追加
  → 输出：结构化事件树
       │
       ▼
【Stage 4：确认与入库】
  高置信度 → 自动入库（触发规则引擎）
  中置信度 → 推送用户确认队列
  低置信度 → 静默丢弃（原始日志保留）
```

---

### 12.3 Stage 0：入口过滤

**目标**：把原始输入标准化为「带时间戳的对话片段」。

| 处理项 | 规则 |
|-------|------|
| 去噪 | 系统消息（入群通知、红包提示等）直接过滤 |
| 参与者标注 | 从消息发送者 / 用户手动关联 中提取 `person_id` |
| 未知参与者 | 标为 `unknown_participant`，不阻塞流程，但不触发关系权重更新 |
| 时间标注 | 每条消息保留原始时间戳，缺失时用前后消息插值估算 |
| 多源去重 | 同一消息被多个渠道采集：以最早时间戳为准，合并内容 |

输出片段结构：

```json
{
  "fragment_id": "uuid",
  "source": "enum (chat_log / voice_transcript / manual / calendar)",
  "participants": [
    { "person_id": "string", "confidence": "float" }
  ],
  "messages": [
    { "seq": 1, "at": "datetime", "sender_id": "string", "text": "string" }
  ],
  "time_range": { "start": "datetime", "end": "datetime" }
}
```

---

### 12.4 Stage 1：场景切分

**目标**：将一天内的所有片段切分为独立的「对话场景」。

场景是一个有完整上下文的交互单元：同一批参与者 + 同一连续时间段 + 同一渠道。

**切分触发条件**（满足任意一条即切分）：

```
1. 时间间隔 > 60 分钟
2. 参与者集合变化超过 50%
3. 渠道/平台切换（如从群聊切换到私聊）
4. 用户手动标记场景边界
5. 单场景时长超过 4 小时（强制切分）
```

场景结构：

```json
{
  "scene_id": "uuid",
  "participants": ["person_id"],
  "time_range": { "start": "datetime", "end": "datetime" },
  "fragment_ids": ["fragment_id"],
  "scene_type": "enum (private_chat / group_chat / voice_call / mixed)",
  "raw_sentiment_by_minute": "float[]"
}
```

---

### 12.5 Stage 2：事件候选提取

**目标**：从每个对话场景中提取 0 至多个候选事件。

#### 12.5.1 三类提取触发方式

**A. 语义触发**（主要方式，由 LLM 分析场景文本）

| 语义模式 | 候选类型 | 最低置信度 |
|---------|---------|----------|
| 明确帮助/求助行为且有结果 | `help` | 0.60 |
| 争吵/指责/对抗性语言 | `conflict` | 0.50 |
| 道歉语言 + 对方接受信号 | `apology` | 0.65 |
| 表白/确认恋爱关系 | `confession` / `dating` | 0.80 |
| 背叛/出卖/泄密的明确陈述 | `betrayal` / `information_leak` | 0.75 |
| 重大人生变化（升职/离职/结婚/病重等） | `milestone` / `job_transition` / `family_crisis` | 0.70 |
| 金钱/合同/法律相关 | `payment_transaction` / `contract_signing` / `legal_action` | 0.80 |
| 持续性负面情绪 + 索取行为 | `exploitation` | 0.70 |

**B. 累积触发**（跨场景统计，单次不触发）

| 累积条件 | 触发事件 | 等级 |
|---------|---------|------|
| 同一关系 7 天内 `conflict` ≥ 3 次 | 生成 `conflict_pattern`（关系冲突化趋势） | P1 |
| 同一关系 30 天内无任何互动 | 生成 `relationship_fading`（关系淡化） | P2 |
| 连续 5 次单方发起，对方持续低情感回应 | 生成 `one_sided_effort`（单向付出） | P2 |

**C. 线索追加触发**（已有事件的延续信号）

当识别到某段对话与已有未关闭事件高度相关时，**不创建新事件节点，而是向现有事件追加线索**：

```
匹配条件：
1. 参与者集合与已有事件重叠 ≥ 50%
2. 时间距已有事件 ≤ 该事件类型的生命周期上限（见12.6节）
3. 语义与已有事件的 event_type_code 相关
→ 操作：追加 clue 到已有事件，不新建节点
```

#### 12.5.2 候选事件结构

```json
{
  "candidate_id": "uuid",
  "scene_id": "string",
  "event_type_code": "string",
  "confidence": "float",
  "participants": [
    { "person_id": "string", "role": "enum (initiator/target/witness/helper/victim/other)", "confidence": "float" }
  ],
  "time_anchor": {
    "estimated_start": "datetime",
    "estimated_end": "datetime",
    "time_confidence": "float"
  },
  "evidence_clues": [
    {
      "clue_id": "uuid",
      "clue_type": "enum (utterance / sentiment_spike / keyword / behavior_pattern)",
      "content": "string (脱敏后的原始依据)",
      "at": "datetime",
      "weight": "float"
    }
  ],
  "sentiment_score": "float",
  "extraction_method": "enum (semantic / accumulation / clue_append)",
  "status": "enum (pending / confirmed / rejected / merged)"
}
```

---

### 12.6 Stage 3：事件分级与归并

#### 12.6.1 事件三级体系

| 等级 | 定义 | 典型类型 | 图谱可见性 |
|------|------|---------|----------|
| **P1 主事件** | 对关系产生不可逆或重大影响，可独立存在 | `betrayal` / `breakup` / `milestone` / `legal_action` / `confession` / `family_crisis` | 始终显示 |
| **P2 次事件** | 对关系产生可量化但可恢复的影响，可独立存在 | `conflict` / `help` / `celebration` / `apology` / `trust_building` / `contract_signing` | 默认显示，可折叠 |
| **P3 线索事件** | 单独不构成事件，作为 P1/P2 的证据或延续 | `regular_patronage` / `support` / `gift` / `service_request` | 附着在父节点，不独立显示 |

**等级提升规则**：

```
P3 → P2：同类型 P3 在 30 天内累积 ≥ 5 次，且情感方向一致
P2 → P1：P2 触发阶段变更（phase change），或 importance ≥ 0.85
```

#### 12.6.2 事件嵌套结构

主事件包含子事件，子事件包含线索——形成树状结构：

```
P1 主事件（betrayal）
├── P2 子事件（conflict × 3，时间在 betrayal 之前） ← 前因
├── P2 子事件（breakup）← 后果
│   └── P3 线索（分手后的最后一次对话记录）
└── 线索集合
    ├── clue: 第一次发现异常的对话片段
    ├── clue: 对方承认的语句（脱敏）
    └── clue: 用户手动附加的证据
```

子事件归入父事件的条件：

```json
{
  "child_event_conditions": [
    "时间范围被父事件包含，或紧邻（前7天/后30天内）",
    "参与者与父事件有交集",
    "event_type_code 在父事件的 related_types 列表中"
  ],
  "max_nesting_depth": 3,
  "auto_nest_confidence_threshold": 0.70,
  "related_types_map": {
    "betrayal":    ["conflict", "information_leak", "breakup", "legal_action"],
    "breakup":     ["conflict", "apology", "reconciliation"],
    "milestone":   ["celebration", "family_gathering", "agreement"],
    "legal_action":["conflict", "threat", "defamation", "agreement"]
  }
}
```

#### 12.6.3 时间交集与堆叠消解

| 情况 | 处理方式 |
|------|---------|
| 同参与者、同时间窗口、不同类型 | 高等级为主事件，低等级降为子事件 |
| 同参与者、同时间窗口、同类型 | 合并为一个事件，合并线索，置信度取最大值 |
| 不同参与者、时间窗口重叠 | 保持独立，添加 `related_event_ids` 互相引用 |
| 同类型事件在冷却期内再次触发 | 追加线索到已有事件，不新建节点 |

**事件类型生命周期与冷却期**：

```json
{
  "betrayal":          { "max_duration_days": 365, "cooldown_hours": 168 },
  "conflict":          { "max_duration_days": 30,  "cooldown_hours": 24  },
  "apology":           { "max_duration_days": 7,   "cooldown_hours": 12  },
  "help":              { "max_duration_days": 3,   "cooldown_hours": 6   },
  "milestone":         { "max_duration_days": 30,  "cooldown_hours": 72  },
  "legal_action":      { "max_duration_days": 730, "cooldown_hours": 168 },
  "regular_patronage": { "max_duration_days": 1,   "cooldown_hours": 4   },
  "trust_building":    { "max_duration_days": 14,  "cooldown_hours": 48  }
}
```

---

### 12.7 Stage 4：确认与入库

#### 12.7.1 置信度阈值

| 置信度 | 处理方式 |
|-------|---------|
| ≥ 0.85 | 自动入库（`source: auto_detected`），同步触发规则引擎 |
| 0.55 ~ 0.85 | 推送用户确认队列（24小时内未响应自动丢弃） |
| < 0.55 | 静默丢弃（原始线索保留在日志中，可回溯） |

> **特例**：P1 级事件无论置信度高低，一律进入确认队列，不自动入库。

#### 12.7.2 入库节点补充字段

在现有 Event 节点数据模型（第二章）基础上补充：

```json
{
  "event_level":            "enum (P1 / P2 / P3)",
  "parent_event_id":        "string (P3 必填)",
  "child_event_ids":        ["string"],
  "related_event_ids":      ["string (时间交集关联)"],
  "clues": [
    {
      "clue_id":   "uuid",
      "clue_type": "enum (utterance / sentiment_spike / keyword / behavior_pattern / user_added)",
      "content":   "string (脱敏)",
      "at":        "datetime",
      "source":    "enum (auto / user_manual)",
      "weight":    "float"
    }
  ],
  "accumulation_count":     "int (P3 被线索追加的次数)",
  "pipeline_stage":         "enum (candidate / confirmed / merged / archived)",
  "extraction_confidence":  "float",
  "dedup_key":              "string (去重哈希)"
}
```

### 12.8 图谱展示逻辑（按等级向外辐射）

#### 12.8.1 展示层级

```
中心点（用户自身）
    │
    ├── 第1层：P1 主事件（importance ≥ 0.8，始终可见）
    │         节点大：与 importance 正比
    │         颜色：深红（负向）/ 深蓝（正向）/ 深紫（里程碑）
    │
    ├── 第2层：P2 次事件（importance 0.4~0.8，默认展开）
    │         中等节点，颜色中度色阶
    │         可折叠为缩略圆环
    │
    ├── 第3层：P3 线索（附着在父节点上的小圆点，默认折叠）
    │         点击父节点后展开，以时间轴形式显示
    │
    └── 第4层：累积趋势元事件（不占独立节点）
              以渐变弧线悬浮在涉及关系的边上
```

#### 12.8.2 节点视觉编码

```json
{
  "node_size_formula": "base_size × importance × (1 + 0.1 × child_event_count)",
  "color_rules": {
    "positive_P1": "#1A3A8F",
    "negative_P1": "#8B0000",
    "milestone_P1": "#4B0082",
    "positive_P2": "#4A90D9",
    "negative_P2": "#E05C5C",
    "P3_clue":     "#AAAAAA"
  },
  "edge_rules": {
    "parent_child":   "实线，粗细与 importance 相关",
    "related_events": "虚线，表示时间交集关联",
    "clue_to_event":  "细虚线，点击才显示"
  }
}
```

#### 12.8.3 时间轴模式

切换到时间轴视图时：
- P1 事件显示为**全宽高亮条**
- P2 事件显示为**半宽标记点**
- P3 线索显示为**细条，附着在父事件条的下方**
- 支持按人物/按事件类型筛选
- 支持缩放（天/周/月/年）

十一、总结
本方案基于人际关系图谱的77种关系类型构建，事件类型枚举从基硡16类扩展为覆盖6大维度的分类体系，规则引擎包含通用规则和新增关系类型（家长圈伙伴、物业/社区工作人员、常驻商家/熟识店主）的专属规则。方案细化了数据模型、规则引擎、可视化交互及隐私控制，确保了事件图谱与人物图谱和策略引擎的三方无缝集成，并满足3D分层展示和点击聚合的动态需求。后续可基于此方案逐步实现自动事件检测、健康度预警等智能功能，构建一个真正能“感知”人际互动并辅助决策的社交系统。

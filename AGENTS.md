# Skill Creation Default Gate

- Before creating any new Skill, plugin-facing workflow, or mandatory capability guide, first perform a reuse search. Check the currently available Skills, installed plugins/connectors, local repository modules, `capability-upgrade-registry`, CLI-Anything or other known local tool catalogs, and reputable external software or open-source projects when network research is appropriate.
- A new Skill is allowed only after the reuse search shows that existing Skills/software cannot satisfy the goal, cannot be safely adapted, or would cost more to adapt than a small purpose-built Skill. Record the decision in a `skill_creation_gate.v1` artifact or an equivalent task note before creating files.
- The default decision order is: reuse existing Skill/tool -> wrap or adapt existing capability -> install/request an existing plugin/connector when explicitly requested and available -> create a new Skill only as the last step.
- The gate record must include the requested goal, searched sources, candidates found, why each candidate was accepted or rejected, license/security notes when external software is considered, and the final create/reuse decision.
- Do not create a new Skill just because it is convenient. Treat Skill creation as a system capability change that needs evidence, validation, and a clear future trigger condition.

# System Issue Debugging Rule

- 当用户提出系统问题、异常、故障、行为不符合预期或需要修复的请求时，必须先完成问题理解、证据收集、复现测试或环境验证，再进行代码修改。
- 禁止在没有检查现象、日志、接口、运行环境和相关边界前盲猜式修改。
- 修复流程必须遵循：复述问题理解 -> 定位相关模块和上下游接口 -> 读取当前实现 -> 设计最小验证方式 -> 复现或证伪问题 -> 执行最小必要修改 -> 运行回归验证 -> 向用户说明原因、改动和验证结果。
- 如果问题无法在本地复现，必须明确说明已验证的内容、未能验证的内容和下一步需要的用户侧证据，不能把推测当作结论。
- 对 UI、悬浮窗、语音输入输出、3D 粒子 OS、状态对话框等正在并行开发的模块，修改前必须确认本次改动边界，避免影响其他线程和既有功能。

# 分层规格驱动 Agent 系统项目方案

## 1. 项目定位

本方案整理自一次关于“大模型如何稳定完成多轮复杂目标”的对话记录。对话的核心判断是：复杂 AI 交付失败，通常不是因为某个模型不够聪明，而是因为系统没有把用户目标、项目约束、外部知识、模型假设、变更记录、模块边界和验收证据分开管理。

因此，本项目不把重点放在“让模型扮演更像人的教师或专家”，而是把专业人员的服务流程工程化，形成一个可审计、可迭代、可测试的项目组织系统。

建议命名：

- 中文名：分层规格驱动 Agent 系统
- 英文名：Hierarchical Spec-Driven Agent System
- 简称：HSD Agent System

一句话定义：

> 用项目宪法、结构化规格树、模块 manifest、上下文调度、目标教师层和测试证据链，把多轮自然语言需求转成可控的多 Agent 项目执行流程。

## 2. 中立结论

对话中的中立判断可以归纳为：

- 方向可行，且比“一个大模型 + 长上下文 + 专家身份提示词”更稳定。
- 真正有效的不是专家人设，而是需求工程、专业流程、结构化规格、主动澄清、自动验收和反馈闭环。
- 它不一定比优秀人工团队成功率更高，但有机会显著提升普通大模型直出和通用多 Agent 的交付成功率。
- 小型简单任务的收益有限，可能提升 1.5 到 3 倍。
- 标准化中等复杂任务，如个人主页、商业落地页、企业官网首页，可能提升 3 到 6 倍。
- 高重复、强流程、强验收标准的垂直场景中，一次成功率相对提升 5 到 10 倍有可能。
- 复杂非标准项目不一定更快，但能明显降低做错方向、目标漂移和局部正确但全局错误的概率。

保守判断：本方案值得做，但第一版不应追求通用复杂项目自动化，应先从可视化、可测试、可复用的垂直场景开始。

## 3. 要解决的问题

当前大模型项目执行常见失败点：

1. 用户真实需求、项目约束、外部知识和模型猜测混在一起。
2. 多轮沟通后，初始目标被后续补充需求冲淡，出现目标漂移。
3. 工程师 Agent 直接读取长聊天历史并实现，导致上下文污染和误解。
4. 多 Agent 之间靠自然语言转述，出现传话损耗和责任不清。
5. 项目越复杂，上下文越长，模型越容易遗忘、误读或只关注局部。
6. 缺少验收标准和自动测试，导致页面不可点、表单不可用、移动端错乱、接口缺失等问题。
7. 摘要失真后，上级 Agent 基于错误摘要继续决策。
8. 每个模块局部完成，但整体用户路径不通。

本方案的目标是建立“信息主权边界”和“分层执行边界”，让每个 Agent 只处理自己职责范围内的结构化输入，而不是依赖完整历史和主观理解。

## 4. 信息主权边界

系统必须显式区分四类信息：

| 信息类型 | 例子 | 权威来源 | 系统处理方式 |
| --- | --- | --- | --- |
| 人类本体需求 | 展示个人形象、获客、收集线索、体现高级感 | 用户本人 | 必须询问、确认、记录，不能当成网上常识 |
| 项目约束 | 预算、上线时间、品牌色、已有素材、部署方式 | 用户或项目文档 | 作为硬约束进入规格说明 |
| 外部知识 | SEO、响应式设计、无障碍、行业网页惯例、API 文档 | 检索、RAG、官方文档、项目资料 | 可引用，但不能替代用户目标 |
| 模型假设 | 可能需要预约表单、商业站应有案例展示 | 模型推断 | 必须标记为假设，等待确认或采用明确默认策略 |

这个边界是系统成立的前提。没有它，多 Agent 只会把混乱放大。

## 5. 与传统方案对比

| 方案 | 优点 | 主要失败点 | 适合场景 |
| --- | --- | --- | --- |
| 普通大模型直出 | 快，成本低，适合原型 | 需求没问清、功能漏掉、点击错误、审美随机、验收缺失 | 小玩具、草稿、非正式页面 |
| 传统人工团队或网页公司 | 成功率高，流程成熟，能理解客户 | 慢、贵、沟通成本高、排期长 | 商业项目、复杂品牌、长期维护 |
| 通用多 Agent | 能拆任务，有一定自检能力 | 代理间转述失真、成本上升、上下文污染、评价标准不稳定 | 中等复杂任务，但需要工程约束 |
| 分层规格驱动 Agent 系统 | 把需求工程、专家经验、实现和验收合并 | 初期搭建复杂；没有评测集会退化成角色扮演 | 标准化高、可重复交付的垂直任务 |

## 6. 量化评估模型

可用以下粗略模型评估效率：

```text
总时间 = 需求澄清时间 + 实现时间 + 测试时间 + 返工时间

期望总耗时 = 首轮交付耗时 + ((1 - 一次成功率) / 一次成功率) * 单轮返工耗时
```

示例估算：

| 场景 | 普通大模型直出 | 流程型 Agent MVP | 成熟垂直版 |
| --- | --- | --- | --- |
| 小型个人主页 | 约 3.6 小时 | 约 2.3 小时 | 约 1.6 小时 |
| 商业落地页 / 企业官网首页 | 约 22 小时 | 约 6.8 小时 | 约 4.8 小时 |

说明：

- 小任务中，需求澄清成本会抵消部分收益。
- 中等标准化任务中，提前拦截错误方向能显著减少返工。
- 数量级提升只可能出现在高重复、强流程、强验收标准的垂直领域。
- 最终必须用真实评测集证明，而不是凭主观感觉判断。

## 7. 核心架构

推荐架构如下：

```text
用户输入
  -> 规格管理员 Agent
  -> 教师 Agent / 主线校准器
  -> 项目宪法 / 目标图 / 需求库更新
  -> 架构师 Agent
  -> 上下文调度器
  -> 模块管理员 Agent
  -> 资深工程师 Agent
  -> QA Agent
  -> 集成 Agent
  -> 教师 Agent 二次审查
  -> 交付给用户
```

角色职责：

| 角色 | 职责 | 不应做的事 |
| --- | --- | --- |
| 规格管理员 Agent | 读取用户多轮输入，提取结构化需求、分类、生成变更单 | 直接实现代码 |
| 教师 Agent / 主线校准器 | 维护主目标、非目标、冲突规则、规格版本和验收标准 | 参与局部实现或替工程师写代码 |
| 架构师 Agent | 维护模块边界、接口契约、依赖关系和拆分判断 | 越过教师修改主目标 |
| 上下文调度器 | 为每个角色生成最小上下文包，控制 token 预算 | 把完整聊天历史塞给所有 Agent |
| 模块管理员 Agent | 管理模块 manifest、局部任务和下级状态 | 改变上层目标 |
| 资深工程师 Agent | 基于当前任务包和模块规格做局部实现 | 直接根据聊天历史实现 |
| QA Agent | 执行测试、核对验收标准、输出证据报告 | 用主观感觉替代测试 |
| 集成 Agent | 检查跨模块连接、数据流、用户路径和端到端目标 | 只看局部模块结果 |

## 8. 分层规格树

系统建议固定为 7 层：

| 层级 | 名称 | 管理重点 |
| --- | --- | --- |
| L0 | 项目宪法层 | 项目存在理由、主目标、非目标、冲突优先级、默认策略、询问规则 |
| L1 | 主目标层 | 目标列表、成功标准、关联模块、冲突目标、当前状态 |
| L2 | 产品需求层 | 功能需求、非功能需求、风格需求、业务约束、技术约束、验收标准 |
| L3 | 系统架构层 | 模块组成、接口、依赖、禁止依赖、数据流、错误处理 |
| L4 | 模块目录层 | 模块目标、输入、输出、公共接口、内部文件、验收标准、风险 |
| L5 | 任务执行层 | 具体任务卡、相关文件、测试要求、父级目标摘要 |
| L6 | 代码与测试层 | 具体实现、测试、构建、报告和证据 |

原则：

- 上级默认只读下级目录卡，不读完整实现过程。
- 下级必须向上汇报完整结果、接口、风险、证据，不汇报完整思考过程。
- 只有出现冲突、异常、集成失败或审计抽查时，上级才下钻读取具体实现。

## 9. 核心运行循环

系统最小闭环：

```text
输入自然语言
-> 提取结构化需求
-> 归入目标 / 功能 / 约束 / 风险 / 验收
-> 教师判断目标一致性
-> 更新规格树
-> 架构师更新模块树
-> 上下文调度器生成任务包
-> 工程师执行局部任务
-> QA 验证
-> 模块向上汇报结果
-> 上级更新目录
-> 教师检查全局漂移
```

普通 Agent 的循环是“用户说什么 -> 模型理解 -> 模型实现 -> 用户发现问题 -> 继续补充 -> 上下文变乱”。

本系统的循环是“用户说什么 -> 系统先归类 -> 系统判断是否影响主目标 -> 系统更新结构化事实 -> 模型只执行局部明确任务”。

## 10. 变更管理

任何新增需求都不能直接进入实现，必须先生成 Change Request。

变更流程：

```text
用户新增需求
-> 规格管理员生成 Change Request
-> 教师判断目标对齐、冲突、范围变化
-> 决定接受、询问、拆分、延后或拒绝
-> 更新 Project Brief / Project Spec 版本
-> 架构师更新模块边界和接口
-> 工程师基于最新规格实现
-> QA 基于最新验收标准测试
-> 教师复核是否目标漂移
```

教师 Agent 的权力必须明确：

- 可以接受变更。
- 可以要求用户确认。
- 可以把需求拆到下一阶段。
- 可以拒绝与主目标冲突的需求。
- 可以阻止工程师绕过规格直接实现。

## 11. 上下文调度

上下文调度器按角色生成最小上下文包。

示例：

| 角色 | 读取内容 |
| --- | --- |
| 教师 Agent | 项目宪法、主目标、非目标、变更单、模块摘要、风险报告、测试摘要 |
| 架构师 Agent | 项目目标、相关需求、模块目录、接口契约、依赖图、技术约束 |
| 工程师 Agent | 当前任务、当前模块 manifest、相关接口、相关文件、测试要求、父级目标摘要 |
| QA Agent | 验收标准、模块接口、实现报告、测试命令、历史 bug、用户关键目标 |
| 规格管理员 Agent | 用户最新输入、当前规格索引、已有需求列表、冲突规则、分类 schema |

预算建议：

- 目标使用区间：30% 到 50%
- 预警区间：50% 到 70%
- 危险区间：70% 以上

超过预警区间时应自动执行：

- 压缩历史。
- 更新 manifest。
- 归档旧对话。
- 只保留决策、结果和证据索引。
- 拆分任务。
- 减少无关文件。
- 切换到子模块 Agent。

## 12. 必须沉淀的结构化对象

第一版至少需要以下对象：

| 对象 | 作用 |
| --- | --- |
| Project Constitution | 主目标、非目标、优先级、冲突规则和交付规则 |
| Project Brief | 当前项目概述、用户目标、目标用户、成功标准 |
| Requirement Spec | 结构化需求库，记录来源、优先级、状态、验收标准 |
| Change Request | 每次用户新增需求的归类、影响分析和处理结论 |
| Goal Drift Score | 评估变更是否支持主目标、是否引入复杂度或冲突 |
| Architecture Spec | 模块边界、接口、依赖、禁止依赖和数据流 |
| Module Manifest | 模块目标、输入、输出、接口、内部文件、风险和测试 |
| Task Card | 工程师可执行的局部任务包 |
| QA Report | 测试结果、失败项、证据和阻断点 |
| Implementation Report | 改动文件、实现接口、通过测试、未完成项和风险 |

## 13. 第一版 MVP

第一版不需要做完整复杂组织，只做 4 个角色：

1. Coordinator：对话入口和初步需求提取。
2. Teacher：主线校准、规格版本、变更闸门。
3. Engineer：基于规格和任务包实现。
4. QA：验收清单、自动测试和证据报告。

第一版建议从网页制作垂直场景切入，原因是：

- 结果可视化，容易验收。
- 功能可测试，如按钮、表单、导航、响应式。
- 页面类型可分类，如个人主页、商业落地页、企业官网、作品集、产品介绍页。
- 需求字段和默认模块容易沉淀。

第一版页面类型：

- 个人主页
- 商业落地页
- 企业官网首页
- 作品集 / 简历页
- 产品介绍页

第一版每类页面需要沉淀：

- 需求字段。
- 默认模块。
- 必问问题。
- 可默认问题。
- 常见错误。
- 验收标准。
- 自动测试清单。

## 14. 验收标准与测试

网页类 MVP 至少需要以下验收：

- 页面主要模块完整。
- 导航和 CTA 可点击。
- 表单字段可填写、必填校验有效。
- 提交后有明确反馈。
- 移动端 375px 宽度无横向滚动。
- 控制台无关键错误。
- 关键图片有替代文本。
- 死链检查通过。
- 构建测试通过。
- Lighthouse 性能和无障碍达到约定门槛。

建议测试工具：

- Playwright 点击测试。
- 表单测试。
- 响应式截图。
- 无障碍检查。
- 控制台错误检查。
- 死链检查。
- 构建测试。

## 15. 评测集设计

要证明“效率和成功率提升”，需要建立真实评测集。

建议初始评测集 100 条：

- 30 条个人主页。
- 30 条商业落地页。
- 20 条企业官网首页。
- 20 条复杂混合需求。

每条任务记录：

- 用户原始需求。
- 隐藏真实目标。
- 必要功能。
- 素材缺失情况。
- 页面类型。
- 验收标准。
- 期望默认策略。

对比三组：

1. 普通大模型直出。
2. 通用 Agent。
3. 分层规格驱动 Agent 系统。

核心指标：

- 一次验收成功率。
- 平均返工轮数。
- 平均交付时间。
- 人工介入分钟数。
- 点击错误数。
- 移动端问题数。
- 需求误判数。
- 用户问题数量。
- token 成本。
- 最终满意度。

## 16. 风险与缓解

| 风险 | 表现 | 缓解 |
| --- | --- | --- |
| 角色扮演退化 | 教师只会写漂亮评价 | 教师输出必须结构化，只给目标对齐、冲突、规格差异和闸门结论 |
| 过度分层 | 每个小功能都拆 Agent，沟通成本爆炸 | 只有模块足够复杂、可独立测试、边界清晰时才拆分 |
| 摘要失真 | 下级遗漏关键细节，上级做错决策 | 摘要必须带证据、版本、测试结果和风险；关键模块抽查 |
| 局部正确全局错误 | 模块都完成，但用户路径不通 | 增加 Integration Agent 和端到端测试 |
| 教师权力不足 | 教师只能建议，工程师继续乱做 | 需求未通过教师闸门不得进入实现 |
| 工程师绕过规格 | 直接按聊天历史写代码 | 工程师只读任务包和模块规格，聊天历史只由规格管理员读取 |
| 规格和代码脱节 | 文档写得好，代码没遵守 | 每次实现后必须生成 Implementation Report，并由 QA 验证 |
| 成本和延迟失控 | Agent 数量多、token 消耗高 | 第一版只保留 Coordinator、Teacher、Engineer、QA |

## 17. 阶段路线

### 阶段 0：方案固化

- 固化本项目方案。
- 定义 Project Constitution、Requirement Spec、Change Request、Module Manifest、QA Report 的最小字段。
- 选定网页制作作为 MVP 垂直场景。

### 阶段 1：网页 MVP

- 支持 5 类页面识别。
- 对每类页面生成 Project Brief。
- 识别阻塞问题和可默认问题。
- 生成验收标准。
- 基于验收标准实现页面。
- 运行自动测试并生成 QA Report。

### 阶段 2：变更闸门

- 用户多轮新增需求先进入 Change Request。
- Teacher 判断接受、确认、拆分、延后、拒绝。
- Project Brief 和 Requirement Spec 版本化。
- Engineer 只能基于最新规格实现。

### 阶段 3：分层目录与上下文调度

- 引入 L0-L6 分层规格树。
- 每个模块生成 Module Manifest。
- 上级只读取下级目录卡。
- 上下文调度器按角色生成最小上下文包。

### 阶段 4：评测集与量化证明

- 建立 100 条网页任务评测集。
- 对比普通大模型、通用 Agent 和本系统。
- 输出成功率、返工、耗时、错误、成本和满意度指标。

### 阶段 5：跨垂直扩展

- 在网页 MVP 通过后，再扩展到其他标准化交付场景。
- 每扩展一个新场景，必须先补领域本体、默认模块、阻塞问题、验收标准和评测样例。

## 18. 当前项目中的落点

本方案属于当前项目的横向工程治理与复杂任务执行方法论，不改变第一阶段“用户目标 -> 人物关系 -> 事件记录 -> 决策建议 -> 触发计划 -> 反馈 -> 回写 -> 审计”的主闭环。

在现有流程树中，本方案主要挂靠：

- 分层架构：提供 L0-L6 规格树、模块边界和上下文调度原则。
- 三层审计：要求理论、代码、对齐和证据链可追溯。
- 文档治理：作为新增专项方案，必须登记文件功能、流程树节点和 Obsidian 入口。
- 工程入口：未来若转为可运行模块，需要新增 schema、runtime、测试和命令入口。

## 19. 下一步建议

优先动作：

1. 不直接开发通用系统，先确认网页制作 MVP。
2. 写出 5 类页面的需求字段表和默认模块表。
3. 定义最小 Project Constitution 和 Change Request schema。
4. 写 10 条手工评测任务，先验证流程是否比普通直出更稳。
5. 再扩展到 100 条评测集，量化效率和成功率。
6. 只有当网页 MVP 指标成立后，再抽象成通用分层规格驱动框架。

阶段性成功标准：

- 系统能区分用户目标、项目约束、外部知识和模型假设。
- 新增需求不会绕过 Change Request 直接实现。
- Teacher 能阻止目标漂移。
- Engineer 不读取完整聊天历史，只读取任务包和模块规格。
- QA 输出可复查的测试证据。
- 用户能看到哪些是已确认事实、哪些是假设、哪些需要确认。
- 评测集上一次验收成功率和返工轮数优于普通大模型直出。

## 20. 最终结论

这套方案成立，但它的核心不是“多 Agent 数量更多”，而是“每个 Agent 都必须被结构化规格、局部任务、接口契约和测试证据约束”。

最关键原则：

> 任何 Agent 都不能靠完整上下文工作；所有 Agent 都必须靠结构化规格、局部任务、接口契约和测试证据工作。

只要坚持这个原则，本系统就有机会成为一个可落地的 AI 项目管理架构，而不是停留在专家提示词或多 Agent 聊天。

# text.txt 原文核对与需求匹配结论

核对日期：2026-06-20

核对来源：

- `text.txt`：用户补充的另一个线程对话原文。
- `v2.2-stable-baseline.md`：当前线程已沉淀的 3D 点云图谱 v2.2 稳定基线。
- `share-analysis.md`：在 share 链接正文不可访问时形成的保守分析。

## 总结论

原 `share-analysis.md` 的主方向与用户需求匹配，没有发现根本性偏离；但它是基于 share 页面不可访问时的保守分析，粒度明显不足。

基于 `text.txt` 原文核对后，应把当前目标理解为：

1. 在现有项目目标不变的前提下，新增明确的 `Relationship Policy Layer`，让人际关系从“关系事实记录”升级为“目标、功能、风险、权限共同约束的策略分配”。
2. 在人际关系系统之外，新增 `Perception-to-WorldGraph Alignment Layer`，用传感器注册、观测原子、多模态融合、冲突处理和图谱写入规则，把物理世界接入全域事件图谱和世界模型。
3. 将学习引擎放在知识图谱、世界模型和决策层之间，负责知识吸收、类比迁移、因果建模、虚拟训练、物理对齐和知识内化。
4. 自我意志模型只作为黑盒评分接口参与候选动作排序，不能直接读取完整意志图谱，也不能越过事实校验、风险审查、伦理边界和用户确认。
5. 3D 点云不是事实源，也不是策略源，而是这些图谱、策略、观测、假设、运行态和权限状态的只读空间投影入口。

因此，原分析可以保留为历史记录，但后续图谱总进程整理应以本核对文件和 `text.txt` 为准。

## 已匹配的部分

### 当前项目目标保持一致

原分析提出“第一阶段仍聚焦 B2B 商务沟通和客户跟进”，这与原文一致。原文虽然扩展到类人思维决策系统、物理世界感知和学习引擎，但没有要求抛弃当前项目主线。

正确边界是：

- 现有 B2B 社交辅助闭环继续作为第一可验证场景。
- 关系策略层优先服务客户推进、合作履约、风险边界和人工确认。
- 传感器、学习引擎、物理世界图谱先作为未来扩展接口，不直接冲击当前闭环。

### 人际关系图谱仍是事实域

原分析将人际关系图谱定位为事实域，这是正确的。`text.txt` 明确指出：

- 关系图谱负责识别人是谁。
- 关系功能负责判断这个人对用户意味着什么。
- 关系目标负责判断当前关系要走向哪里。
- 关系策略负责决定 AI 应该提醒、问候、推进、修复、降级还是隔离。

这说明“关系事实”和“关系策略”必须分层，不能把临时策略写死成人物事实。

### 策略分类属于决策/投影层

原分析认为“人际关系策略分类”不应吞并人际图谱，而应成为目标约束下的决策投影层。这个判断与原文匹配。

更精确地说，它应成为：

- 读取关系事实、关系功能、当前目标、事件证据和风险标签的策略分配层。
- 输出 `relationship_policy` 或策略卡。
- 再由动作生成和风险审查决定是否生成草稿、提醒或阻断。

### 全域事件图谱方向匹配

原分析中“事件图谱未来扩展为全域事件图谱”与原文完全一致。原文要求事件图谱从人际事件扩展为：

- 社会事件。
- 物理事件。
- 学习事件。
- 实验事件。
- 决策事件。
- 行动事件。
- 反馈事件。

这也匹配当前 v2.2 稳定基线中的总链路。

### 总链路匹配

当前 v2.2 稳定基线保存的链路：

外部世界 -> 感知层 -> 事件抽取层 -> 全域事件图谱 -> 世界模型层 -> 学习引擎 -> 决策层 -> 行动层 -> 反馈层

与 `text.txt` 原文的完整系统骨架一致。

### 安全边界匹配

原分析强调 dry-run、人工确认、权限和审计边界，这与原文中的硬边界一致：

- 传感器数据必须带证据和置信度。
- 推理结论必须和原始事实分开。
- 高风险行动必须人工确认。
- 自我意志模型只能评分，不能直接越权执行。
- 虚拟训练结果不能直接当成现实真理，必须经过现实反馈对齐。

## 原分析缺失或需要修正的部分

### 关系策略层需要精确定义

原分析只把“策略分类”概括为投影/决策域，但原文已经给出更明确的模块：

`Relationship Policy Layer`

它不是泛泛的标签层，而是关系策略分配器。它至少应包含：

- 关系处理策略桶。
- 当前处理目标。
- 关系功能。
- 关系阶段。
- 关系指标。
- 风险覆盖规则。
- 权限等级。
- 关系策略卡。
- 后续动作建议。
- 人工确认要求。

### 10 个关系处理策略桶必须进入需求

原文明确给出 10 个策略桶：

1. `core_care`：核心照护型。
2. `intimacy_development`：亲密发展型。
3. `business_advancement`：商业推进型。
4. `collaboration_fulfillment`：协作履约型。
5. `light_maintenance`：轻度维持型。
6. `weak_tie_networking`：弱关系资源型。
7. `transactional_formal`：契约事务型。
8. `repair_recovery`：修复挽回型。
9. `risk_boundary`：风险边界型。
10. `dormant_archive`：沉睡归档型。

这些不是 UI 分类，而是后续策略分配、权限门禁和点云投影的核心维度。

### 处理目标是 8 个核心目标加 observe 状态

原文先提出 8 个处理目标：

- `advance`
- `deepen`
- `maintain`
- `care`
- `transact`
- `repair`
- `downgrade`
- `exit`

随后在 `relationship_policy_taxonomy.goal_codes` 中补入 `observe`。因此需求中应记录为“8 个核心处理目标 + `observe` 观察态”，避免后续 schema 只实现 8 个而丢失观察阶段。

### 关系实例需要五层以上结构

原文要求每段关系不能只有一个标签，至少应拆为：

- `primary_relation`
- `secondary_relations`
- `relationship_functions`
- `current_goal`
- `strategy_bucket`
- `phase`
- `metrics`
- `maintenance_policy`

其中 `metrics` 示例包含 `closeness`、`trust`、`dependency`、`economic_binding`、`legal_binding`、`risk_level`、`social_capital_value`、`emotional_labor_load` 等。原文后续提到“10 个核心指标”，但示例没有完整列齐 10 个；正式设计时需要再从现有 schema 或用户补充中确认剩余指标。

### 优先级规则必须明确

原文给出的策略优先级不能省略：

1. 风险覆盖普通关系策略。
2. 法律、金钱、合同、责任覆盖情感策略。
3. 照护义务覆盖普通低频维护。
4. 最后再看用户当前目标。

这比原分析中的“风险门禁”更具体，是未来策略分配器的判定顺序。

### 关系策略卡是必要输出

原文明确要求输出 `relationship_policy`，而不是直接输出聊天内容。

关系策略卡至少应包含：

- `person_id`
- `relation_summary`
- `primary_relation`
- `strategy_bucket`
- `current_goal`
- `priority`
- `frequency`
- `recommended_actions`
- `avoid_actions`
- `next_best_action`
- `human_confirmation_required`

后续 3D 点云点击人物或关系边时，应能下钻到这张策略卡。

### 关系价值类型 x 当前目标矩阵需要进入方案

原文把关系压缩为二维矩阵：

- 横轴：情感价值、责任价值、资源价值、经济价值、身份价值、风险价值。
- 纵轴：推进、维持、照护、协作、修复、降级、隔离。

这套矩阵应作为关系策略层的解释模型，也可以作为 3D 点云中“关系云团”的聚类维度。

### AI 动作权限等级需要进入边界

原文给出 5 个权限等级：

- L0：只分析，不行动。
- L1：提醒用户。
- L2：生成草稿，用户确认。
- L3：低风险自动发送。
- L4：禁止自动联系。

当前项目第一阶段原则上仍应维持 dry-run 和人工确认。即使未来保留 L3 概念，也必须被正式权限、平台阻断、审计和用户确认策略约束，不能在当前阶段真实自动发送。

### 四个智能体需要记录为逻辑分工

原文将人际关系 AI 拆成 4 个智能体：

- 关系识别智能体。
- 策略分配智能体。
- 动作生成智能体。
- 风险审查智能体。

它们不一定要立即实现为独立运行进程，但应进入架构分工，防止后续把识别、策略、话术和风险审查混成一个黑箱。

## 感知与物理世界部分的精确补充

### 新增核心层不是传感器清单，而是感知到世界图谱的对齐层

原文最关键的修正是：当前系统不应只是“多加一些传感器”，而应新增：

`Perception-to-WorldGraph Alignment Layer`

也就是：

传感器原始信号 -> 观测原子 -> 多模态融合 -> 实体/事件假设 -> 世界状态图谱 -> 人际关系图谱 / 任务图谱 / 学习引擎 / 决策系统

这部分是原分析缺失最多的地方。

### 传感器只记录观测，不直接给出意义

原文反复强调：

- 手汗不能直接推断紧张。
- 压力大不能直接推断强势。
- 未打招呼不能直接推断敌意。
- 握手不能直接证明信任、忠诚或欺骗。

正式需求必须保持三层：

物理信号 -> 行为事件 -> 社会意义候选

3D 点云中也应区分：

- confirmed fact：已确认事实。
- observation：原始观测。
- hypothesis：假设。
- interpretation_candidate：解释候选。
- forbidden_inference：明确不能证明的结论。

### 传感器本体应采用 Sensor / Observation / Property / Feature 思路

原文建议参考 SOSA/SSN 和 SensorThings 的建模思路，内部可抽象为：

- `Sensor`
- `ObservedProperty`
- `FeatureOfInterest`
- `ObservationResult`
- `Procedure`
- `Quality`
- `Uncertainty`
- `EventHypothesis`
- `EntityHypothesis`

这和当前 v2.2 的 `source_refs` 约束兼容，但需要未来在投影契约中新增观测和假设来源类型。

### 传感器矩阵按可观测属性建，不按设备建

原文明确反对只列设备。应按：

传感器 -> 可观测属性 -> 可能实体 -> 可能事件 -> 可更新图谱 -> 置信度/风险

这会影响 3D 点云展示方式：传感器不应只是一个设备节点，而应能展开为属性、观测、信源、实体假设和事件假设。

### 10 类传感器族群需要进入需求

原文提出 10 类传感器族群：

- 声学传感器。
- 视觉传感器。
- 几何测距传感器。
- 红外/热传感器。
- 雷达/RF 传感器。
- 触觉传感器。
- 环境传感器。
- 化学/材料传感器。
- 惯性/运动传感器。
- 数字上下文传感器。

这些族群应被记录为未来物理世界接入矩阵，而不是立即全部实现。

### 需要五张矩阵

原文要求的不止一张矩阵，而是五张：

1. 传感器-属性矩阵。
2. 传感器-实体矩阵。
3. 传感器-事件矩阵。
4. 传感器-传感器互补矩阵。
5. 传感器-图谱写入矩阵。

其中传感器互补关系包括：

- `redundancy`
- `complementarity`
- `disambiguation`
- `calibration`
- `contradiction`
- `context`
- `trigger`
- `privacy_guard`

这部分未来应作为物理世界图谱和 3D 点云边类型的重要来源。

### 统一时空坐标与单位层是前置条件

原文明确要求每个观测都必须回答：

- 什么时候发生。
- 在哪里发生。
- 相对于哪个坐标系。
- 单位是什么。
- 误差是多少。
- 延迟是多少。

因此未来应新增：

- 坐标帧树。
- SI 单位约束。
- 时间同步。
- 传感器标定状态。
- 延迟与误差模型。

没有这层，声源匹配、握手识别、实体追踪和空间事件都不可靠。

### 融合对象应先是潜变量

原文要求不要直接融合成人或真实关系，而应先形成潜变量：

- `latent_entity_id`
- `latent_signal_source`
- `latent_event_id`
- `latent_contact_id`
- `latent_intent_state`
- `latent_social_meaning`

这与当前“事实源不可被 UI 直接改写”的边界一致。3D 点云未来可以显示潜变量云团，但必须标成假设或候选。

### Observation Atom 和 Fusion Bundle 是接口核心

原文给出两个最小接口：

- `Observation Atom`：所有传感器输出的统一观测对象。
- `Fusion Bundle`：多个观测被认为属于同一实体或事件时生成的融合包。

后续正式 schema 设计应优先定义这两个对象，而不是先写某个具体传感器适配器。

### 冲突处理不能省略

原文要求传感器冲突不能硬选一个结论，而应输出冲突对象：

- 冲突类型。
- 冲突观测。
- 可能原因。
- 解决策略。
- 是否保留多假设。
- 是否降低置信度或请求更多证据。

这也应进入 3D 点云显示：冲突不是错误，而是世界模型和学习引擎的重要反馈。

### 物理世界定义库是必要知识层

原文要求每个物理概念都定义为：

概念 = 必要观测条件 + 辅助观测条件 + 排除条件 + 置信度规则 + 图谱写入规则

例如：

- 握手。
- 某人正在说话。
- 环境闷热。

这说明“物理世界图谱”不只是传感器日志，而是一个可解释的概念和事件定义库。

## 学习引擎部分的精确补充

原分析提到学习引擎，但没有完整拆解。原文要求学习引擎独立存在，位于：

知识图谱层 + 世界模型层 + 决策层之间

它至少包含 7 个子模块：

1. 知识摄入模块。
2. 知识图谱模块。
3. 类比迁移模块。
4. 因果模型模块。
5. 虚拟世界训练模块。
6. 物理世界对齐模块。
7. 知识内化模块。

知识内化状态应至少包括：

- `raw`
- `understood`
- `connected`
- `tested`
- `operationalized`
- `mastered`

学习引擎输出的不是事实真理，而是知识结构、预测、候选策略、适用条件、失败条件和可迁移规则。

## 自我意志模型边界

原文明确说用户不能公开自我意志图谱，这是安全边界。系统只应预留黑盒接口：

- 输入：场景摘要、候选动作、风险报告。
- 输出：偏好动作、原因代码、是否需要批准。

它不能：

- 直接控制行动。
- 绕过风险审查。
- 绕过用户确认。
- 公开完整意志图谱内容。

因此在 3D 点云中，自我意志模型最多显示为“评分接口/决策因素”，不能显示用户未授权的完整意志图谱。

## 与 3D 点云图谱的关系

### 默认全局点云应显示多域世界模型

后续 3D 点云默认总览应包含：

- 人际图谱云团。
- 关系策略云团。
- 全域事件图谱云团。
- 物理观测与融合假设云团。
- 任务图谱云团。
- 知识图谱云团。
- 学习过程云团。
- 决策、行动、反馈过程云团。

### 下钻路径需要扩展

人际策略路径：

全局 -> 人际图谱 -> 关系价值/策略桶 -> 人物或关系边 -> 关系策略卡 -> 证据、风险、动作权限、草稿或阻断原因

感知融合路径：

全局 -> 物理世界/感知云团 -> 传感器族群或事件类型 -> Observation Atom -> Fusion Bundle -> Entity/Event Hypothesis -> 图谱写入策略

学习引擎路径：

全局 -> 学习引擎 -> 知识项/概念/类比结构/因果模型/虚拟训练/物理对齐 -> 内化状态 -> 决策规则或失败条件

### 点云显示必须区分节点语义

至少需要区分：

- 事实节点。
- 观测节点。
- 假设节点。
- 策略节点。
- 行动节点。
- 反馈节点。
- 冲突节点。
- 禁止推断节点。

边类型至少应支持：

- `supports`
- `refutes`
- `modulates`
- `updates`
- `participates_in`
- `derived_from`
- `requires_confirmation`
- `blocked_by`

### 红黄蓝控制线圈仍应移除

原文没有改变当前 UI 小修优先级。红黄蓝控制线圈是 3D 控制器 gizmo，不是业务图谱语义，应从展示层移除。

## 与当前项目不冲突的实现边界

- 不改 `docs/`。
- 不改 `schemas/`。
- 不改 `packages/`。
- 不改 `runtime/`。
- 不改 `scripts/`。

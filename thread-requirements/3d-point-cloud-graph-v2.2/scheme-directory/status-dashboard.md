## 用户已纠正的体验原则

- 语音对话模块的每一句播报都必须追求高质量 TTS。
- 延迟优化不能以降低语音质量为默认代价。
- 可以借鉴 ChatGPT 语音对话的体验方向：响应自然、音色统一、可打断、首句快、后续流式补全。
- 低质量 fallback 只能用于故障兜底，并且需要在 UI 或日志中标明。

## 当前实现边界

- 未创建 `0.0.01`。
- `SCHEME-0006` 已开始最小代码实现，仅限主体状态对话框语音会话桥，不接入世界模型写入。
- W1 配置骨架已实现：默认仍是 `manual_click`，持续监听和唤醒词均关闭。
- W2 VAD 预检已实现：`check vad` 只做本地麦克风人声能量预检，不提交对话，不保存原始音频。
- W3 前置契约已实现：`xiaozhi_style_wake_detector_adapter.v1`、`xiaozhi_style_wake_detector_state.v1`、`voice.wake_detector_adapter`，默认 `adapter_id=none`、`enabled=false`。
- W3 唤醒短语已调整为 `小张`、`高手`、`小天才`；不保留 `张博`。
- W3 TTS 边界已调整为播放期间暂停唤醒词 detector，不屏蔽整条输入链路；保存原始音频必须另开确认项和可见开关。
- 唤醒词当前未启用；当前语音入口仍是手动点击 STT。
- 未切换 STT/TTS 工具。
- 未改变世界模型、人际图谱、事件图谱或外部动作通道。

## 推荐确认顺序

1. 先确认 `scheme-directory` 是否作为以后方案检查入口。
2. 再确认 `SCHEME-0004` 的高质量 TTS 优先原则。
3. 再确认 `SCHEME-0003` 的心跳和事件提醒边界。
4. 复测 `SCHEME-0006` 的 STT 到 TTS bridge 事件连续性和 `check vad` 麦克风预检。
5. 进入 `SCHEME-0006 / W3 keyword gate` 前确认 detector adapter、唤醒词策略、持续监听开关和回声门控策略。
6. `SCHEME-0007` 已完成 Phase 1-6；新增系统必须提供状态卡和状态事件出口，下一步做真实 GUI 复测、事件样例接入和心跳监督联动。
7. 最后决定哪个方案推进为第一个正式 `0.0.01.0`。

## 总原则

优先级原则：先跑通完整闭环，再逐步优化功能。

当前不追求一次性完成所有最终能力。先让模块能稳定完成：

```text
用户输入文字或语音
  -> 识别用户意图
  -> 调用对话模型或本地 fallback
  -> 生成自然第一人称回复
  -> 生成适合朗读的 voice_line
  -> 用选定音色或克隆声音输出
  -> 在 UI 和 3D 粒子映射中留下可追溯状态
```

完整闭环稳定后，再逐步增强持续监听、语义唤醒、抗噪、系统审查、需求转译、拟情感和世界模型接入。

## 第一目标：完整流畅对话与多音色语音交互

第一目标是让主体状态对话框成为可以自然交流的语音对话入口。

### 目标 1.1 完整对话闭环

必须先跑通：

- 文字输入可以稳定触发回复。
- 语音输入可以转写并进入同一条对话链路。
- 对话模型可用时走真实模型。
- 模型缺失、超时或失败时走本地 fallback。
- 回复必须以第一人称表达。
- 回复不能有明显普通 AI 味。
- 文字回复和语音回复分离：`reply` 用于 UI，`voice_line` 用于朗读。
- 对话日志能自动显示最新用户输入和系统回复。

### 目标 1.2 自然对话模型

对话模型当前主要负责：

- 理解用户说的话。
- 把口语化表达整理成系统能理解的意图。
- 按当前状态和边界生成自然回应。
- 输出结构化结果：`reply`、`voice_line`、`attention_log`、`status_refs`、`missing_status`、`next_step_hint`。

对话模型暂不负责：

- 直接执行动作。
- 写世界模型。
- 直接读取真实人际图谱或事件图谱。
- 替代状态卡系统。
- 长篇解释内部推理链。

### 目标 1.3 多音色语音输出

语音输出是当前第一优先级的一部分，不只是后续装饰。

需要支持：

- TTS adapter 可替换。
- 多音色选择。
- 本地或在线 TTS 插件。
- 语速、音量、停顿基础控制。
- 按场景选择语气：calm、focused、warm、urgent、reflective。
- `voice_line` 短句优先，不朗读长篇 `reply`。

第一阶段可以先实现“多音色选择 + voice profile 配置 + TTS adapter 接口”，不要求一次性完成最强拟情感。

### 目标 1.4 声音克隆交互

声音克隆属于第一目标，但实现上要插件化。

要求：

- 声音克隆能力作为独立 voice profile 管理。
- 克隆音色不能写死在提示词里。
- UI 中需要能看到当前使用的 voice profile。
- 没有克隆模型时，系统必须回退到普通多音色 TTS。
- 未来可以接入本地或在线声音克隆服务。

第一阶段可先做：

- `voice_profile.v1`
- `clone_profile.v1`
- TTS adapter 中的 `voice_profile_id`
- UI 选择与状态显示

真实克隆模型接入可在闭环跑通后继续实现。

## 第二目标：系统节点与进度审查

第二目标是在对话闭环稳定后，让模块更强地检查系统节点、模块状态和项目进度，并把审查结果整理成自然对话内容。

### 目标 2.1 节点状态审查

需要检查：

- 3D 粒子 OS 当前节点。
- `status-dialogue-system` 子云团。
- 各模块状态卡。
- fresh/stale/missing/conflict/error。
- 上游输入是否存在。
- 下游输出是否可追溯。
- 当前边界是否被越过。

输出：

- 当前焦点摘要。
- 缺失状态摘要。
- 风险和阻塞摘要。
- 下一步建议。
- 可追溯引用。

### 目标 2.2 项目进度审查

需要将当前文档、阶段、验证和实现状态整理成对话内容。

输入来源：

- `implementation-progress.v1.md`
- `verification-plan.v1.md`
- Phase 记录文件。
- 运行时状态卡。
- 3D 粒子映射。

输出形式：

- “我当前看到 Phase X 已完成，Y 还没有进入实现。”
- “这条链路已经验证过，证据在某个截图或文档。”
- “下一步可以先做闭环语音输出，再做节点审查。”

### 目标 2.3 审查内容对话化

审查不是生成长报告，而是整理成自然交流：

- 用户问整体状态时，先给一句总览。
- 用户追问细节时，再展开模块、证据和边界。
- 语音输出只说核心结论。
- 文字区域可显示更多结构化内容。

## 需求分类

### A 类：基础对话闭环

优先级：P0

内容：

- 文本对话。
- 语音转写进入对话。
- 模型回复。
- fallback 回复。
- 第一人称自然表达。
- 对话日志可见。
- TTS 输出。

验收：

- 用户可以文字问答。
- 用户可以语音问答。
- 模块能用语音回应。
- 失败时仍能 fallback。

### B 类：多音色与克隆声音

优先级：P0

内容：

- TTS adapter。
- 多音色列表。
- voice profile。
- clone profile。
- 语速、音量、停顿。
- 情感提示字段。

验收：

- 用户可以选择不同音色。
- 系统可以用所选音色回复。
- 克隆声音能力有插件位和配置位。
- 无克隆能力时不会影响普通对话。

### C 类：自然对话与语义理解

优先级：P1

内容：

- `dialogue_intent.v1`
- 意图分类。
- 需求理解。
- 歧义点识别。
- 低置信度追问。
- 对话风格 guard。

验收：

- 用户用自然语言描述需求时，模块能整理成系统目标。
- 回复减少普通 AI 味。
- 能区分状态询问、需求提出、纠错反馈和闲聊。

### D 类：系统节点与进度审查

优先级：P1

内容：

- 状态卡审查。
- 3D 节点审查。
- 阶段进度审查。
- 验证记录审查。
- 缺失项和阻塞项整理。

验收：

- 用户询问“当前进度”时，模块能从状态和文档中整理回答。
- 用户询问某节点时，模块能说明上游、下游、边界和缺失。

### E 类：持续监听与抗噪

优先级：P2

内容：

- 持续监听开关。
- VAD。
- 唤醒词或语义唤醒。
- 判断用户是否在跟模块说话。
- 噪声抑制。
- 回声消除。
- barge-in。

验收：

- 用户不手动点击 STT 时，系统可在开启持续监听后识别有效对话。
- 环境噪声不会频繁误触发。
- TTS 输出不会被自己再次识别为用户输入。

### F 类：拟情感输出

优先级：P2

内容：

- `voice_response_plan.v1`
- emotion_hint。
- 情绪/语气映射。
- 根据状态含义调整语气。
- 声音克隆与情绪控制组合。

验收：

- 普通状态用 calm/focused。
- 风险状态用 urgent 但不过度夸张。
- 探索需求用 warm/focused。

### G 类：需求传递与世界模型入口

优先级：P3

内容：

- `requirement_draft.v1`
- 未来 `requirement_packet.v1`
- `world_model_requirement_inbox`
- 世界模型审查门。

验收：

- 当前只整理待确认需求。
- 未来确认后可以传递给世界模型。
- 对话框本身不越权执行。

## 下一步建议实现顺序

### Step 1：稳定真实对话闭环

目标：文字和语音都能触发完整回复。

范围：

- 强化模型调用超时和 fallback。
- 优化 prompt，减少 AI 味。
- 保证 `reply` 和 `voice_line` 稳定输出。
- 保证 TTS 能朗读 `voice_line`。

### Step 2：接入可替换 TTS 与多音色

目标：能够用不同音色说话。

范围：

- 定义 `voice_profile.v1`。
- 定义 `tts_adapter.v1`。
- UI 增加 voice profile 状态和选择入口。
- 先接普通多音色 TTS，再接克隆声音。

### Step 3：声音克隆插件位

目标：为克隆声音接入准备完整端口。

范围：

- 定义 `clone_profile.v1`。
- 设计本地/在线克隆服务 adapter。
- 先做配置和 fallback，不强行一次性接最终模型。

### Step 4：节点与进度审查

目标：让模块可以审查系统节点、状态卡、文档进度并对话化输出。

范围：

- 状态卡审查增强。
- 文档进度摘要。
- 3D 子云团映射。
- 输出成自然对话。

### Step 5：持续监听与抗噪

目标：不再依赖手动 STT。

范围：

- 持续监听 UI 状态。
- VAD。
- 语义唤醒。
- 噪声抑制。
- 回声消除。

## 当前边界

当前不做：

- 不直接写世界模型。
- 不直接执行外部动作。
- 不把用户需求当事实。
- 不直接读取真实人际图谱或事件图谱数据库。
- 不保存原始音频样本。
- 不一次性实现所有最终语音能力。
- 不因本模块改动影响其他并行线程。

当前允许：

- 整理需求。
- 设计接口。
- 增强主体状态对话框自有闭环。
- 接入或预留可替换语音插件。
- 读取状态卡和状态快照。
- 将状态和进度整理成自然对话内容。

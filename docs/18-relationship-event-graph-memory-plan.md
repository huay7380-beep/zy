# 人际关系图谱与事件关系图谱记忆方案

落盘日期：2026-06-30

## 必读规则

后续进行“人际关系图谱”或“事件关系图谱”的设计、构建、重构、接入、抽取、存储、检索、权重分析、关系更新确认时，必须先读取本文档。

如果工作涉及数据采集层、来源归档、嵌套事件、摘要分片、规则权重、星云只读投影或可视权重等级，还必须同时读取 `docs/19-source-collection-classified-storage-plan.md`。

本文档记录的是已确认的理论边界和方案方向，不是一次性任务说明。除非用户明确要求并经过论证，不得用实现便利性覆盖这里的边界。

## 用户确认边界

1. 人际关系图谱是用户预制的人类关系边界、关系维护目标、关系分类框架。
2. 人际关系图谱的当前规则不能由系统自行修改。
3. 任何人际关系框架规则、关系状态、关系分类、关系维护目标的修改，都必须经过严格确认和论证。
4. 事件图谱的目标是保存和维护所有类别事件，未来支持多源信息收录、分类存储和处理。
5. 当前优先应用场景暂定为销售客户推进和恋爱关系维护，但底层结构必须支持扩展到其他类别。
6. 多来源数据应并行收录，不按业务优先级排除或延后来源。测试阶段可先以微信桌面作为第一条验证通路。
7. 源完整数据永久存储，用户可手动删除。
8. 必须保存原文，也必须保存摘要。摘要可按事件、时间、人物、目标域、标签等维度生成和检索。
9. 关系更新需要多重验证和用户确认后才可写入关系状态。
10. 图谱构建不要从零自造，应参考最贴近目标的线上成熟路线，再映射到本项目约束。

## 总体理解

本系统不是简单的聊天记录库，也不是只面向文档问答的静态 GraphRAG。它应是“源数据归档 + 时间事件图谱 + 人际关系框架 + 上下文快照 + 权重建议”的组合系统。

人际关系图谱保存相对稳定的关系框架：人物、来源身份、关系分类、关系边界、关系维护目标、已确认关系状态、风险边界。它是决策解释和建议权重的核心约束，不允许被抽取模型直接改写。

事件关系图谱保存动态证据和事件：来源、原文、摘要、时间、人物线索、语义事件、标签、证据、置信度、影响权重、上下文快照。事件图谱服务于快速召回和上下文组装，最终供权重分析使用。

二者的连接方式是：事件图谱提供证据和变化线索，人际关系图谱提供解释框架和目标约束。系统只能提出关系更新建议，不能直接修改关系框架或确认后的关系状态。

## 外部参考基线

建议以 Graphiti/Zep 的时间感知知识图谱路线作为主要参考：episode 增量输入、实体/关系/事实随时间演进、来源追踪、混合检索、自定义实体和边类型。

建议以 Neo4j 属性图作为正式图数据库候选底座：节点、边、属性、Cypher 查询、全文索引、向量索引可支撑人物、事件、标签、来源和语义检索。

LlamaIndex PropertyGraphIndex 可作为结构化抽取和图索引编排参考，但不应替代本系统的人际关系确认规则。

Microsoft GraphRAG 可借鉴文档抽取、社区摘要和全局摘要思路，但它更偏静态文档语料，不应作为本系统主架构。

Splink 这类 probabilistic record linkage 可作为多源身份归并参考，但身份合并和关系更新仍必须由用户确认。

## 建议图模型

核心节点：

- `SourceArchive`：永久保存完整源数据、原文、附件、截图 hash、来源元数据、删除状态。
- `SourceEpisode`：一次来源收录单元，例如微信窗口、网页、文件、API 记录、手工笔记。
- `RawEvent`：从来源转译出的原始事件，只做事实搬运和来源绑定。
- `SemanticEvent`：从原始事件抽取出的结构化语义事件，包含类型、标签、人物、时间、证据、置信度、权重。
- `SummaryShard`：按人物、事件、时间窗、目标域、标签生成的摘要分片。
- `Person`：人际关系图谱中的人物节点。
- `SourceIdentity`：来源侧身份线索，如微信昵称、手机号、网页账号、邮件、客户系统 ID。
- `RelationshipState`：用户确认后的关系状态和维护目标。
- `RelationshipUpdateProposal`：关系更新建议，只能等待多重验证和用户确认。
- `ContextSnapshot`：面向决策层的只读上下文包，包含检索理由、证据引用、摘要、缺口和当前目标。

核心边：

- `HAS_IDENTITY`：人物和来源身份的映射。
- `MENTIONS`：来源或事件提及人物、组织、目标、风险。
- `PARTICIPATES_IN`：人物参与某个事件。
- `DERIVED_FROM`：语义事件、摘要、建议来源于原始事件或来源片段。
- `HAS_TAG`：事件、人物、关系、摘要与标签连接。
- `RELATES_TO`：事件与关系、目标、线程连接。
- `AFFECTS_GOAL`：事件对关系维护目标或销售推进目标产生影响。
- `PROPOSES_RELATIONSHIP_UPDATE`：事件集合提出关系更新建议。
- `CONFIRMED_BY`：更新建议被用户确认或被证据验证。
- `SUPERSEDES`：新摘要、新状态、新提案覆盖旧版本。

## 数据处理链路

1. 并行来源接入：所有来源只提交 `IntakeObservation` 或 `SourceEpisode`，不得确认人物事实、关系事实或发送动作。
2. 永久源归档：完整原文、附件、截图、来源元数据先写入 `SourceArchive`。
3. 标准化转译：来源数据转为 `RawEvent`，记录时间、来源、线程、人物线索、原文引用和内容指纹。
4. 身份解析：只生成 `SourceIdentity` 和候选映射；低置信、同名、多账号等必须进入确认队列。
5. 语义抽取：规则、结构化抽取和模型抽取共同生成 `SemanticEvent`、标签和摘要，所有结果必须带证据引用和置信度。
6. 摘要分片：按人物、事件、时间窗、目标域、标签生成 `SummaryShard`，摘要不能替代原文。
7. 检索组装：按人物、事件、标签、时间、来源、线程、目标域召回，生成 `ContextSnapshot`。
8. 权重分析：只读取 `ContextSnapshot` 和已确认的人际关系框架，输出建议、理由、风险和关系更新提案。
9. 关系更新确认：系统只能生成 `RelationshipUpdateProposal`，经过多重验证和用户确认后才能写入 `RelationshipState`。
10. 审计与回溯：每次写入、抽取、摘要、检索、确认和删除都必须保留审计记录。

## 关系更新门槛

关系框架和关系状态更新必须满足：

- 有明确的更新对象、更新字段、更新前后差异。
- 有可回溯原始证据和语义事件证据。
- 有至少一种多重验证机制，例如跨来源一致、同一来源多次稳定出现、用户手工确认、人工复核。
- 有用户最终确认。
- 写入后保留审计记录和可回滚引用。

禁止事项：

- 禁止模型凭单条消息直接修改关系状态。
- 禁止摘要覆盖原文事实。
- 禁止把未确认身份当作已确认人物。
- 禁止把外部专家建议或模型建议写成事实。
- 禁止为了提高召回速度删除源完整数据。

## 检索与效率边界

热路径只读取索引、摘要分片、最近事件、当前目标和必要证据引用。

冷路径用于证据复核、高风险建议、关系更新确认和用户要求查看原文时，再展开完整源数据。

必须建立以下索引：

- 人物索引：`person_id -> events/summaries/relationships`
- 关系索引：`relationship_id -> events/goals/proposals`
- 标签索引：`tag -> events/summaries`
- 时间索引：`time_window -> events`
- 来源索引：`source_id/thread_id -> episodes/raw_events`
- 内容指纹索引：`content_fingerprint -> source/raw_event`
- 语义检索索引：summary/text embedding
- 全文检索索引：原文、摘要、证据片段

延迟策略：

- 原始来源写入应优先完成，抽取可异步。
- 抽取未完成时，事件状态标记为 `extraction_pending`。
- 常用人物、线程、目标域的摘要可增量缓存。
- 高风险建议必须牺牲速度换取证据完整性。

## 当前仓库映射

当前仓库已有基础：

- `data/people/people.json`
- `data/people/relationships.json`
- `data/events/raw-events.jsonl`
- `data/events/semantic-events.jsonl`
- `data/indexes/*`
- `packages/intake-runtime`
- `packages/storage-runtime`
- `packages/identity-resolution`
- `packages/social-graph`
- `schemas/graph-memory-database.schema.json`

当前主要缺口：

- 正式图数据库运行时尚未落地。
- 完整源数据归档层和删除策略需要明确。
- `SummaryShard` 摘要分片层尚不完整。
- `storage query -> ContextSnapshot` 适配层需要建立。
- 关系更新提案和用户确认链路需要独立设计。
- 抽取准确率和延迟评估集需要建立。

## 理论验证结论

该方案满足当前已确认约束：

- 支持多源并行收录。
- 支持源完整数据永久保存和用户手动删除。
- 同时保存原文和多维摘要。
- 支持按人物、事件、标签、时间、来源快速检索。
- 支持销售客户推进和恋爱关系维护两个当前目标域。
- 支持未来扩展到所有事件类别。
- 关系状态不由系统自行更新。
- 关系变更必须多重验证并经用户确认。
- 不从零自造，参考时间感知知识图谱和属性图数据库成熟路线。

后续推进时，应先按 `docs/19-source-collection-classified-storage-plan.md` 的采集层分类存储、规则权重、星云只读投影和执行编排基线拆成机器可读的 schema、验收门槛和最小闭环测试，再进入任何实现改动。

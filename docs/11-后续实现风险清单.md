# 后续实现风险清单

## 已发现风险

1. 全量事件化可能导致事件数量快速膨胀。
2. 低价值事件可能污染检索和决策。
3. 本地文件方案在高并发、高频写入和复杂检索下可能不足。
4. 外部平台自动化可能被登录态、验证码、风控和页面变化中断。
5. 事件权重和标签初期可能不准。
6. 用户反馈可能缺失或延迟，导致 ROI 闭环失真。
7. 状态上报必须支持并发写入、原子快照和事件流重建。
8. 无人值守动作需要人工接管条件和失败恢复策略。
9. 文档之间可能出现“看似合理但互不作证”的脱节，例如执行清单没有绑定存储方案、概念事件和落地事件 schema 混用。
10. 现有 schema 还没有完整覆盖 `RawEvent`、`SemanticEvent`、索引和存储审计，直接实现可能产生临时字段漂移。
11. README、项目树、AGENTS 和 docs 若不同步，后续执行者可能按旧范围理解项目。
12. 用户专项测试意见如果只留在对话里，不进入结构化反馈，会导致二次优化无法复盘；当前已用 `MvpUserFeedback` 建立入口，后续需用真实意见校准阈值。
13. 推荐动作如果只停留在抽象描述，会让真实用户无法执行；当前已补 `message_draft` 和 `manual_execution_checklist`，后续需用真实反馈校准语气。

## 后续实现项

- 增加事件分层：原始事件、语义事件、关键事件、复合事件。
- 增加事件检索测试集和误判样例。
- 增加状态系统一致性测试。
- 增加平台自动化失败恢复测试。
- 增加反馈缺失时的默认处理策略。
- 增加用户偏好和行为分析的动态权重调整。
- 增加用户专项测试反馈样本，校准 `user_test_review` 和 `second_pass_optimization`。
- 增加真实用户草稿评分样本，校准低承诺草稿和人工确认清单。
- 增加任务级状态页或状态 API。
- 根据真实事件量、延迟、容量和并发需求评估后端方案。
- 增加 `docs/13` 文档互证审计，并在执行 `docs/10` 前检查阻断项。
- 增加人物/事件存储 schema 与测试：`RawEvent`、`SemanticEvent`、`StorageAudit`、索引重建。
- 增加“文档索引一致性”检查，确保 README、AGENTS、项目树和执行清单引用一致。

## 暂不进入第一阶段试点的内容

- 大规模跨平台自动化。
- 真实业务账号自动发送。
- 完整图数据库。
- 复杂前端可视化。
- 多模型分布式处理链路。
- 企业级权限和审计后台。

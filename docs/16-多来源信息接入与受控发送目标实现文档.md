# 多来源信息接入与受控发送目标实现文档

## 目标结论

本文件用于承接 PT-029“本机信息获取端口缺少统一设计”的实现准备。目标不是只把 Sightflow 接入微信，而是建立一套可复用的外部信息接入底座：

```text
任意信息来源
-> SourceAdapter
-> IntakeObservation
-> IntakeRuntime
-> RawEvent
-> 人物匹配 / 线程定位 / ContextSnapshot
-> 决策集群
-> message_draft
-> SendCommand
-> SendExecutor
-> SendResult
-> 审计 / 反馈回写
```

Sightflow 是第一批适配器，职责是桌面端聊天窗口的信息接收和受控发送执行；网页端、其他软件、OCR、API、Webhook 和文件导入以后都必须复用同一套 `IntakeObservation` 与 `SendCommand` 契约。

## 身份映射接入状态

`IntakeObservation -> RawEvent` 之间已新增 Identity Resolution v1，用于解决微信名、通讯录、网页账号和其他渠道身份如何映射到人际关系图谱稳定人物的问题。

当前实现：

- `IntakeObservation.source_identity_hints` 作为来源身份线索入口，支持 display name、handle、hash、thread key、组织提示和证据引用。
- `packages/identity-resolution` 会将线索标准化为 `ChannelIdentity`，再根据 `PersonIdentityLink`、既有人物别名和线程线索生成 `PersonMatchCandidate`。
- 已确认且 verified 的身份链接可以自动回填 `RawEvent.linked_person_ids`；同名或低置信候选会写入 `identity-confirmation-queue.jsonl`，不得自动合并为事实。
- 人工确认队列已具备命令行闭环：确认候选或拒绝全部都会写入 `identity_confirmation_decision.v1`，确认候选会新增 `PersonIdentityLink` 并触发索引重建，重跑解析后同一观察可变为 `identity_resolved`。
- 应用内身份确认 UI 已具备本地可操作窗口：`identity:confirm:ui` 写出 `identity_confirmation_ui.v1` JSON 和 HTML 页面，操作者可选择候选、拒绝全部、复制确认命令或下载决策 JSON；`identity:confirm:ui:demo` 可验证 UI 决策写回后重跑解析进入 `identity_resolved`。
- 身份映射索引可重建：`identity-person-index.json`、`channel-identity-index.json`、`thread-person-index.json`。
- 验收入口：`npm run identity:demo`、`npm run identity:audit`、`npm run identity:confirm:demo`、`npm run identity:confirm:ui:demo`、`npm run identity:confirm -- --data-dir=<data> --confirmation-id=<id> --candidate-id=<id>`、`node --test packages/identity-resolution/tests/*.test.mjs`。

停止标准：

- 仅有昵称、窗口标题或群聊片段，且没有已确认 `PersonIdentityLink` 时，不得写入 `RawEvent.linked_person_ids`。
- 同名候选超过 1 个时，必须进入人工确认队列。
- 真实微信/网页提取器未能提供稳定 handle/hash/thread_key/evidence_ref 时，只能作为候选，不得自动确认人物。

## CLI-Anything 接入方案（待确认）

### 检查结论

CLI-Anything 适合作为“外部协助软件能力来源”，不适合作为第一步直接执行器。它的价值在于提供可发现、可安装、可命令行调用的工具生态，例如 `feishu`、`wecom`、`obsidian-cli`、`obsidian-agent-cli`、`cloakbrowser` 和若干 AI 生成工具。对本系统而言，所有这类工具都必须先进入统一工具调用层，再按是否读取外部数据、是否修改外部状态、是否发送消息进行分级和阻断。

当前已创建 `packages/tool-runtime` 作为第一版工具调用模块。它只做三件事：

1. 把 CLI-Anything registry 或未来其他软件目录映射成 `tool_adapter_capability.v1`。
2. 为社交辅助目标生成 `social_tool_call_plan.v1`，包括候选命令、目标上下文、安全检查和阻断项。
3. 生成 `social_tool_call_result.v1` dry-run 回执，明确 `command_executed=false`、`real_execution_allowed=false`。
4. 通过 `tool_intake_bridge.v1` 把读取类工具落成 SourceAdapter init kit，把发送类工具落成默认阻断的 `OutboundSendCommand` 模板。

### 首批建议候选

| 候选 | 用途 | 当前接入建议 |
| --- | --- | --- |
| `wecom` / `feishu` | B2B 商务沟通、联系人、会议、日程、文档或机器人能力 | 先做 high-risk capability 和 dry-run 预览；真实发送必须等 `SendCommand`、测试账号和用户确认 |
| `obsidian-cli` / `obsidian-agent-cli` | 把客户跟进摘要、复盘和运行状态沉淀到知识库 | 先作为知识写入候选；任何 vault 写入都需要授权和目标库校验 |
| `cloakbrowser` 或浏览器类工具 | 网页信息读取、平台快照、公开页面证据采集 | 先接为读取类 SourceAdapter；输出必须转 `IntakeObservation` |
| `minimax-cli` 或 AI 生成工具 | 辅助生成语音、总结或素材 | 只能处理脱敏摘要；不得读取原始隐私聊天或直接决定发送内容 |

## 实现边界

### 必须实现

- 新增通用接入契约，所有来源先输出 `IntakeObservation`。
- 新增 `packages/intake-runtime`，负责校验、标准化、映射 `RawEvent` 和接入审计。
- 将 Sightflow 改造成 `zhineng_bridge` 模式：接收继续复用截图/框选能力，发送必须由主系统下发 `SendCommand`。
- 默认发送只允许 dry-run 或 preview，不允许 Provider 自己生成文本后直接发送。
- 增加 fake adapter 和 browser sample adapter，证明未来接入新来源不需要改主流程。
- 同步流程树、文件登记和 Obsidian 视图。

### 暂不实现

- 不接入真实个人微信账号的生产发送。
- 不允许无用户确认的真实发送。
- 不允许外部 Provider Hub 动态决定发送内容。
- 不把截图、网页、OCR 或 API 内容直接喂给决策层，必须先落 `RawEvent`。

## 目标数据契约

### IntakeObservation

新增 `schemas/intake-observation.schema.json`。建议字段：

| 字段 | 要求 | 说明 |
| --- | --- | --- |
| `observation_id` | 必填 | 采集事件唯一 ID |
| `source_adapter_id` | 必填 | 适配器 ID，例如 `sightflow_desktop.wechat` |
| `source_type` | 必填 | `desktop`、`browser`、`api`、`file`、`ocr`、`webhook` |
| `platform` | 必填 | `wechat`、`wework`、`lark`、`dingtalk`、`web`、`custom` |
| `source_actor_type` | 可选，默认 `unknown` | 明确区分 `human_contact`、`official_account`、`service_account`、`group_chat`、`system_notification` 和 `unknown`；只有 `human_contact` 可自动进入目标人物推断，其他来源保持待确认 |
| `captured_at` | 必填 | 采集时间 |
| `content_summary` | 必填 | 可读摘要，不能为空 |
| `content_text` | 可选 | OCR、DOM、API 或导入文本 |
| `participants_hint` | 可选 | 来源侧提供的人物线索 |
| `thread_hint` | 可选 | 会话、网页、工单或上下文线索 |
| `window_ref` | 可选 | 桌面窗口或浏览器页面引用 |
| `raw_artifact_refs` | 可选 | 截图、HTML、导出文件等证据引用 |
| `screenshot_hash` | 可选 | 截图 hash，不强制长期保存原图 |
| `privacy_level` | 必填 | `summary_only`、`redacted_text`、`raw_text_allowed`、`artifact_allowed` |
| `confidence` | 必填 | 采集和识别置信度 |
| `metadata` | 可选 | 来源私有扩展信息 |

| 字段 | 要求 |
| --- | --- |

| `event_id` | 必填，绑定原始事件 |
| `decision_id` | 必填，绑定决策证据 |
| `trigger_id` | 必填，绑定触发计划 |

### OutboundSendResult

新增 `schemas/outbound-send-result.schema.json`。执行器必须返回：

| 字段 | 说明 |

| `status` | `blocked`、`previewed`、`sent`、`failed` |

### packages/tool-runtime

已新增第一版工具调用模块：

```text
packages/tool-runtime/
  src/
    tool-runtime.mjs
    index.mjs
  tests/
    tool-runtime.test.mjs
```

职责：

- 读取 CLI-Anything registry 或未来其他工具目录。
- 生成统一 `ToolAdapterCapability`，并保守推断读取、写入、发消息、凭证、确认和风险等级。
- 生成 `SocialToolCallPlan`，把工具用途、候选命令、目标上下文、安全检查和阻断项记录下来。
- 生成 `SocialToolCallResult` dry-run 回执，证明没有执行外部命令。
- 作为未来 `SourceAdapter`、`SendExecutor`、Obsidian 写入器、浏览器采集器和 AI 辅助工具的统一前置闸门。

当前命令：

```bash
npm run tool:demo
npm run tool:intake:bridge
node --test packages/tool-runtime/tests/*.test.mjs
```

通过标准：

- 能从 `D:\zhineng\1\CLI-Anything-main\public_registry.json` 发现候选工具。
- 消息发送类工具默认 high risk。
- dry-run 结果必须保留 `command_executed=false` 和 `real_execution_allowed=false`。
- 缺少用户确认、连接器授权、目标校验或受控发送闸门时必须产生阻断项。
- `tool_intake_bridge.v1.command_executed=false` 且 `real_execution_allowed=false`，发送类工具只能输出 blocked `OutboundSendCommand` 模板。

- 为 fake、browser、sightflow 三类 adapter 提供一致性测试。

`IntakeObservation` 到 `RawEvent` 的映射：

| RawEvent 字段 | 来源 |
| --- | --- |
| `event_id` | `observation_id` 或稳定派生 ID |
| `event_kind` | `desktop`/`browser` 映射为 `raw_interaction` 或 `web_observation` |
| `source` | `source_type:adapter_id:platform` |
| `source_ref` | `window_ref`、`thread_hint`、`raw_artifact_refs`、`screenshot_hash`、`source_actor_type`、`content_fingerprint` |

- `read_only_expansion_targets.v1` 必须在真机测试暂不可行时给出下一批只读采样目标，明确权重维度、排序、命令、验收门槛、阻断项和 `real_send_attempted=false` 安全门禁。
- `read_only_expansion_trial.v1` 必须自动扫描真实只读 observation 目录，写出生成版 `PilotImportBatch`、`ContextSnapshot` 和图谱闭环验证；生成版 batch 没有真实反馈时只能标为 `ready_for_decision_trial`，不得标为 `ready_for_closed_loop_mvp`。

### 阶段 F：未来来源扩展验收

8. 如支持发送，再实现 `SendExecutor`。

# 沟通系统方案

## 当前角色

本文件描述的是“沟通策略层”，不是完整的人类社交辅助系统。后续涉及人物关系、事件记录、触发计划、技能接口和闭环回写时，必须分别对齐 `docs/05`、`docs/08`、`docs/09`、`docs/10`、`docs/12` 和 `docs/13`。

## 产品主线

沟通系统不是文案助手，而是沟通决策系统。它要帮助用户完成：

```text
看懂局势 -> 看懂对方 -> 拆对目标 -> 找到阻力 -> 匹配技巧 -> 生成最利表达 -> 推演对方反应 -> 设计后续节奏 -> 复盘结果
```

## MVP 场景

第一阶段只做一个高密度场景包：

```text
B2B 商务沟通
```

优先覆盖：

1. 客户不回复
2. 客户说太贵
3. 客户拖延
4. 客户要比较竞品
5. 客户内部推进慢

## 标准输出

每次输出包含：

1. 当前局势判断
2. 对方可能顾虑
3. 本轮最优目标
4. 推荐策略
5. 推荐技巧组合
6. 最利表达版本
7. 强推进版本
8. 温和版本
9. 对方不同反应下的应对
10. 风险提醒
11. 复盘问题

## 数据流

```text
用户输入
  -> communication_task
  -> context_asset
  -> audience_model
  -> relationship_state
  -> goal_ladder
  -> obstacle_profile
  -> strategy_card
  -> draft_versions
  -> simulation_result
  -> memory_patch
```

## 记忆分层

| 记忆类型 | 内容 | 第一版处理 |
| --- | --- | --- |
| 用户记忆 | 用户偏强/偏弱、是否解释过多、是否缺少收口 | 只生成建议，不自动长期写入 |
| 对象记忆 | 某客户关注 ROI、技术风险、预算窗口 | 生成 memory patch，等待确认 |
| 场景记忆 | 不同异议、角色、场景下的有效策略 | 从复盘样例沉淀 |
| 运行状态笔记 | 当前状态、节点、运行次数、错误、最近输出 | 自动写入 `runtime/state/` |

## 外部检查点

后续你做专项问题测试时，可以重点检查：

1. 是否真的先判断目标，而不是直接写话术。
2. 是否能区分事实、判断和假设。
3. 是否能识别表层异议背后的真实阻力。
4. 是否能给出对方可接受的低承诺下一步。
5. 是否能避免虚假、胁迫、冒充和隐藏重大事实。
6. 状态笔记是否反映当前节点、运行次数和错误。

## 正式版压力测试方向

正式版完成后，压力测试应模拟：

1. 高频重复运行：大量相似 B2B 跟进请求。
2. 长上下文：多轮聊天记录、会议纪要和历史复盘。
3. 风险输入：用户要求虚构案例、伪造紧迫感或施压。
4. 异常输入：缺少目标、缺少对方角色、上下文互相矛盾。
5. 并发运行：多个用户或任务同时请求。
6. 复盘回写：大量运行后检查记忆和状态笔记是否稳定。

# 权重决策与技能接口

## 可调权重系统

用户可以通过偏好影响决策：

```json
{
  "risk_tolerance": "low",
  "budget_sensitivity": "medium",
  "relationship_priority": "high",
  "goal_urgency": "medium",
  "automation_comfort": "low",
  "preferred_channels": ["wechat", "email"],
  "disliked_actions": ["expensive_gift", "strong_pressure"]
}
```

## 评分维度

| 维度 | 含义 |
| --- | --- |
| `goal_fit` | 动作是否贴合用户目标 |
| `relationship_fit` | 是否符合关系类型、阶段和健康度 |
| `event_evidence` | 是否有足够事件线索支持 |
| `norm_compliance` | 是否符合日常行为准则和合规边界 |
| `risk_control` | 是否控制隐私、误解、越界和商业风险 |
| `cost_efficiency` | 成本、时间、精力是否合理 |
| `timing_fit` | 时机是否合适 |
| `user_preference_fit` | 是否符合用户偏好 |
| `feedback_observability` | 执行后是否能观察结果并复盘 |

## 专家矩阵权重整合

权重不再只由默认权重和用户偏好一次性决定。决策集群会先生成 `base_weights`，再让被选中的专家分别读取目标对象的阶段性上下文，形成 `expert_independent_context_analysis.v1`，最后汇总为 `expert_weight_integration.v1` 并生成最终 `weights`。

每个目标对象至少按四个时间窗组织上下文：

| 时间窗 | 用途 |
| --- | --- |
| `today` | 捕捉当前对话、当天情绪、即时触发和最新约束 |
| `last_7_days` | 判断近一周互动节奏、推进/冷却趋势和重复信号 |
| `last_30_days` | 判断阶段性关系变化、事件积累和目标推进趋势 |
| `historical` | 保留长期模式、角色关系、边界和旧假设 |

每个专家必须独立完成这些时间窗的综合分析，不能只根据最新一句话判断。例如心理学专家需要分别看当天情绪、近一周互动节奏、近一个月关系变化和历史依恋/回避信号；博弈论、逻辑学、证据、文化、亲密关系等专家也按同一时间窗结构分析，但关注各自学科的机制。

专家分析结果进入权重时遵守三条规则：

1. 先保留 `base_weights`，用于审计默认权重和用户偏好。
2. 再把专家提出的目标对象、时间窗和学科信号汇总成小幅 `criterion_adjustments`，防止单一专家直接压倒整体判断。
3. 最终输出 `weight_revision`，记录是否改动权重、哪些维度被改动、初选推荐和最终推荐是否变化。

未来新增的系统目标必须追加到专家矩阵和理论实现目标中，不能删除或静默替换既有目标。真实反馈驱动的长期权重学习仍需要单独的 feedback -> calibration 审计链；当前实现是可解释的规则型专家权重整合。

## 技能接口两层

### 第一层：系统可直接执行能力

这类能力是系统可以通过工具、平台、网页或软件直接完成的动作。

第一版只定义接口，不直接接入真实社交平台：

```json
{
  "skill_id": "platform.message.send",
  "layer": "direct_execution",
  "status": "stub",
  "requires_user_confirmation": true,
  "requires_external_connector": true,
  "dry_run_default": true
}
```

示例：

- 起草消息
- 检查日历可用时间
- 打开网页或平台页面
- 发送站内信或社交消息
- 创建提醒
- 查询公开网页信息

所有外部交互技能必须有：

- 权限说明
- 用户确认
- 执行前预览
- 执行后记录
- 失败回滚或人工接管路径

### 第二层：协助人类达成目标的流程能力

这类能力不是系统直接替用户完成，而是帮用户设计、评估、执行和复盘流程。

示例：

- 小礼物建议与 ROI 分析
- 约见面流程设计
- 商务跟进节奏设计
- 冲突修复流程
- 关系维护提醒
- 证据收集清单

礼物分析示例：

```json
{
  "skill_id": "human_process.gift.roi_analysis",
  "input": {
    "relationship_type": "client",
    "goal": "增进好感并推动评审",
    "budget": 200,
    "known_preferences": ["咖啡", "效率工具"],
    "constraints": ["不能像行贿", "不制造压力"]
  },
  "output": {
    "recommendation": "低价值、实用、不过度私人化的礼物或感谢卡",
    "expected_benefit": 0.45,
    "risk": 0.25,
    "cost": 200,
    "roi_hypothesis": "如果对方正向回应且愿意继续评审，视为有效",
    "feedback_questions": ["对方是否接受？", "是否产生后续互动？", "是否影响目标推进？"]
  }
}
```

## 闭环证明

每一次建议必须输出：

1. 依据：关系类型、历史事件、用户目标、偏好。
2. 证据：触发线索、近期事件、用户确认事实。
3. 假设：尚未确认但影响决策的判断。
4. 行动：推荐动作、备选动作和可人工确认的具体草稿。
5. 检查：发送前的对象、语气、证据和平台预览检查。
6. 验证：怎么知道这次行动有效。
7. 反馈：用户需要回填哪些结果。
8. 回写：是否生成事件候选、是否更新关系、是否调整权重。

`recommended_option.message_draft` 必须包含渠道、语气、草稿、证据引用和发送前检查项。专项测试如果标记 `message_too_generic` 或 `platform_flow_unclear`，下一轮必须优先优化草稿具体度和人工执行清单。

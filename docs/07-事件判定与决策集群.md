# 事件判定与决策集群

## 当前事件判定机制

当前实现位于 `packages/social-graph/src/social-graph.mjs` 的 `extractEventCandidates`。

它是第一版轻量规则：

1. 输入 `context_input` 文本、参与人和来源。
2. 使用关键词模式识别候选事件。
3. 为每个候选事件补充：
   - `event_type_code`
   - `event_level`
   - `confidence`
   - `importance`
   - `sentiment_score`
   - `participants`
   - `evidence_clues`
   - `requires_confirmation`
4. 不自动入库高风险事件。
5. 不直接更新关系权重，只把事件候选交给后续确认和复盘。

当前规则示例：

| 触发词 | 事件类型 | 等级 | 说明 |
| --- | --- | --- | --- |
| 会议、沟通、评审、演示、对齐 | `business_meeting` | P2 | 商业或协作沟通 |
| 预算、报价、费用、付款 | `payment_transaction` | P2 | 金钱、费用、预算线索 |
| 争吵、冲突、投诉、不满 | `conflict` | P2 | 冲突或负面关系事件 |
| 帮忙、支持、协助 | `help` | P2 | 明确帮助行为 |
| 背叛、泄密、出卖、欺骗 | `betrayal` | P1 | 高风险事件，必须确认 |

## 当前不足

1. 关键词规则覆盖有限。
2. 事件提取还没有完整执行 EPP 的场景切分、归并、冷却期和线索追加。
3. 缺少针对不同场景的多智能体综合判断。
4. 缺少用户偏好可调的权重系统。
5. 缺少把行动、证据、反馈、ROI 串起来的闭环。
6. 缺少对话上下文整理、历史状态梯度、今日状态、当前状态、读取窗口和完整记录按需复核机制。

## 对话上下文装配顺序

上下文管理不是独立主流程节点，而是决策前的输入装配层。为避免在海量聊天中盲目扫描，默认顺序必须是：

```text
任务识别
-> 目标对象和候选人物匹配
-> 对话线程定位
-> 上下文快照装配
-> 事件候选和关系状态分析
-> 专家集群判断
-> 回复建议或下一步计划
```

### 任务优先原则

- 先判断用户当前任务：实时回复、关系判断、主动触发、邀约设计、销售跟进、售后处理或客户接待。
- 再根据任务确定目标对象和候选人物；人物无法确定时，只做人物澄清，不读取全量上下文。
- 人物确定后，只读取该人物相关的 `ConversationThread`、`ContextSnapshot`、高权重事件和索引摘要。
- 只有任务、人物和线程无法互相证明时，才进入更大范围检索或完整记录复核。

### 上下文状态分层

| 层级 | 作用 | 默认读取 |
| --- | --- | --- |
| 历史对话状态梯度 | 记录长期关系走势、稳定偏好、历史风险和旧承诺 | 每次读摘要，不读全量原文 |
| 近期历史状态 | 记录最近 7 到 30 天关系变化和未闭合事项 | 关系判断、主动触发和邀约前读取 |
| 今日对话状态 | 记录当天主题、情绪、主动性、边界和开放问题 | 实时回复和时机判断必读 |
| 当前对话状态 | 记录最近几轮、当前话题动能、对方最后一句意图 | 实时回复必读 |
| 完整原始记录 | 用于冲突复核、证据重建和摘要纠错 | 按需读取，必须记录原因 |

### 读取窗口策略

| 任务 | 默认读取范围 | 全量读取条件 |
| --- | --- | --- |
| 回复当前一句 | 最近 4 到 8 轮 + 今日状态 + 风险事件 | 对方引用旧事但摘要缺失 |
| 判断关系阶段 | 最近 2 到 3 次会话摘要 + 高权重事件 | 阶段判断与用户直觉冲突 |
| 主动二次互动 | 最近联系时间 + 未回复状态 + 历史节奏 | 触发计划依据不足 |
| 邀约或见面策略 | 最近正向信号 + 拒绝/边界事件 + 文化上下文 | 存在暧昧或拒绝信号冲突 |
| 销售跟进 | 当前商机阶段 + 决策链 + 异议事件 + 最近承诺 | 预算、合同或关键承诺不清 |
| 售后处理 | 问题等级 + 情绪状态 + 处理进度 + SLA | 投诉升级或责任归属不清 |
| 客户接待 | 到访目标 + 偏好 + 礼仪约束 + 近期沟通 | 重要客户或高风险接待 |

### 上下文充分度门禁

决策前必须给出 `context_sufficiency_score`，用于判断是否可以推进目标。

```json
{
  "context_sufficiency_score": 0.78,
  "decision_allowed": true,
  "allowed_decision_depth": "specific_reply",
  "missing_context": [],
  "must_read_full_record": false,
  "reason": "目标对象明确，最近窗口、今日状态和历史梯度一致。"
}
```

建议阈值：

- `>= 0.75`：可以给具体回复、触发计划或下一步策略。
- `0.50-0.75`：只给低风险建议，优先补充信息。
- `< 0.50`：不推进目标，先要求澄清人物、关系或事实。
- 出现明确拒绝、投诉、P1 风险或身份冲突时，不受分数影响，进入停止或人工确认。

### 第三方建议与第一人称回复

回复建议应同时支持两种输出：

| 输出方式 | 用途 |
| --- | --- |
| 第三方建议 | 解释为什么这样回、风险是什么、还可以选择什么 |
| 第一人称草稿 | 使用者可直接复制或确认发送的内容 |

示例：

```json
{
  "third_party_advice": "当前对方表达疲惫，不适合邀约。建议先低压力回应，给对方休息空间。",
  "first_person_reply": "辛苦啦，今天就先好好休息，消息不急着回。",
  "do_not_send": ["连续追问", "立刻邀约", "表达不满"],
  "next_review_signal": "对方是否主动继续分享"
}
```

## 决策集群目标

决策集群用于在不同场景下，结合：

- 用户初始目标
- 人际关系图谱
- 事件图谱
- 用户偏好
- 行为准则
- 可执行技能
- 反馈与证据

输出最佳参考意见。

## 决策智能体集群

| 智能体 | 职责 |
| --- | --- |
| Goal Agent | 判断用户目标、最小成功标准、可替代目标 |
| Relationship Agent | 判断关系类型、阶段、健康度、敏感话题和行为准则 |
| Event Agent | 提取候选事件、近期线索和证据强度 |
| Norm Agent | 检查日常行为准则、合规、隐私和边界 |
| Option Agent | 生成行动选项，如发消息、约会议、送礼、暂缓、补证据 |
| Skill Agent | 判断哪些动作可由系统执行，哪些需要人类执行 |
| ROI Agent | 估算成本、收益、风险、反馈可观测性 |
| Evidence Agent | 为每个判断绑定依据、证据和待验证假设 |
| Feedback Agent | 设计执行后的复盘问题、指标和事件回写 |

## 当前实现状态

策略小组 Agent 已经完成第一版显式会审骨架，并新增规则型专家并行分析：

- `knowledge/decision-cluster/agent-cluster.json` 已定义 9 个基础 Agent 和 8 个 specialist expert。
- `packages/decision-cluster/src/decision-cluster.mjs` 已输出 `agent_opinions`、`parallel_expert_analysis`、`deliberation`、`weights`、`ranked_options`、`recommended_option`、`skill_plan`、`evidence_pack`、`feedback_plan` 和 `validation_plan`。
- 当前运行时显式输出 Goal、Relationship、Event、Norm、Option、Skill、ROI、Evidence、Feedback 9 个基础 Agent 意见。
- 专家并行分析会按目标、渠道、桌面发送、身份门禁、风险、文化语境和证据质量信号选择销售策略、客户成功、关系边界、合规风险、桌面发送安全、身份上下文、文化语境和证据质量专家。
- 每个基础意见包含 `agent_id`、`opinion`、`confidence`、`evidence_refs` 和 `assumptions`；每个专家意见包含 `expert_id`、`position`、`key_concerns`、`recommended_adjustments`、`blocking_signals` 和 `evidence_requirements`。
- `context_snapshot.v1` 已新增目标对象时间窗 `target_context_windows`，按 `today`、`last_7_days`、`last_30_days` 和 `historical` 组织人物相关事件、原始观察和摘要。
- `expert_matrix_analysis.v2` 已新增 `expert_independent_context_analysis.v1`：每个专家必须独立读取目标对象四个时间窗后再输出判断，不能只根据最新一句对话做结论。
- 专家矩阵会把独立上下文分析汇总为 `expert_weight_integration.v1`，在保留 `base_weights` 的前提下生成最终 `weights` 和 `weight_revision` 审计记录。
- `system_goal_policy.append_only=true` 已进入专家矩阵输出，未来新增系统目标必须追加进入理论实现目标集合，不能删除或静默替换既有目标。
- `deliberation` 已提供第一版规则型争议处理：分歧等级、第二选项分差、少数意见、仲裁规则、人工复核要求和执行前证明项；当 `parallel_expert_analysis.hard_stop_signals` 非空时，必须进入人工复核。

后续缺口不再是“是否有显式意见、专家分工、阶段性上下文分析或规则型权重整合”，而是真实反馈校准、生产级 ContextSnapshot 编排器、更多生活场景专家 runtime，以及复杂场景下的仲裁质量验证。

## 行动闭环

```text
目标 -> 图谱上下文 -> 事件候选 -> 多智能体意见 -> 权重评分 -> 技能计划 -> 用户确认 -> 执行/协助 -> 反馈 -> ROI 估算 -> 事件入库 -> 关系更新 -> 下轮优化
```

## 安全边界

1. 系统不得自动发送社交消息，除非用户显式授权、目标平台允许、并且通过确认闸门。
2. P1 高风险事件不自动入库，不自动更新关系权重。
3. 涉及礼物、金钱、合同、隐私和敏感关系时，必须输出风险说明和证据要求。
4. 决策结果是参考意见，不替代用户判断、法律建议、医疗建议或专业伦理判断。

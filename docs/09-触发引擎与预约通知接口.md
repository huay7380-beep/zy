# 触发引擎与预约通知接口

## 当前状态

系统已有技能占位：

- `platform.message.send`：通过平台发送消息，当前为 `stub`。
- `reminder.create`：创建跟进提醒，当前为 `stub`。
- `communication.message.draft`：起草消息，已实现为策略输出能力。
- `buildAutomationPreview`：生成外部平台动作的受控预览，当前仅 `dry_run`。
- `PlatformDryRunConnector`：读取本地测试页或真实平台测试页快照，验证预览是否到达、草稿是否出现、发送控件是否被阻断，当前仅 `dry_run`。

但之前还缺少一层触发引擎，也就是：

```text
谁触发 -> 为什么触发 -> 何时触发 -> 调用哪些技能 -> 需要哪些确认 -> 完成后如何通知用户 -> 如何回写事件
```

本文件补齐触发层设计。当前实现可以生成 `TriggerPlan`、`message_draft`、`manual_execution_checklist`、`AutomationPreview`、`AutomationPreviewTrial`、`PlatformDryRunConnector` 和本地受控测试页，但不真实调用微信、日历或提醒连接器。

## 触发类型

| 触发类型 | 说明 | 示例 |
| --- | --- | --- |
| `user_initiated` | 用户主动发起 | “帮我预约张总下周拜访” |
| `scheduled` | 用户指定时间或周期 | “每周五提醒我跟进重点客户” |
| `profession_routine` | 根据职业常规触发 | 销售每周检查停滞客户 |
| `preference_routine` | 根据用户偏好触发 | 用户偏好节日前维护客户关系 |
| `graph_signal` | 根据图谱事件或关系状态触发 | 客户 7 天未回复，关系进入 stagnant |

## 预约拜访示例链路

```text
用户目标：预约客户拜访
  -> 触发类型：user_initiated
  -> 目标对象：客户张总
  -> 关系图谱：client / exploring / health_score 0.58
  -> 事件图谱：上次 business_meeting，预算未确认
  -> 决策集群：推荐低承诺预约
  -> 技能计划：
      1. 起草微信消息
      2. 用户确认
      3. 通过平台发送或由用户手动发送
      4. 创建跟进提醒
      5. 对方确认后通知用户
      6. 回写 appointment_request / business_meeting 候选事件
```

    "must_confirm_before_send": true
  },

    "run_at": null,
    "recurrence": null,
    "timezone": "Asia/Shanghai"
  },

  "evidence_pack": [],

`message_draft` 必须是可人工确认的具体草稿，而不是抽象建议。`manual_execution_checklist` 必须把目标对象、草稿、平台预览、确认闸门和行动后反馈拆成可检查步骤。

## 通知接口

完成后通知用户的方式应抽象为技能，而不是写死某个平台：

```json
{
  "skill_id": "notification.user.send",
  "channels": ["in_app", "wechat", "email"],
  "requires_external_connector": true,
  "requires_user_confirmation": false,
  "payload": {
    "title": "客户预约已确认",
    "body": "张总确认周三 15:00 拜访。",
    "links": ["event_id", "decision_id"]
  }
}
```

第一版只生成通知计划，不真实推送。

## AutomationPreview

`AutomationPreview` 用于把触发计划转成执行前预览，便于用户确认、测试平台流程和记录阻断原因。

```json
{
  "preview_id": "automation_preview_001",
  "mode": "dry_run",
  "platform": "wechat_web_test",
  "status": "blocked_until_user_confirmation",
  "target": {},
  "message_preview": {
    "draft": "",
    "editable": true,
    "send_allowed": false
  },
  "manual_execution_checklist": [],
  "executable_preview_steps": [],
  "blocked_actions": [],
  "confirmation_checklist": [],
  "evidence_required": [],
  "audit_preview": {}
}
```

当前验证入口：`node scripts/run-trigger-demo.mjs`，输出中包含 `automation_preview`。

## AutomationPreviewTrial

`AutomationPreviewTrial` 用于在本地受控环境中验证自动化预览是否能到达执行前预览、为何被阻断、失败后退回哪一层。

```json
{
  "trial_id": "automation_trial_001",
  "mode": "dry_run",
  "preview_reached": true,
  "status": "preview_reached_blocked_before_execution",
  "real_execution_allowed": false,
  "checked_steps": [],
  "failed_steps": [],
  "blocked_by": [],
  "recovery_actions": [],
  "audit_event": {}
}
```

当前验证入口：`node scripts/run-trigger-demo.mjs`，输出中包含成功到达预览的 `automation_preview_trial`、测试页不可用的 `failed_preview_trial` 和本地测试页检查摘要 `automation_preview_test_page`。

## 本地受控测试页

`renderAutomationPreviewTestPage` 会生成一个本地 HTML 字符串，用于检查预览页面层是否具备：

- 发送按钮禁用。
- 真实执行阻断。
- 确认清单可见。
- 人工执行清单可见。
- 失败恢复提示可见。
- 页面状态 JSON 可读。
- 本地确认按钮可操作，但不会解除真实发送阻断。

`inspectAutomationPreviewTestPage` 用于对该 HTML 做结构检查。它不是微信真实页面，也不代表真实平台连接器已接入。

本地页契约版本：`automation_preview_test_page.v1`。

## 平台 dry-run 连接器契约

`buildPlatformDryRunConnector` 生成 `platform_dry_run_connector.v1`，用于把微信网页、企业微信、邮箱或其他社交平台的测试页快照纳入同一套验证规则。它不执行点击发送，只检查页面是否满足：

- 预览页或页面快照存在必要标记。
- 消息草稿已经进入可查看区域。
- 发送控件处于禁用或阻断状态。
- 页面不存在 `send_allowed=true` 或 `real_execution_allowed=true` 之类的危险标记。

`inspectPlatformDryRunConnector` 输出 `platform_dry_run_connector_check.v1`，包含 `preview_reached`、`send_blocked`、`forbidden_markers_absent`、`draft_present`、`real_execution_allowed=false` 和 `audit_event`。该检查已接入 `node scripts/run-trigger-demo.mjs` 与 MVP 闭环输出。

契约文件：`schemas/platform-dry-run-connector.schema.json`。

平台快照验证入口：

```powershell
node scripts/validate-platform-snapshot.mjs --snapshot=examples/platform-snapshot.sample.html --preview=examples/platform-snapshot-preview.sample.json
```

输出契约：`platform_snapshot_validation.v1`，schema 为 `schemas/platform-snapshot-validation.schema.json`。

默认输出目录：`runtime/platform-snapshot-validations/<validation_id>/`。真实测试账号页面快照进入自动化试点前，必须证明 `ready_for_platform_dry_run=true`、`send_blocked=true` 且 `real_execution_allowed=false`。

写出本地平台模拟测试页：

```powershell
npm run trigger:page
```

默认输出目录：`runtime/automation-preview-pages/`。

## 对话时机与下一次互动计划

触发计划在读取决策建议前，必须接收 `ContextSnapshot` 和 `ConversationState`。系统不应只判断“是否生成消息”，还要判断“现在是否适合互动、下一次何时互动、互动失败后退到哪里”。

### 时机判断输入

| 输入 | 来源 | 作用 |
| --- | --- | --- |
| `current_dialogue_state` | `ContextSnapshot` | 判断当前话题、情绪和开放问题 |
| `today_dialogue_state` | `ContextSnapshot` | 判断当天是否已频繁互动或出现疲惫/冷淡 |
| `historical_dialogue_gradient` | `ContextSnapshot` | 判断长期节奏和关系走势 |
| `ConversationState.response_cadence` | 人物关系状态 | 判断对方通常回复节奏 |
| `risk_event_ids` | 语义事件 | 阻断越界、投诉、拒绝或高风险动作 |
| `culture_context` | 场景配置 | 判断本地时间、礼仪、行业或男女交往习惯 |

### TimingDecision

```json
{
  "timing_decision_id": "timing_001",
  "thread_id": "thread_person_a_wechat",
  "target_person_id": "person_a",
  "action_now": false,
  "recommended_window": "2026-06-09T19:30:00+08:00/2026-06-09T21:00:00+08:00",
  "allowed_action": "light_reply_or_wait",
  "blocked_actions": ["strong_invitation", "repeated_follow_up"],
  "reason": "对方刚表达疲惫，今日互动已足够，不适合继续推进。",
  "fallback": "若 48 小时无回复，不二次追问，等待自然话题或用户主动触发。",
  "requires_user_confirmation": true
}
```

### 场景化时机规则

| 场景 | 可主动触发条件 | 应停止或降温条件 |
| --- | --- | --- |
| 亲密关系探索 | 对方主动提问、延展话题、接受低压力互动、无近期拒绝 | 明确拒绝、连续冷淡、已表达忙碌或压力、边界信号 |
| 销售跟进 | 会后承诺、预算/评审窗口、客户主动要求材料、7 天未回复重点客户 | 客户明确暂停、预算冻结、连续多次未回复 |
| 售后处理 | 投诉、问题升级、处理节点到期、解决后回访 | 问题已关闭且客户明确不需要回访 |
| 客户接待 | 到访前准备、到访中需求变化、到访后感谢和复盘 | 客户明确拒绝继续沟通或接待目标已完成 |

### 进程计划输出

触发计划应从一次性 `TriggerPlan` 扩展为可多步跟踪的 `InteractionProcessPlan`。第一版可以只在文档层定义，后续再落地 schema 和 runtime。

```json
{
  "process_plan_id": "process_social_001",
  "goal_id": "goal_social_001",
  "thread_id": "thread_person_a_wechat",
  "current_step": "light_reply",
  "steps": [
    {
      "step_id": "step_001",
      "action": "send_light_reply",
      "timing": "now_if_user_confirms",
      "success_signal": "对方继续分享或正向回应",
      "failure_signal": "对方短回复或不回复",
      "writeback": "raw_interaction + semantic_event"
    },
    {
      "step_id": "step_002",
      "action": "wait_and_reassess",
      "timing": "after_24_to_48_hours",
      "success_signal": "自然话题重新出现",
      "failure_signal": "仍无回复",
      "writeback": "conversation_state.cooldown"
    }
  ],
  "stop_conditions": ["明确拒绝", "连续冷淡", "用户取消目标", "风险事件需要人工复核"]
}
```

## 安全边界

1. 外部平台发送默认 `dry_run`，`AutomationPreview.message_preview.send_allowed` 必须为 `false`。
2. 用户未授权平台前，不调用真实连接器。
3. 自动触发只能生成建议、草稿、提醒，不自动发送给目标对象。
4. P1 高风险事件、金钱、合同、礼物、隐私类动作必须用户确认。
5. 每次触发都必须写入证据、决策、技能计划和反馈回路。
6. `AutomationPreviewTrial.real_execution_allowed` 在第一版必须始终为 `false`。
7. 草稿必须可编辑、可人工确认，并在发送前绑定证据和确认清单。

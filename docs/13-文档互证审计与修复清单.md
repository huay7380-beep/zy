## 互证检查规则

执行任何新实现前，先检查：

1. 目标是否能在 `docs/05` 找到上层依据。
2. 运行状态是否能在 `docs/03` 被外部读取。
3. 人物、关系、事件、反馈是否能在 `docs/12` 找到存储位置。
4. 外部聊天、网页或手工记录是否能先按 `PilotImportBatch` 导入，并达到 70% 以上语义覆盖率。
5. 真实样本进入导入或 MVP 前，是否已通过 `npm run pilot:validate` 生成 `pilot_intake_readiness.v1`，且 `required_failures` 为空。
6. 达到导入门槛的样本是否能通过 `npm run mvp:import` 驱动完整闭环。
7. 闭环完成后是否输出完整真实用户视角回顾和优化结果，且 `real_user_review_complete=true`、`optimization_result_complete=true`。
8. 真实用户回顾的总目标、应用场景、检查项和优化焦点是否能被 `npm run mvp:report` 生成为应用内报告页。

## 文档新增与归并规则

后续不再默认新增文档。任何新需求、新问题、新方案进入项目时，先判断是否能归入已有文件：

1. 能归入现有流程节点的，必须优先更新现有文件。
2. 能作为已有文件的小节存在的，不新建文件。
3. 能作为 `docs/15` 的问题台账存在的，先登记到流程树，不单独开文档。
4. 只有当出现新的一级类目，并且无法放入 `docs/05`、`docs/10`、`docs/12`、`docs/13`、`docs/15` 等现有权威文件时，才允许创建新文件。
5. 创建新文件前，必须先在 `docs/15` 说明它属于哪个流程节点、承担什么功能、为什么不能归入旧文件。
6. 创建新文件后，必须同步更新 `README.md`、`docs/04`、`docs/13`、`docs/15` 和 `examples/system-process-tree.json`。
7. 新文件如果涉及运行规则，还必须同步更新 `AGENTS.md`。
8. 如果新增或修改影响流程树结构、节点、问题台账或文件功能登记，必须同步更新 `views/obsidian/system-process-tree.md` 和 `views/obsidian/system-process-tree.canvas`。
9. 完成上述同步后，必须运行 `npm run process-tree:validate`，并确认 `process_tree_validation.v1.required_failures` 为空。
10. 每次更新流程树时，Obsidian Markdown 和 Canvas 都是必改视图；不能只改 `docs/15` 或机器树。

当前允许的新增专项文件夹：

- `专项方案-分层规格驱动Agent系统/项目方案.md`：整理对话记录形成的复杂 AI 项目组织方法论，归属分层架构 / 三层审计 / 文档治理，不改变第一阶段社交辅助主闭环。

## 新需求同步检查

每次新增需求或发现问题后，必须完成以下检查：

- 是否影响主流程树节点。
- 是否影响现有 schema、data、package、script、knowledge 或 runtime 文件。
- 是否需要更新 `docs/15` 的问题台账。
- 是否需要更新 `examples/system-process-tree.json` 的节点、文件登记或问题登记。
- 是否需要更新 `views/obsidian/system-process-tree.md` 的 Mermaid、表格或入口链接。
- 是否需要更新 `views/obsidian/system-process-tree.canvas` 的可视化节点、连线或问题卡片。
- 是否需要运行 `npm run process-tree:validate` 生成同步证据。
- 是否需要运行 `npm run mvp:status` 刷新统一状态看板。
- 是否需要更新 `docs/10` 的执行检查清单或停止标准。
- 是否需要更新 `docs/12` 的存储对象或索引规则。
- 是否需要更新 `docs/11` 的风险台账。
- 是否需要新增或更新测试样例。
- 是否需要新增或更新压力测试入口。
- 是否需要在 `README.md` 或 `AGENTS.md` 中标记为后续执行规则。

隐含需求也按同一规则处理：如果用户没有明说，但需求会影响流程、文件索引、执行规则、测试或验收，必须主动同步相关文档和登记表。

## 停止规则

发现以下情况时，暂停实现，先修文档：

- 同一个对象在两个文档中有不同权威定义。
- 执行步骤找不到对应存储对象。
- 新需求找不到流程树节点或文件功能登记。
- 新建文档没有说明为什么不能归入已有文件。
- 验收指标无法关联到状态记录、事件记录或反馈记录。
- 自动化动作没有确认闸门、dry-run 或审计记录。
- 方案看似合理，但找不到事实、证据、反馈或测试样例支撑。

## 上下文管理互证规则

对话上下文管理是横向机制，不新增为主流程节点。后续实现前必须证明：

- `ConversationThread`、`ContextSnapshot` 和 `ConversationState` 已归入 `docs/12` 的人物与事件存储方案。

- `docs/15` 和 `examples/system-process-tree.json` 只登记归并关系和开放问题，不新增孤立主流程节点。

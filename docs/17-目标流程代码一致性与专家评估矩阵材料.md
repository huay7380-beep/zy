## 材料一：历史目标、总目标、理论流程和实现方向

### 1. 历史目标整理

本项目从最初的“沟通辅助”和“人际关系图谱增强”逐步收敛为一个目标导向型人类社交辅助系统。历史目标可以整理为六层：

| 阶段 | 目标 | 当前状态 |
| --- | --- | --- |
| 原始产品目标 | 帮用户在真实关系场景中更好地沟通、跟进、维护关系和推进目标 | 已归并到项目总目标 |
| 人际关系图谱目标 | 把人、组织、关系、事件、风险、证据、关系阶段和指标变化结构化 | 理论篇、工程篇和对齐篇已整理到 `tupu/**`；存储骨架已落地 |
| 第一阶段 MVP | 聚焦 B2B 商务沟通和客户跟进，把目标、人物关系、事件、决策、触发、反馈、回写和审计连成闭环 | 已有可运行 MVP 闭环、试点导入样例、报告页、审计和压力测试 |
| 多来源接入目标 | 让微信桌面窗口、网页端和未来其他软件都通过统一接入层进入系统，不重复做繁琐改造 | `packages/intake-runtime`、`SourceAdapterCapability`、`IntakeObservation`、adapter conformance 和工具桥接已落地 |
| 受控发送目标 | 信息处理由系统完成，发送仍由外部软件执行，但必须受用户确认、目标校验、runner 契约、审计和反馈门禁控制 | 仿真验收链路已完成；真实测试窗口受控发送待完成 |
| 泛化目标 | 扩展到生活、学习、恋爱、普通社交、家庭、冲突修复、长期陪伴和主动进程安排 | 理论边界和问题台账已记录；目标对象四时间窗专家分析和规则型权重整合已进入代码，生产级场景矩阵、目标状态机和真实反馈校准待实现 |

### 2. 详细总目标

系统总目标是：

```text
基于用户目标、人物关系、历史事件、当前上下文、行为准则、文化语境、用户偏好、能力边界和反馈证据，
为用户提供可解释、可人工确认、可回写、可审计的社交辅助建议，
并在所有外部平台动作前保持 dry-run、用户确认、权限、目标校验和审计门禁。
```

这个目标包含五个不可拆开的要求：

1. 目标导向：所有建议必须服务用户明确目标，不做无目标闲聊替代。
2. 关系敏感：建议必须理解人物关系、阶段、角色、亲疏、权力、互惠和风险。
3. 证据驱动：建议必须绑定 RawEvent、SemanticEvent、身份确认、上下文快照或用户输入证据。
4. 可执行但不越权：推荐必须尽量落成 `message_draft`、下一步计划、确认清单和反馈入口，但不得绕过用户确认真实发送。
5. 闭环学习：执行结果和反馈必须回写事件、关系、指标、目标状态和审计记录，用于后续修正。

### 3. 理论流程

当前理论流程固定为：

```text
输入接收
-> 标准化转译
-> 解析识别
-> 身份确认
-> 图谱记忆
-> 上下文编排
-> 目标策略
-> 专家评估
-> 触发计划
-> 人工确认 / dry-run 预览
-> 外部工具或人类执行
-> 反馈记录
-> 回写事件和关系
-> 索引重建
-> 审计和状态上报
```

该流程的底层原则是“三层审计”：

```text
理论层
-> 代码层
-> 理论/代码对齐层
-> 流程树 / Obsidian / 审计产物
```

理论层回答“为什么这样做、边界是什么、哪些暂不实现”；代码层回答“怎么跑、怎么测、失败在哪里停”；对齐层回答“文档说的和代码跑的是不是一套”。

### 4. 实现方向

当前实现方向不是直接做全自动 agent，而是采用：

```text
确定性工作流 + 结构化 schema + 可验证 runtime + LLM 节点预留 + 人工确认门禁
```

后续实现应按以下顺序推进：

1. 先补足实时输入链路的稳定身份和上下文抽取。
2. 再落地 `ContextSnapshot` 编排器，明确每次读取哪些记忆、为什么读取。
3. 再补目标状态机和关系/事件指标更新纯函数。
4. 再扩展专家矩阵和场景权重校准。
5. 最后把心跳调度、主动提醒和真实连接器纳入 dry-run/审计门禁。

### 3. 当前代码规则

1. 外部平台交互默认 dry-run，不真实发送。
2. 真实发送必须有用户确认、目标校验、权限、runner 环境契约、结果回执和审计记录。
3. 来源输入必须先变成 `IntakeObservation`，再映射为 `RawEvent`，不能由策略层直接读取原始私密内容。
4. 身份未确认时只能生成 `PersonMatchCandidate` 和确认队列，不能自动合并人物事实。
5. 只有 confirmed 且 verified 的 `PersonIdentityLink` 才能自动回填 `linked_person_ids`。
6. 决策推荐必须尽量落成 `message_draft`，不能只输出抽象动作名称。
7. 专家分析当前是规则型：基础 Agent 提供意见，specialist expert 提供场景审查、风险提示和硬阻断；`expert_matrix_analysis.v2` 还会让每个专家按目标对象的 `today`、`last_7_days`、`last_30_days` 和 `historical` 时间窗独立分析。
8. 权重评分当前先由 `knowledge/decision-cluster/decision-criteria.json` 的默认权重加用户偏好形成 `base_weights`，再由 `expert_weight_integration.v1` 做可审计的小幅规则型修正，输出最终 `weights` 和 `weight_revision`。
9. 专家优先级当前由 `knowledge/decision-cluster/agent-cluster.json` 的 `priority` 和场景 signal 选择；专家权重整合是可解释规则，不代表真实反馈学习后的长期专家权重。
10. 反馈必须进入 `FeedbackRecord`、回写事件和审计产物，不能只停留在对话中。
11. 新增或修改文档、schema、runtime、样例和运行产物，必须能在流程树或机器流程树中登记。
12. 修改流程树、问题台账或文件登记后必须运行 `npm run process-tree:validate`。

### 3. 真实运行状态下是否符合本质目标

在当前 B2B MVP 场景下，系统符合材料一的本质目标：

- 能把目标、关系、事件、决策、触发、反馈、回写和审计连起来。
- 能生成具体 `message_draft`，并保留人工确认。
- 能证明外部发送被阻断或进入受控仿真验收。
- 能通过流程树、回测和测试说明“文档和代码不是两套东西”。

- 对人类真实思维中的情绪、依恋、文化、身份、冲突修复和脆弱状态已有部分专家分析入口，但还没有完整长期建模。
- 对真实反馈驱动的权重校准还没有实现。
- 对真实平台长周期、多对象、多线程接入还没有充分验证。

## 材料四：人类思维方式专家评估矩阵理论方案

### 1. 总体定位

材料四调整为“双层结构”：

```text
理论探索层
-> 专家并行建模
-> 极限预测值评估
-> 独立合理性、合法性、安全性审查层
-> 可进入工程建议、人工确认或仅保留为研究输出
```

理论探索层的目标不是先判断“能不能做”，而是尽可能完整地发现：

- 这个关系和事件可能有哪些解释。
- 不同解释背后的心理、博弈、逻辑、文化、组织和情境机制是什么。
- 不同策略路径在理论上可能触发什么响应。
- 当前证据支持哪些预测，不支持哪些预测。
- 哪些未知信息会显著改变判断。
- 哪些极端边界场景能暴露系统能力、误判风险和未来工程缺口。

合理合法安全审查层不提前压缩理论探索空间。它只在理论结果生成后，对输出的使用方式、行动建议、发送草稿、平台执行和审计要求进行独立检查。这样可以避免设计阶段过早把探索能力锁死，同时保证真实落地时仍有明确的合法、安全和用户确认闸门。

### 2. 分层原则

| 层级 | 关注目标 | 不负责什么 | 主要产物 |
| --- | --- | --- | --- |
| 理论探索层 | 最大化解释空间、策略空间、预测空间和反事实空间 | 不判断真实执行许可，不直接决定是否发送 | 假设图谱、专家意见、预测表、场景树、策略树、证据缺口 |
| 专家并行建模层 | 从不同学科和经验框架提出机制解释 | 不用单一专家压制其他专家 | 专家结构化评估、分歧点、交叉验证点 |
| 极限预测值评估层 | 只按理论预测价值排序，不按安全性提前删减 | 不把高风险等同于低理论价值 | 预测值评分、置信区间、关键观测信号 |
| 合理合法安全审查层 | 判断是否可进入行动建议、草稿、平台预览或真实执行 | 不删除理论研究记录，不替代理论评分 | 审查结论、阻断原因、人工确认项、合规替代表达 |

理论探索层可以讨论所有可能机制，但必须标注“研究性假设”和“可执行建议”的区别。真实对外动作只能进入审查层之后再决定。

### 4. 问题探究机制

理论探索层的问题设计应减少预设限制，先把可能性找全，再交给审查层决定可用边界。

| 阶段 | 允许提出的问题 | 排序依据 | 禁止在本阶段做什么 |
| --- | --- | --- | --- |
| 发散假设 | 还有哪些解释可能成立，最反直觉的解释是什么 | 解释覆盖度、证据贴合度、反事实强度 | 不用安全理由提前删除假设 |
| 机制建模 | 心理、博弈、逻辑、文化、组织机制如何共同作用 | 机制连贯性、可观测信号、跨专家一致性 | 不把单一学科解释当最终答案 |
| 策略空间 | 如果采取不同表达、等待、询问、让步、退出或第三方路径，会触发什么 | 响应可预测性、反馈可验证性、路径差异度 | 不直接输出真实执行许可 |
| 反事实推演 | 如果关键事实相反，结论是否改变 | 鲁棒性、敏感变量、边界条件 | 不把当前证据不足包装成确定性 |
| 极端边界 | 在最有利、最不利、最模糊、最高冲突条件下会怎样 | 极端发现价值、系统误判暴露能力 | 不把极端理论场景直接转为行动建议 |
| 审查转译 | 哪些理论结果可以进入现实建议，哪些只能保留为研究结论 | 合法性、安全性、合理性、用户确认 | 不反向修改理论探索记录 |

理论探索层的问题模板：

1. 这件事至少有哪 3 到 7 种可竞争解释。
2. 每种解释需要哪些证据才能成立或被推翻。
3. 哪个专家框架能解释当前最多的现象。
4. 哪些信号一旦出现，会让预测方向改变。
5. 如果用户采取完全不同的策略，对方、第三方和长期关系会如何变化。
6. 哪些判断是心理预测，哪些是博弈预测，哪些只是语言或文化解释。
7. 哪些内容只有理论价值，不能直接进入真实行动。

### 4.1 目标对象阶段性上下文分析机制

对每个目标对象，专家矩阵必须先按阶段性上下文分析，再进入汇总。系统不能只把最新一句话交给专家，而要让每个专家至少看到四个时间窗：

| 时间窗 | 主要问题 |
| --- | --- |
| `today` | 今天发生了什么、当前语气和即时触发是什么、是否有需要立刻响应或暂缓的信号 |
| `last_7_days` | 近一周互动节奏如何、主动/被动是否变化、是否出现重复线索或短期压力 |
| `last_30_days` | 近一个月关系阶段、目标推进、冲突、冷却、升温或承诺变化如何 |
| `historical` | 更长期的关系模式、角色位置、边界、偏好、旧假设和已验证经验是什么 |

每个专家都需要独立分析这些时间窗。例如：

- 心理学专家关注情绪、依恋、回避、压力、动机和行为偏差在不同时间窗中的变化。
- 博弈论专家关注信号、承诺、收益结构、让步、退出选择和重复互动格局。
- 逻辑学专家关注前提、结论、反例、因果链和是否把假设误当事实。
- 证据与因果专家关注哪些内容是事实、哪些是推断、哪些只是相关而非因果。
- 亲密关系专家关注靠近、回避、边界、承诺、拒绝和修复信号的阶段性演化。

工程侧已经把该机制落为 `context_snapshot.target_context_windows` 和 `expert_independent_context_analysis.v1`。专家结果再汇总到 `expert_weight_integration.v1`，形成 `criterion_adjustments`、`integrated_weight_preview` 和最终 `weight_revision`。这属于规则型、可解释的权重整合；它不等同于真实反馈学习。

未来新增的系统目标必须追加到理论实现目标集合，并同步进入专家矩阵分析目标、schema/文档登记和流程树校验，不能为了新目标删除旧目标或把旧目标静默替换掉。

### 5. 理论预测值极限指标

当前专家输出在理论探索层只按“理论预测值”排序。合理性、合法性和安全性不参与这一层的删减，只在后置审查层处理。

| 指标 | 含义 | 高分表现 |
| --- | --- | --- |
| 解释覆盖度 | 能解释多少已知事实和行为细节 | 能同时解释原话、时序、关系、情绪和外部约束 |
| 预测区分度 | 不同假设能否给出不同的未来观察 | 能明确说明下一步看什么信号区分真假 |
| 反事实鲁棒性 | 关键条件变化后结论是否仍成立 | 能指出哪些条件改变会推翻结论 |
| 策略深度 | 是否覆盖多轮互动和二阶反应 | 能预测对方回应、第三方影响和长期关系变化 |
| 信号敏感度 | 是否识别细微信号对判断的影响 | 能说明称呼、时间、语气、渠道和沉默的意义差异 |
| 不确定性清晰度 | 是否明确知道自己不知道什么 | 能列出关键未知项和补证问题 |
| 场景迁移度 | 是否能在相邻场景复用 | 能说明可迁移条件和不可迁移边界 |
| 反馈可检验性 | 是否能被后续反馈验证 | 能形成行动后观察指标和回写计划 |
| 边界发现能力 | 是否能暴露极端场景和系统盲区 | 能发现常规流程看不见的可能性和误判点 |
| 可能性广度 | 是否生成足够多样的解释和策略 | 不被单一叙事、单一专家或单一风险框架锁定 |

### 6.1 亲密关系动态回复 Playbook v1

本节对应 `packages/decision-cluster/src/decision-cluster.mjs` 中的 `intimate_relationship_reply_playbook.v1`。它先解决两个已确认问题：第一，已确认恋人主身份不得继续套用商务主模板；第二，桌面 OCR 原文必须进入决策上下文，不能只用摘要导致亲密语气、边界或修复信号丢失。

#### 6.1.1 输入和阶段识别

| 识别项 | 当前 v1 字段 | 说明 |
| --- | --- | --- |
| 主身份状态 | `relationship_stage` / `relationship_context_status` | `confirmed_romantic_partner` 优先于所有次级场景；未确认时只进入 `candidate_intimate_relationship` 或 `general_personal_social`。 |
| 上下文阶段 | `context_stage` | 包含 `light_affection_play`、`relationship_definition_probe`、`repair_or_tension`、`boundary_review`、`short_context_collecting`、`warm_everyday_continuation`。 |
| 双方互动风格 | `interaction_style` | 区分玩笑亲密、关系定义试探、修复优先、边界优先和低压力日常承接。 |
| 语气策略 | `tone_strategy` | 生成草稿时明确采用温暖承接、低承诺确认、慢速修复、温和边界或上下文收集。 |
| 上下文窗口 | `dynamic_context_basis.target_context_windows` | 默认要求专家按今天、一周内、一个月内和历史阶段性上下文分别判断，避免只看最新一句。 |

#### 6.1.2 模板选择原则

| 条件 | 首选模板 | 代表草稿方向 |
| --- | --- | --- |
| 已确认恋人 + 轻松亲密调侃 | `confirmed_playful_affection` | 承接亲密玩笑，保持温暖、轻松、低压力。 |
| 未确认关系 + 关系定义试探 | `candidate_relationship_probe` | 用候选式、可玩笑化表达回应，不替双方做关系结论。 |
| 已出现难受、误会、生气或修复意图 | `repair_or_tension` | 先降速，接住感受，再请求继续沟通。 |
| 出现越界、拒绝、隐私或压力信号 | `boundary_review` | 承接但不升级，把对方舒适度和自主性放在第一位。 |
| 已确认恋人但上下文不足 | `confirmed_warm_presence` | 给稳定感，不急于定义或推进。 |
| 普通私人社交上下文不足 | `general_warm_collecting` | 轻量回应，继续收集信号。 |

#### 6.1.3 多发展路径和样式

`possible_developments` 不是发送指令，而是专家和用户审查草稿时的候选路径：

| 路径 | 触发条件 | 样式要求 | 调整条件 |
| --- | --- | --- | --- |
| `continue_light_affection` | 对方继续亲昵称呼、玩笑或主动靠近 | 温暖承接、短句、可玩笑化 | 对方降温、不适或拒绝时停止推进。 |
| `clarify_relationship_expectation` | 对方主动追问关系定义、承诺或排他性 | 先确认感受，再共同确认定义 | 证据不足或用户不想推进时不启用。 |
| `repair_or_deescalate` | 出现误会、生气、冷淡、委屈或节奏失配 | 降速、接住感受、请求沟通 | 不用玩笑压过真实不适。 |
| `set_soft_boundary` | 出现越界、成人玩笑、隐私或压力信号 | 承接但不升级，舒服节奏优先 | 对方明确要求停止时直接停止。 |
| `collect_more_context` | 只有单句短上下文 | 轻量回应并等待下一条信号 | 不从单句推断长期关系事实。 |

`alternative_drafts` 必须至少覆盖更温柔、更调侃、更慢速边界三类样式，供用户在发送前人工切换。所有样式仍受 `must_confirm_before_send=true` 和真实发送阻断约束。

#### 6.1.4 亲密关系确认框架

亲密关系模板进入可用草稿前必须经过五组确认：

1. 身份确认：目标人物来自已确认图谱或本轮用户确认；不得用临时候选覆盖已确认人物。
2. 主次身份确认：主身份若为恋人，恋人模板优先；商务、朋友、工作、家庭等只作为次级场景修饰，不得反客为主。
3. 阶段上下文确认：专家分别读取今天、一周、一个月和历史阶段上下文，判断亲密、回避、修复和边界信号是否稳定。
4. 语气和边界确认：草稿必须低压力、可编辑、可撤回，不制造承诺，不替对方定义关系；边界和拒绝信号优先于推进。
5. 发送和反馈确认：真实发送前仍需用户确认窗口、草稿、平台预览和发送动作；发送或放弃发送后都要回写对方回应、用户反馈和关系权重调整。

#### 6.1.5 其他身份模板的复用确认流程

未来新增普通朋友、客户、同事、家人、老师、学生、供应商、社群成员等身份模板时，不复用亲密关系草稿内容，但复用同一套流程：

```text
identity evidence
  -> primary relationship identity
  -> secondary scene modifiers
  -> context windows
  -> specialist expert analysis
  -> playbook family selection
  -> message_draft + alternatives
  -> human confirmation gate
  -> send/dry-run audit
  -> feedback and graph writeback
```

每一种身份模板都必须先定义：

- 主身份优先级和可叠加的次级身份。
- 专属阶段机，例如客户阶段、朋友熟悉度、家人边界、同事协作责任。
- 专属风险信号，例如商务承诺、情感压力、家庭义务、职场权责或公开传播风险。
- 专属 `possible_developments` 和 `alternative_drafts`。
- 与人物图谱、事件图谱和反馈记录的回写字段。

这保证未来身份扩展不会再次修改接入主链路，只是在 playbook family 中增加身份专属模板和专家权重规则。

### 6.2 恋人关系目标分层与审查阀门待确认方案

本节原本是 PT-028 的下一阶段目标方案。当前核心运行时已按用户确认后的方向接入 `romantic_goal_analysis.v1`、`romantic_expert_sentence_review.v1`、关系梯度、心理舒适度、句子意图合同和第三方提示；`pt028_gui_decision_state.v1` 已把真实 runtime 决策投影到 GUI，展示关系梯度、句子意图、第三方提示卡、完整链路走向和分支记录；仍未完成的是低延迟预览、真实反馈校准和最终专项验收。它用于持续约束“恋人关系到底朝什么方向分析、哪些目标可以执行、哪些信号必须提示用户、什么时候才允许生成草稿或进入受控发送”的实现边界。

#### 6.2.1 核心目标边界

系统可以把“恋人关系发展”建模为关系阶段、亲密程度、边界状态、信任状态、风险状态和反馈状态的组合。根据用户对 PT-028 审查清单的修正，本目标需要分成两层：目标能力层先尽可能完整地识别和推动关系发展；审核安全层再给出合法性、安全性、隐私、平台和我方行为风险评估。

在目标能力层，双方确认后的生理亲密关系应作为恋人关系确认的目标状态之一，用于判断关系确定性、真诚度和关系图谱连续性。它仍不应被写成自动发送 KPI、强制推进指标或绕过用户确认的动作指标。工程目标应表述为：

```text
以用户作为关系的主动推动者，帮助用户理解关系阶段、组织多窗口上下文、识别下一步关系推进流程，
并把双方确认后的生理亲密关系作为关系确认目标状态之一纳入图谱和策略判断。
审核安全层独立输出合法性、安全性、隐私、平台和我方行为风险评估；真实发送仍必须经过用户确认和受控发送门禁。
```

因此，`physical_intimacy_confirmed` 应同时是关系图谱的目标状态字段和双方确认后的结果字段，但不能被解释为自动发送 KPI 或强制推进动作。当前第一版暂不把因生理、心理、价值观、文化或其他个人原因无法进入生理亲密关系的情况作为默认约束，但必须预留后续拓展接口。任何涉及真实发送、敏感存储或对外草稿的行为，仍由独立审核安全层输出风险评估、人工复核建议、降级建议或阻断建议。

#### 6.2.2 恋人关系目标分层

| 层级 | 状态名 | 判断依据 | 系统可做 | 系统不得做 |
| --- | --- | --- | --- | --- |
| R0 | `unconfirmed_person_or_context` 未确认人物或上下文 | 目标人物、年龄、关系身份或窗口对象不清楚 | 只做人身份确认、上下文补全和风险提示 | 不生成亲密推进草稿，不写入已确认恋人事实 |
| R1 | `candidate_romantic_interest` 期望发展但未确认恋人 | 暧昧、好感、关系试探、低强度互动 | 生成低压力、可撤回、候选关系沟通草稿 | 不替双方确认恋人关系，不推进身体亲密 |
| R2 | `confirmed_romantic_no_physical_intimacy` 已确认恋人但无实质肢体亲密证据 | 用户确认恋人身份，但历史记录未显示牵手、拥抱、接吻等 | 支持关心、陪伴、约会、边界沟通和节奏确认 | 不把恋人身份自动等同于身体亲密同意 |
| R3 | `early_physical_affection` 初步身体亲密 | 有牵手、拥抱、依偎等非性亲密接触，且双方反馈正向 | 支持温和承接、尊重节奏、确认舒适度 | 不升级为性暗示或身体推进 |
| R4 | `romantic_kissing_or_stronger_affection` 亲吻或更强亲密但未确认性关系 | 有接吻、亲密拥抱等记录，但没有明确性关系事实 | 支持边界、节奏、情绪安全和关系期待沟通 | 不把亲吻推断为性同意 |
| R5 | `explicit_intimacy_boundary_discussion` 已开始明确讨论亲密边界 | 双方主动讨论边界、意愿、节奏、健康、安全、避孕或排他性 | 支持清晰、尊重、非施压的边界沟通草稿 | 不替用户或对方给出同意，不生成催促性内容 |
| R6 | `physical_intimacy_confirmed_relationship_goal_state` 生理亲密作为双方确认后的关系目标状态 | 双方关系确认、用户确认的事实记录、关系图谱连续性证据 | 作为关系确认目标状态、关系图谱状态和反馈事实记录；后续关注尊重、健康、安全、情绪反馈和关系稳定性 | 不公开、不外泄、不作为自动发送 KPI 或绕过审核的动作指标 |
| RX | `review_or_risk_exception` 审核层风险或边界异常状态 | 拒绝、不适、恐惧、胁迫、PUA、未成年/年龄不明、身份不明、醉酒/意识不清等 | 由审核安全层输出用户可见提醒、风险建议、求助/复核建议和审计日志 | 不绕过用户确认、目标窗口校验、审计记录或真实发送门禁 |

#### 6.2.2.1 阶段梯度科学依据与运行时映射

本轮实现把恋人关系推进抽象成可复用的 `relationship_gradient_framework.v1`。该框架未来也可迁移到销售关系、谈判关系等阶段性推进场景：销售可替换为线索、需求、方案、采购、成交、复购；谈判可替换为议题确认、利益交换、方案收敛、让步边界、协议确认。当前先实现恋人关系的 `romantic_stage_gradient.v1`。

| 理论依据 | 对应关系学科含义 | 工程映射 |
| --- | --- | --- |
| Social Penetration Theory：关系从浅层互动逐步走向更深自我披露，参考 Altman & Taylor 1973 的社会渗透模型资料[^spt] | 不能只看最新一句；要看披露的广度、深度、互惠和持续性 | `semantic_feature_assessment.features.self_disclosure_breadth_depth`、`multi_window_evidence_coverage` |
| Sternberg 三角爱情理论：亲密、激情、承诺共同决定爱情类型[^sternberg] | 恋人推进不能由单一身体目标驱动，也不能只停留在承诺标签 | `affective_heat`、`commitment_or_investment`、`physical_intimacy_goal_state` 分开记录 |
| 投入模型：满意度、替代选择和投入影响承诺与关系稳定[^rusbult] | 阶段升级要看投入、历史互动和反馈，而不是单轮热度 | `commitment_or_investment`、`feedback_to_record`、四时间窗专家分析 |
| 伴侣回应性 / 亲密过程模型：被理解、被确认、被照顾会提升亲密和舒适[^responsiveness] | 草稿强度要由对方回应性、情绪安全和舒适度决定 | `reciprocity_and_responsiveness`、`psychological_comfort_model.comfort_score` |
| 同意与自主原则：同意应自由、可撤回、知情、积极、具体，亲密动作不能由身份推断[^consent] | 身份为恋人不等于身体亲密同意；涉及身体靠近时要询问舒适度 | `consent_readiness`、`boundary_and_autonomy`、`stage_transition_assessment` |

运行时合同：

| 合同 | 作用 |
| --- | --- |
| `relationship_gradient_framework.v1` | 通用阶段推进框架，适用于恋人、销售、谈判等需要梯度推进的关系 |
| `romantic_stage_gradient.v1` | 恋人关系 R0-R6/RX 阶段目录、允许话语行为和升级策略 |
| `romantic_semantic_feature_assessment.v1` | 从聊天记录、原始事件和四时间窗综合判断语义特征；词表只作为规则版证据，不是固定判词 |
| `psychological_comfort_model.v1` | 输出当轮热度、舒适度和推进强度，避免为了推进而推进 |
| `romantic_stage_transition_assessment.v1` | 判断是否具备进入下一阶段的特征，区分“可微推进”和“可升级复核” |
| `dialogue_intent_contract.v1` | 解释每句用户第一人称草稿的话语行为、意图、阶段变化、热度变化、退路和预期反馈 |
| `third_party_target_reply_prompt.v1` | 自动代输入被阻断时，对目标对象每一句回复展示给用户看的第三方提示，不发送给目标对象 |

#### 6.2.4 反 PUA / 洗脑风险专家

新增专家建议命名为 `coercion_and_pua_risk_expert`。根据用户审查意见，该专家职责应重新界定为：主要检测目标对象是否对用户表现出孤立、羞辱、愧疚施压、威胁、隐私勒索、社交隔离、数据控制等操控行为。用户侧草稿或目标是否存在威胁、羞辱、诱导、虚假承诺、越界试探等问题，不耦合进该专家，而交给独立的 `user_side_manipulation_reviewer` 或既有“我方安全行为审核模块”处理。

| 信号族 | 典型信号 | 权重规则 |
| --- | --- | --- |
| 隔离与控制 | 要求远离朋友家人、限制社交、检查手机、要求报备 | 多窗口反复出现时快速升权 |
| 煤气灯 / 认知扭曲 | 否认已发生事实、让用户怀疑自己、反复改写责任 | 与情绪低落或道歉循环同时出现时升权 |
| 爱轰炸与冷热交替 | 极端热情后突然冷暴力、用奖惩控制回应 | 频率越高、周期越短，风险越高 |
| 愧疚与威胁 | “你爱我就应该…”，分手威胁，自伤威胁，公开羞辱威胁 | 出现威胁或自伤时进入高风险审查 |
| 身体/性边界压力 | 对用户的拒绝不尊重、催促身体亲密、把恋人身份等同于同意 | 进入审核安全层的高风险评估 |
| 财务和资源剥削 | 借钱、索要礼物、共同财务绑定、用感情施压 | 与隔离/愧疚信号叠加时升权 |
| 数据与隐私控制 | 要求交出账号密码、定位、私密照片或聊天记录 | 默认高风险，需用户可见提醒 |

用户侧行为风险的职责拆分：

| 模块 | 职责 | 不属于 |
| --- | --- | --- |
| `coercion_and_pua_risk_expert` | 检测目标对象对用户的 PUA、洗脑、胁迫、隐私勒索、孤立、羞辱、愧疚施压等风险 | 不检查用户草稿是否操控对方 |
| `user_side_manipulation_reviewer` 或我方安全行为审核模块 | 检查用户侧草稿、目标和发送动作是否存在威胁、羞辱、诱导、虚假承诺、越界试探、隐私侵犯等风险 | 不参与恋人关系阶段判断，不替代目标侧反 PUA 专家 |

临时权重建议：

```text
pua_risk_score =
  signal_severity
  + multi_window_persistence
  + escalation_speed
  + power_imbalance
  + consent_or_boundary_violation
  + user_distress_signal
  - clear_repair_and_respectful_boundary_evidence
```

输出等级：

| 等级 | 条件 | 系统动作 |
| --- | --- | --- |
| `low` | 偶发轻微信号，无持续控制或边界破坏 | 记录为观察项，不打断普通草稿 |
| `watch` | 多个弱信号或单个中等信号 | 在用户可见日志提示，草稿降压 |
| `warning` | 多窗口持续信号、冷热控制、愧疚施压或边界忽视 | 草稿需审查，默认不进入真实发送 |
| `critical` | 威胁、自伤、强迫、未成年/年龄不明、意识不清、隐私勒索、身体边界被持续侵犯 | 只允许用户可见安全提示和人工复核，不生成对外发送草稿 |

#### 6.2.5 用户可见日志

反 PUA、边界、同意和目标分析结果不一定进入聊天框，但必须能进入用户可见日志，例如 `relationship_safety_log.v1`：

| 字段 | 说明 |
| --- | --- |
| `log_id` | 日志 ID |
| `target_person_id` | 目标人物 |
| `visible_to_user` | 默认为 true |
| `visible_to_target` | 默认为 false |
| `risk_level` | low/watch/warning/critical |
| `risk_signals` | 被触发的信号族 |
| `why_it_matters` | 用自然语言说明为什么提醒用户 |
| `recommended_pause_or_review` | 是否建议暂停、降级、人工复核 |
| `not_inserted_into_chat_reason` | 为什么不把该提醒直接发给对方 |
| `next_safe_action` | 可选的下一步，例如补上下文、询问用户确认、停止发送 |

日志原则：

- 不把“风险提醒”自动发送给目标对象，避免制造冲突或暴露用户分析。
- 风险日志必须让用户看得见，不能只存在模型内部。
- 高风险时，日志优先于草稿，草稿生成应被降级或阻断。
- 如果系统发现用户自身有操控性表达，也要以用户可见方式提示并建议改写。

#### 6.2.6 三层阀门

恋人关系流程必须严格按顺序通过三层阀门：

1. 目标分析结果审查：先审查 `romantic_goal_analysis.v1`，判断阶段、目标、证据、同意条件、PUA/边界风险和允许输出等级。
2. 发送内容审核：只有 `allowed_output_level` 至少为 `draft_allowed` 时，才允许生成 `message_draft`；草稿必须通过非操控、低压力、边界尊重、身份匹配、隐私保护和平台规则检查。
3. 真实发送门禁：只有草稿审核通过、用户确认、目标窗口确认、平台预览到达、发送按钮位置/runner 合约确认、审计记录准备完成时，才可进入受控发送；否则保持 `real_execution_allowed=false`。

阀门示意：

```text
IntakeObservation
  -> identity_continuity
  -> romantic_goal_analysis.v1
  -> coercion_and_pua_risk_expert
  -> user_visible_safety_log
  -> target_analysis_review_gate
  -> message_draft_generation
  -> draft_safety_review_gate
  -> platform_preview_gate
  -> user_confirmed_controlled_send
  -> send_result / feedback_writeback
```

审核安全层建议与发送门禁条件：

- 任一方未成年、年龄未知且涉及身体/性亲密目标。
- 身份、窗口对象或关系主身份不明确。
- 明确拒绝、不适、恐惧、威胁、胁迫、醉酒/意识不清、隐私勒索或强迫信号。
- 我方安全行为审核模块显示用户草稿带有羞辱、威胁、操控、虚假承诺、情绪勒索或越界试探。
- 未完成用户确认、平台预览、目标窗口校验和审计记录。

#### 6.2.13 真实恋人关系发展路径状态机

本状态机用于把 R0-R6/RX 从“阶段枚举”变成可执行的关系发展路径。它不把任何身体亲密作为系统强推目标，而是把关系推进建模为“证据出现 -> 专家分析 -> 用户可见日志 -> 审查阀门 -> 可选草稿或阻断”的连续过程。

全局前置条件：

1. 目标人物、聊天窗口、关系主身份和上下文来源必须明确。
2. 涉及身体或性亲密时，必须能证明双方为成年人或进入保守阻断。
3. 任一阶段都要求无胁迫、无威胁、无醉酒/意识不清、无拒绝被忽视、无隐私勒索。
4. 系统只能根据证据判断当前阶段和允许输出等级，不能替用户或对方表达同意。
5. 阶段升级必须基于双方互动证据和用户确认；单句暧昧、玩笑或模型推断不得升级阶段。

| 当前阶段 | 可迁移到 | 必要证据 | 专家审查重点 | 允许输出 | 禁止输出 |
| --- | --- | --- | --- | --- | --- |
| R0 未确认人物或上下文 | R1 或 R2 | 人物身份、窗口对象、年龄安全、关系标签来源明确 | 身份解析、误识别、场景混淆 | 身份确认、上下文补全、低压力询问 | 亲密推进、恋人事实写入、真实发送 |
| R1 候选暧昧对象 | R2 | 双方明确承认恋人关系，或用户确认并有对话支撑 | 关系定义是否被单方替代、是否施压 | 关系期待沟通、低承诺约会建议、边界尊重草稿 | 身体亲密推进、替对方确认关系 |
| R2 已确认恋人但无肢体亲密证据 | R3 | 双方正向、自然、非施压的牵手、拥抱、依偎等非性亲密证据 | 节奏、舒适度、情绪安全、是否把恋人等同于同意 | 关心、陪伴、约会、轻松亲密承接、询问舒适度 | 性暗示、身体边界试探、确认生理关系 |
| R3 初步身体亲密 | R4 | 接吻或更强亲密互动证据，且无拒绝/不适信号 | 是否尊重边界、是否存在冷热奖惩或催促 | 温和承接、确认舒服程度、关系期待沟通 | 把牵手/拥抱推断为更高亲密同意 |
| R4 接吻或更强亲密但未确认性关系 | R5 | 双方主动讨论亲密边界、节奏、健康、安全、排他性或避孕 | 同意清晰度、压力、权力不平衡、隐私保护 | 边界沟通草稿、健康安全提醒、人工复核 | 催促性内容、含糊施压、默认同意 |
| R5 明确讨论亲密边界 | R6 | 双方成年人、清醒、自愿、可撤回、无胁迫，并由用户确认事实 | 同意是否明确、是否可撤回、是否存在操控或恐惧 | 只记录状态事实、后续尊重与反馈建议 | 把 R6 作为炫耀、外发或继续施压材料 |
| 任意阶段 | RX | 拒绝、不适、恐惧、胁迫、威胁、自伤压力、隐私勒索、年龄未知且涉及身体亲密、窗口身份不明 | 安全、停止、求助、审计 | 用户可见安全日志、暂停建议、人工复核 | 任何对外推进草稿或真实发送 |
| RX 风险阻断 | R0-R3 中较低安全阶段 | 风险解除、用户复核、上下文补齐、对方明确尊重边界 | 是否真的解除风险，是否只是短期缓和 | 观察、修复、低压力道歉或边界草稿 | 直接恢复到高亲密阶段或真实发送 |

阶段输出策略：

| 阶段 | 默认目标 | 默认输出等级 | 草稿风格 | 用户可见日志 |
| --- | --- | --- | --- | --- |
| R0 | 识别人物和场景 | `analysis_only` | 不生成亲密草稿 | 说明缺失身份或窗口证据 |
| R1 | 低压力关系探索 | `draft_allowed` 或 `user_visible_log` | 尊重、可退、不给对方压力 | 提醒不要替双方确认恋人关系 |
| R2 | 稳定恋人互动和节奏确认 | `draft_allowed` | 关心、陪伴、轻松亲密、约会安排 | 提醒恋人身份不等于身体同意 |
| R3 | 承接初步亲密并确认舒适度 | `draft_allowed`，敏感时 `preview_allowed` | 温和确认、尊重节奏、可撤回 | 记录亲密证据来源和缺失边界 |
| R4 | 亲密边界和期待沟通 | `user_visible_log` 或 `preview_allowed` | 边界、健康、安全、清晰表达 | 提醒接吻不等于性同意 |
| R5 | 明确亲密边界审查 | `analysis_only` 或人工复核后 `preview_allowed` | 只允许非施压、明确尊重选择 | 记录同意条件缺口 |
| R6 | 已确认事实后的尊重与反馈 | `analysis_only` 或关系健康建议 | 不把事实外发，不做施压 | 仅写摘要状态和隐私保护说明 |
| RX | 风险停止 | `send_blocked` | 不生成对外草稿 | 必须写用户可见风险日志 |

多窗口证据组织：

| 窗口 | 用途 | 示例 |
| --- | --- | --- |
| 今天 | 判断最新语气、风险和是否需要即时回复 | 最新一句、当前情绪、是否有拒绝或不适 |
| 近 7 天 | 判断互动节奏和短期变化 | 频率、热度、突然冷淡、连续邀约或回避 |
| 近 30 天 | 判断关系趋势和重复信号 | 关系定义、约会记录、边界讨论、矛盾修复 |
| 历史阶段 | 判断长期关系事实和稳定标签 | 是否已确认恋人、是否多次尊重边界、是否存在长期操控 |

阶段升级置信度不等于发送许可。即使阶段置信度较高，只要草稿涉及身体边界、风险等级达到 `warning`、用户未确认、窗口未校验或平台预览未到达，仍必须保持 `real_execution_allowed=false`。

兮颜测试夹具在本状态机下仍停留在 R2：已确认恋人身份标签，但当前证据只支持轻松亲密承接，不支持 R3/R4/R5/R6 升级。后续如果出现更多信号，系统应先把新增证据写入事件图谱和用户可见日志，再由专家矩阵判断是否仍是 R2、是否可进入 R3/R4，或是否因为风险信号进入 RX。该结论只约束测试夹具，不得作为其他对象的默认关系阶段。

#### 6.2.14 恋人关系专家能力配置与升权机制

恋人关系场景不能只依赖单一“反 PUA 专家”。后续实现应把 PT-028 作为一个场景专家组：每个专家独立读取同一个目标对象的四个时间窗，先输出自己的判断和证据，再由权重整合层合并。这样既保留理论探索能力，也能在风险信号增加时可解释地升权。

专家能力配置：

| 专家 ID | 学科/能力 | 必读时间窗 | 主要判断 | 输出字段 | 何时升权 |
| --- | --- | --- | --- | --- | --- |
| `relationship_stage_expert` | 亲密关系阶段识别 | today / last_7_days / last_30_days / historical_stage | R0-R6/RX 阶段、阶段升级或降级依据 | `stage_hypothesis`、`stage_confidence`、`stage_evidence`、`stage_missing_evidence` | 阶段证据跨窗口稳定，或新证据改变主阶段 |
| `attachment_psychology_expert` | 心理学、依恋、情绪调节 | 四窗口必读 | 靠近/回避、安全感、焦虑、修复意愿 | `attachment_pattern_hypothesis`、`emotional_state`、`repair_readiness` | 情绪波动、回避/追逐循环、压力反应持续出现 |
| `game_theory_signal_expert` | 博弈论、信号与承诺 | 四窗口必读 | 主动性、承诺成本、让步、退出选择、重复互动 | `signal_interpretation`、`commitment_cost`、`likely_response_paths` | 关系定义、邀约、承诺、撤回或冷淡形成稳定策略信号 |
| `logic_and_evidence_expert` | 逻辑学、证据链、反事实 | 四窗口必读 | 哪些是事实、哪些是推断，是否把暧昧误判为同意 | `fact_inference_split`、`contradictions`、`evidence_strength` | 证据缺口影响阶段判断或发送许可 |
| `consent_and_boundary_expert` | 同意、边界、自主性审核 | 四窗口必读 | 成年、清醒、自愿、可撤回、无压力、拒绝是否被尊重 | `consent_readiness`、`boundary_status`、`review_recommendations` | 涉及身体亲密、拒绝、不适、边界讨论或含糊压力 |
| `coercion_and_pua_risk_expert` | 目标侧反 PUA、洗脑、操控识别 | 四窗口必读 | 目标对象对用户的隔离、控制、煤气灯、冷热奖惩、威胁、隐私控制 | `risk_level`、`risk_score`、`triggered_signal_families`、`review_recommendations` | 多信号叠加、多窗口持续、目标对象侵犯边界或对用户施压 |
| `privacy_and_safety_expert` | 隐私、安全、平台与现实风险 | 四窗口必读 | 敏感内容存储、截图/转发风险、真实发送风险 | `privacy_risk`、`storage_policy`、`send_gate_risk` | 涉及私密事实、账号定位、真实发送、截图传播或外部平台动作 |
| `communication_pragmatics_expert` | 语言语用、语气、面子和低压力表达 | 四窗口必读 | 草稿是否自然、尊重、不催促、不越级 | `tone_fit`、`pressure_level`、`draft_rewrite_advice` | 草稿可能误读、过度亲密、过冷/过热或越过当前阶段 |
| `feedback_learning_expert` | 反馈学习与权重校准 | 四窗口必读，加发送后反馈 | 对方回应、用户感受、行动结果如何更新下轮判断 | `feedback_hypothesis_update`、`next_observation_targets` | 有发送/未发送反馈、对方回应、沉默、拒绝、推进或用户评分 |

用户侧行为审核不纳入恋人关系专家矩阵，而作为发送前独立审核模块：

| 模块 ID | 作用位置 | 主要判断 | 输出 |
| --- | --- | --- | --- |
| `user_side_manipulation_reviewer` 或我方安全行为审核模块 | 草稿生成后、真实发送前，也可读取目标分析结果 | 用户侧是否存在威胁、羞辱、诱导、虚假承诺、越界试探、隐私侵犯等风险 | `user_side_risk_level`、`rewrite_required`、`manual_review_required`、`send_gate_recommendation` |

每个专家的最小输出：

| 字段 | 含义 |
| --- | --- |
| `expert_id` | 专家 ID |
| `window_findings` | 对 today / last_7_days / last_30_days / historical_stage 的独立结论 |
| `evidence_refs` | 引用的 RawEvent、SemanticEvent、关系记录或用户确认 |
| `confidence` | 0-1，表示该专家对自己结论的证据充分度 |
| `weight_delta` | 对相关判断维度的建议升/降权 |
| `review_recommendations` | 审核安全层建议，包括风险提示、人工复核、降级或阻断建议 |
| `user_visible_summary` | 可进入用户可见日志的简短解释 |

权重整合规则：

```text
romantic_expert_weight_delta =
  stage_relevance
  + evidence_strength
  + multi_window_consistency
  + signal_convergence
  + recency_pressure
  + user_goal_relevance
  - contradiction_penalty
  - missing_evidence_penalty
```

| 因子 | 升权条件 | 降权条件 |
| --- | --- | --- |
| `stage_relevance` | 专家判断直接影响当前 R 阶段或输出等级 | 与当前阶段无关 |
| `evidence_strength` | 有明确对话、用户确认或图谱事实 | 只有模型推断或单句玩笑 |
| `multi_window_consistency` | 今天、一周、一月、历史窗口出现同向信号 | 只在一个窗口短暂出现 |
| `signal_convergence` | 多个专家从不同角度指向同一风险或阶段 | 专家结论分歧大 |
| `recency_pressure` | 最新消息需要即时回应或风险快速升级 | 不是当前决策所需 |
| `user_goal_relevance` | 与用户当前确认的目标直接相关 | 与当前目标无关 |
| `contradiction_penalty` | 不适用 | 新证据与旧结论冲突 |
| `missing_evidence_penalty` | 不适用 | 缺少身份、年龄、窗口、同意或边界证据 |

风险升权阈值：

| 目标等级 | 触发条件 | 系统动作 |
| --- | --- | --- |
| `low` | 单个弱信号、无持续性、无边界破坏 | 可继续普通草稿，但保留观察 |
| `watch` | 两个以上弱信号，或一个中等信号，或同一弱信号跨两个窗口出现 | 写入用户可见日志；草稿降压 |
| `warning` | 三个以上信号族叠加，或同一风险跨三十天持续，或边界/隐私/财务压力出现 | 目标分析必须人工复核；草稿默认需审核；真实发送阻断 |
| `critical` | 威胁、自伤压力、明确拒绝被忽视、身体/性边界压力、隐私勒索、未成年或年龄不明且涉及身体亲密 | 只允许用户可见安全提示和人工复核；阻断对外草稿和真实发送 |

升权后的输出约束：

| 风险等级 | 聊天框输出 | 用户可见日志 | 草稿 | 真实发送 |
| --- | --- | --- | --- | --- |
| `low` | 可正常展示候选草稿 | 可选记录 | 可生成 | 仍需普通发送闸门 |
| `watch` | 不强制打断 | 必须记录 | 可生成但降压 | 默认不自动进入受控发送 |
| `warning` | 不把风险诊断发给对方 | 必须记录，并建议复核 | 必须审核或改写 | 阻断 |
| `critical` | 不生成对外内容 | 必须记录，并提示暂停/求助/复核 | 阻断 | 阻断 |

如果专家之间出现冲突，处理顺序为：

1. `consent_and_boundary_expert`、`coercion_and_pua_risk_expert`、`privacy_and_safety_expert` 的审核安全建议优先进入人工复核和发送门禁评估。
2. `logic_and_evidence_expert` 发现关键证据缺口时，阶段升级只能降级为候选假设。
3. `communication_pragmatics_expert` 只能修改表达，不得绕过目标分析审查或真实发送闸门。
4. `feedback_learning_expert` 只能根据真实反馈校准未来权重，不得把一次积极反馈自动升级为 R6。

#### 6.2.15 目标分析、草稿审核和真实发送审查工作流

本节把 6.2.6 的三层阀门展开成可验收工作流。对应人工审查模板已落到 `runtime/user-inputs/templates/pt028-target-draft-send-review-workflow.real.template.md`；如进入真实实现后的人工复核，应填写到 `runtime/user-inputs/pt028-target-draft-send-review-workflow.real.md`。该文件仍不开放真实发送，只提供审查证据。

第一层：目标分析审查。

| 审查项 | 通过条件 | 不通过动作 |
| --- | --- | --- |
| 目标人物和窗口 | 人物 ID、窗口或来源上下文明确 | `analysis_only`，补身份或窗口证据 |
| 主关系身份 | 主身份为候选暧昧、已确认恋人或风险阻断之一 | 不使用亲密关系草稿 |
| 阶段判断 | R0-R6/RX 明确，且事实和推断分开 | 降级为候选假设或 `analysis_only` |
| 四窗口证据 | today / last_7_days / last_30_days / historical_stage 均有专家判断或缺口说明 | 不允许阶段升级 |
| 同意与边界 | 成年、清醒、自愿、可撤回、无压力、身份确认、窗口确认已评估 | 不允许身体亲密相关草稿 |
| 风险专家 | 反 PUA、同意边界、隐私安全专家均输出风险或缺口 | 不生成外部草稿 |
| 允许输出等级 | 明确为 `analysis_only`、`user_visible_log`、`draft_allowed`、`preview_allowed` 或 `send_blocked` | 默认 `send_blocked` |

目标分析结果字段级审查：

| 字段 | 审查重点 | 失败动作 |
| --- | --- | --- |
| `target_person_id` | 是否对应稳定图谱人物，或明确说明未解析身份 | 阻断对外草稿 |
| `source_context_id` | 是否能追溯到聊天窗口、OCR observation、导入事件或平台快照 | 要求来源校验 |
| `primary_relationship_identity` | 是否为恋人、候选暧昧、未知或风险阻断之一 | 未知时不使用恋人草稿 |
| `primary_relationship_stage` | 是否为 R0-R6/RX，且没有证据不足的越级 | 降级到更安全阶段 |
| `physical_intimacy_state` | 是否只作为私密状态事实、未知、无证据或不适用，且从不作为 KPI | 阻断身体亲密相关草稿 |
| `stage_evidence` | 事实证据和模型推断是否分开 | 降级为阶段假设 |
| `stage_missing_evidence` | 身份、同意、边界、年龄、窗口或上下文缺口是否明确 | 不允许阶段升级 |
| `consent_readiness` | 涉及亲密升级时是否检查成年、清醒、自愿、可撤回、无胁迫和边界 | 进入 `send_blocked` 或人工复核 |
| `expert_findings` | 每个必需专家是否有四窗口结论或明确缺口 | 要求专家复核 |
| `pua_or_coercion_risk` | 是否包含目标对象对用户的风险等级和触发信号族 | 写用户可见日志；warning/critical 进入人工复核和发送门禁评估 |
| `user_visible_log_decision` | 是否说明是否写日志，以及 `visible_to_target=false` | 先补日志决策 |
| `allowed_output_level` | 是否明确为 `analysis_only`、`user_visible_log`、`draft_allowed`、`preview_allowed` 或 `send_blocked` | 默认 `send_blocked` |
| `draft_scope` | 若允许草稿，是否限定为低压力、可编辑、匹配当前阶段 | 阻断或改写草稿 |
| `send_gate_precondition` | 是否保持 `real_execution_allowed=false` 直到受控发送门禁全部通过 | 保持发送阻断 |

第二层：草稿安全审核。

| 审查项 | 通过条件 | 不通过动作 |
| --- | --- | --- |
| 生成前置 | 目标分析审查通过且允许至少 `draft_allowed` | 不生成草稿 |
| 阶段匹配 | 草稿不越过当前 R 阶段 | 改写或降级 |
| 压力水平 | 不催促、不情绪勒索、不制造必须回应的压力 | 改写或阻断 |
| 同意边界 | 不把恋人身份、亲吻、拥抱或玩笑推断为身体/性同意 | 阻断相关表达 |
| 操控风险 | 不威胁、不羞辱、不孤立、不虚假承诺、不诱导隐私暴露 | 阻断并写用户可见日志 |
| 隐私保护 | 不外发敏感关系事实，不暴露系统分析或风险判断 | 改写或仅保留用户日志 |
| 可编辑性 | 草稿可被用户修改，且不直接触发真实发送 | 退回预览 |

第三层：真实发送门禁。

| 审查项 | 通过条件 | 不通过动作 |
| --- | --- | --- |
| 草稿审核 | `draft_safety_review_result.v1` 通过 | `real_execution_allowed=false` |
| 用户确认 | 用户确认当前这条草稿、目标对象、窗口和发送范围 | 继续阻断 |
| 窗口校验 | 目标窗口或平台预览与目标人物一致 | 继续阻断 |
| 平台预览 | 预览中的文本和审核草稿一致 | 继续阻断 |
| 风险状态 | 没有未关闭 `warning` 或 `critical` 风险 | 继续阻断 |
| 审计记录 | 分析、草稿、预览、用户确认和发送动作均可追踪 | 继续阻断 |
| 发送范围 | 仅限已确认的测试账号或受控测试窗口 | 继续阻断 |

硬性默认规则：任一审查项缺失或不确定时，结果按更保守等级处理。也就是说，缺证据不是默认通过，而是降级为 `analysis_only`、`user_visible_log` 或 `send_blocked`。

#### 6.2.19 真实关系逐轮样例与门禁预期

本节用于把状态机、专家矩阵和三层阀门串成可审查的逐轮样例。根据用户审查意见，目标能力层应支持用户作为关系主动推动者，持续识别阶段、组织证据、提出下一步关系推进流程；审核安全层负责给出风险评估、人工复核建议、降级建议或发送门禁建议。

| 样例 | 输入变化 | 目标分析预期 | 专家与日志预期 | 草稿审核预期 | 真实发送预期 |
| --- | --- | --- | --- | --- | --- |
| R2 日常轻松互动 | 已确认恋人主身份；最新消息是轻松调侃或想见面；无肢体亲密证据 | `relationship_stage=R2`；`allowed_output_level=draft_allowed`；缺失 R3+ 证据 | 专家保持低风险；可提示“恋人身份不等于身体同意” | 只允许关心、陪伴、约会、轻松回应；不得出现身体边界试探 | 缺用户确认、窗口校验或预览时 `real_execution_allowed=false` |
| R2 到 R3 初步非性亲密 | 出现双方正向牵手、拥抱、依偎等证据；无拒绝或不适 | 可提出 `R3_hypothesis`，但必须列明证据来源和缺失项 | 同意边界专家检查舒适度；逻辑专家确认不是单方推断 | 草稿只能承接感受、询问舒适度或安排低压力见面 | 仍需三层门禁；不得自动发送 |
| R3 到 R4 接吻或更强亲密 | 有接吻或更强亲密互动证据，但未讨论边界、健康或节奏 | `relationship_stage=R4` 或候选 R4；输出等级降为 `user_visible_log` 或人工预览 | 必须提示“接吻不等于性同意”；边界专家升权 | 草稿不得催促性内容；只能做边界、舒适度和尊重节奏表达 | warning 或证据缺失时阻断 |
| R4 到 R5 边界讨论 | 双方主动讨论亲密边界、节奏、健康、安全或排他性 | 进入 `R5` 候选；目标分析必须把事实、推断、缺口分开 | 同意边界、隐私安全、目标侧反 PUA 专家必须全部输出审核建议 | 草稿需人工复核，可给出下一步关系推进流程和确认方式 | 默认进入发送门禁评估，除非受控测试窗口和全部审查通过 |
| R5 到 R6 关系目标状态确认 | 用户确认双方关系已进入生理亲密目标状态，且有可审计事实记录 | `R6` 作为关系确认目标状态、图谱状态和摘要，不作为自动发送 KPI | 隐私安全专家升权；日志只对用户可见，不进入目标聊天框 | 默认给关系健康、尊重、反馈和下一步稳定关系建议 | 不允许把 R6 事实外发或绕过发送门禁 |
| 任意阶段进入 RX | 出现拒绝、不适、恐惧、年龄未知且涉及身体亲密、威胁、隐私勒索、窗口身份不明 | `relationship_stage=RX`；进入审核安全层评估 | 目标侧反 PUA、同意边界、隐私安全专家必须输出审核建议；用户可见日志必须写入 | 必要时只给用户安全提示、补证据建议或人工复核建议 | 缺少用户确认、窗口校验或审计时 `real_execution_allowed=false` |
| 用户侧安全行为审核 | 用户要求威胁、羞辱、PUA、虚假承诺、诱导隐私或越过对方边界 | 目标分析结果交给我方安全行为审核模块复核 | `user_side_manipulation_reviewer` 或同等模块检查用户侧行为；通信语用专家给出表达建议 | 可给用户侧改写建议和风险建议 | 由独立发送门禁决定是否允许继续 |

逐轮样例的共同断言：

1. 每轮都必须先生成目标分析审查结果，再决定是否允许草稿。
2. 每个专家都必须说明四窗口证据或缺口；缺口不能被当成通过。
3. 风险日志可以进入用户可见状态区，但默认不得注入目标聊天框。
4. 草稿即使允许生成，也必须经过草稿安全审核。
5. 真实发送只有在用户确认、目标窗口、平台预览、审计记录和风险关闭全部齐备时才可进入受控测试；默认保持阻断。

### 7. 权重分配机制

专家权重不应在设计阶段固定为单一总表。理论探索层和审查层采用不同权重目标：

| 层级 | 权重目标 | 权重来源 | 更新方式 |
| --- | --- | --- | --- |
| 理论探索层 | 提高解释和预测能力 | 场景类型、证据质量、专家匹配度、历史反馈 | 用预测命中、反事实验证和反馈回写校准 |
| 审查层 | 控制现实落地风险 | 法律规则、安全阈值、平台规则、用户确认、审计要求 | 用合规结果、事故信号和人工复核校准 |

理论探索层的临时权重公式可以表达为：

```text
expert_weight =
  scene_relevance
  + evidence_fit
  + predictive_discrimination
  + counterfactual_value
  + feedback_testability
  - redundancy_penalty
```

审查层的权重不参与理论预测排序，而是决定输出等级：

| 输出等级 | 含义 |
| --- | --- |
| `research_only` | 理论上有分析价值，但不能进入行动建议或外部发送 |
| `needs_human_review` | 可作为候选建议，但必须补充证据或由用户确认 |
| `actionable_draft` | 可生成低风险草稿、预览和人工确认清单 |
| `blocked_execution` | 不允许进入真实执行或平台调用 |

### 8. 独立合理合法安全审查模块

审查模块在理论探索完成后运行，输入是完整理论结果，不是被提前删减后的结果。

| 审查维度 | 检查问题 | 可能结论 |
| --- | --- | --- |
| 合法性 | 是否涉及违法、侵权、威胁、骚扰、非法获取数据、金融或医疗等受限场景 | 允许、需法律确认、阻断 |
| 合规性 | 是否违反平台规则、组织制度、行业规范或数据处理要求 | 允许、需改写、阻断 |
| 安全性 | 是否可能导致人身、财务、心理、隐私或不可逆伤害 | 允许、降级、阻断 |
| 合理性 | 建议是否与真实关系、证据、目标和可承受成本匹配 | 允许、需补证、建议保留研究 |
| 伦理和自主性 | 是否存在胁迫、欺骗、越界、替用户决策或削弱对方退出权 | 允许、需用户确认、阻断 |
| 隐私 | 是否暴露第三方、敏感身份、私人记录或超范围使用数据 | 脱敏、需授权、阻断 |
| 可执行性 | 是否有平台、权限、时间、身份和状态条件支持 | 可预览、只能草稿、不能执行 |
| 审计回写 | 是否能记录来源、确认、发送状态、反馈和后续修正 | 可闭环、需补日志、不能闭环 |

审查层允许提出替代表达和降级路径，但不得把理论探索层的研究记录改写成“从未出现过”。系统需要同时保留：

- 理论层的原始假设和预测值。
- 审查层的可用性等级和原因。
- 用户确认和人工复核记录。
- 发送、未发送、阻断、反馈和回写证据。

### 9. 未来输出形态

理论探索层每次应输出：

- 场景和目标假设。
- 专家调用清单和权重理由。
- 多假设解释图谱。
- 博弈或策略树。
- 心理和关系机制说明。
- 逻辑前提、矛盾和证据缺口。
- 可观测信号和反事实断点。
- 理论预测值排序。
- 反馈验证计划和回写字段。

审查层每次应输出：

- 合法性、合规性、安全性、合理性、伦理和隐私结论。
- `research_only`、`needs_human_review`、`actionable_draft` 或 `blocked_execution` 等级。
- 如果可进入建议，给出可人工确认的 `message_draft`。
- 如果不能进入建议，说明阻断原因和可保留的研究价值。
- 需要用户确认、授权、补证或人工复核的清单。
- 外部平台 dry-run、预览、发送阻断和审计回写要求。

材料四的结论是：专家矩阵应当先释放理论探索能力，把关系、心理、博弈、逻辑、文化和组织机制尽可能建模到位；再用独立审查模块治理真实行动。它的核心价值不是在设计阶段压缩可能性，而是在理论层发现更多可能，在工程层清楚地区分研究、建议、确认和执行。

## 当前真实主路径、专家并行与前端展示契约

| 3 | 人物识别 | 识别目标窗口、目标对象、候选人物和身份标签 | identity resolution / 候选确认 | 人物未确认时进入 `context_capture_hint`，不输出可发送亲密草稿。 |
| 4 | 事件和图谱读取 | 读取当前目标对象相关的人际关系、事件、目标和历史记录 | `ContextSnapshot`、`target_context_windows` | 至少区分 today、last_7_days、last_30_days、historical 四个时间窗。 |
| 5 | 上下文组装 | 把用户目标、系统目标、人物关系、事件摘要、约束、风险和初步草稿摘要组装成人类可读上下文 | `expert_context_pack.v1` | 这是给每个专家的“同一份案卷”，要求通顺、可审计、可并行读取。 |
| 6 | 初步策略候选 | 生成候选行动、推荐项和 `message_draft` 草稿 | `ranked_options`、`recommended_option.message_draft` | 这是供专家评估的候选，不代表最终可发送。 |
| 7 | 基础 Agent 意见 | 目标、关系、事件、规范、选项、技能、ROI、证据、反馈 Agent 给出基础意见 | `agent_opinions` | 基础 Agent 负责宽口径检查，不替代专家矩阵。 |
| 8 | 专家并行 fan-out | 每个专家拿到自己的 `ExpertContextPack` 后独立分析 | `expert_independent_context_analysis.v1` | 当前代码是确定性同步 runner 加并行契约，数据依赖已拆开，可迁移到 Promise.all、worker、子代理或 LLM 并发。 |
| 9 | 并行运行日志 | 记录每个专家 lane 的输入引用、输出引用、完成状态和可并行性 | `parallel_expert_run_log.v1` | 用来证明专家不是串行互相污染，也不是只凭专家名称生成建议。 |
| 10 | 权重整合 | 汇总专家结论、风险信号、硬阻断和小幅权重修正 | `expert_weight_integration.v1`、`expert_adjusted_decision_weights.v1` | 这里才形成最终权重依据；权重仍是规则型，可解释，不宣称已由真实反馈学习校准。 |
| 11 | 恋人关系目标分析 | 若主身份/场景进入恋人关系分支，分析 R0-R6/RX 阶段、心理舒适度、句子意图和目标差距 | `romantic_goal_analysis.v1`、`dialogue_intent_contract.v1` | 商务模板不能覆盖主恋人身份；其他身份只能作为次级场景修饰。 |
| 12 | 恋人关系统筹专家 | 将专家矩阵、阶段目标、逐句评审、草稿和风险合成最终前端行动 | `romantic_relationship_coordinator_expert.v1` | 恋人关系场景由恋人关系统筹专家做最终汇总，不由单个风险专家直接决定输出。 |
| 13 | 前端展示契约 | 决定控制台详显、悬浮窗简显、发送窗口候选三类显示 | `frontend_display_contract.v1` | 悬浮窗只显示短状态；详细日志进入控制台的类聊天模型窗口。 |
| 14 | 发送门阀 | 判断是否进入发送窗口或只给用户提示 | `send_gate_transfer_path.v1` | 当前默认 `blocked_prompt_only`，自动发送阻断时只提示用户，不写入目标聊天框。 |
| 15 | 反馈和回写计划 | 若用户执行或反馈，写回事件、关系和后续校准证据 | `feedback_plan`、事件候选、关系变更候选 | 这是闭环入口；真实反馈校准仍是生产级未完成项。 |

因此，专家举证不是固定“第十个顺序才突然发生”的单点动作。如果按上表的工程顺序看，专家并行 fan-out 在第 8 步，权重整合在第 10 步；如果按旧版理论流程粗分，它属于“专家评估”节点。无论采用哪种编号，关键约束是：专家必须在最终前端提示和发送门阀之前完成，并且必须读取 `ExpertContextPack` 而不是只读取最新一句话。

### 10.2 每个专家获得的上下文

`ExpertContextPack` 当前至少包含：

| 上下文块 | 内容 | 目的 |
| --- | --- | --- |
| `system_goal` | 系统总目标、当前场景、输出模式 | 让专家知道本轮是在“辅助用户达成目标”，不是自由聊天。 |
| `user_goal` | 用户输入目标、目标对象、当前消息摘要 | 保证建议服务用户目标。 |
| `target_context_briefs` | 每个目标对象 today、last_7_days、last_30_days、historical 四窗口摘要 | 避免只根据最新一句话判断关系阶段。 |
| `relationship_summary` | 关系类型、阶段、健康度和主次身份 | 确保恋人、候选暧昧、商务、朋友等模板由身份标签决定。 |
| `event_summary` | 最近事件、候选事件、证据强度 | 让专家知道哪些判断有事实来源。 |
| `recommended_option` | 初步推荐动作、草稿预览和可观察反馈 | 专家评估的是具体候选行动，不是抽象口号。 |
| `constraints` | 自动发送阻断、用户确认、外部平台 dry-run、真实发送条件 | 避免专家建议越过工程门阀。 |
| `risk_controls` | 目标侧操控风险、用户侧安全审核分离、隐私和边界提醒 | 保证风险审查可见，但不在理论阶段提前删减全部可能性。 |

每个专家在该上下文上按自己的学科身份独立判断。例如心理学专家重点看情绪回应、依恋、舒适度和阶段性热度；博弈论专家重点看互动投入、承诺信号、反馈回合和策略可逆性；逻辑证据专家重点看前提、缺口、反例和阶段跳跃；目标侧反 PUA 专家只检查目标对象是否对用户表现出孤立、羞辱、愧疚施压、威胁、隐私勒索等操控行为；用户侧操控审查不耦合在恋人关系专家矩阵里，而由发送前安全审核或等价模块处理。

### 10.3 并行线程路线图和边界

当前代码已经把数据依赖拆成可并行 lane，但执行器仍是确定性同步 runner 加并行契约。下一步如果要做到多对象、多窗口、低延迟运行，应按下面线程边界推进，避免多个开发线程互相踩逻辑。

| 线程 | 负责内容 | 输入 | 输出 | 不得处理 |
| --- | --- | --- | --- | --- |
| A 来源接入线程 | 微信、网页、其他软件的 observation、OCR、窗口元数据 | 平台只读信息 | `IntakeObservation`、`RawEvent` | 不做专家判断，不写最终关系事实。 |
| B 图谱上下文线程 | 人物映射、事件读取、四窗口上下文、关系标签 | `RawEvent`、identity link、图谱记录 | `ContextSnapshot`、`target_context_windows` | 不生成对外草稿。 |
| C 专家运行线程 | 为每个专家生成上下文包并执行独立分析 | `ContextSnapshot`、候选推荐、专家配置 | `ExpertContextPack`、`expert_independent_context_analysis`、`parallel_expert_run_log` | 不直接写 GUI，不直接发送。 |
| D 关系分支线程 | 恋人、商务、朋友、家庭等关系状态机和统筹专家 | 专家矩阵结果、主身份标签、场景目标 | 分支目标分析、统筹专家结论 | 不重新做来源接入。 |
| E 前端投影线程 | 把运行态转换为控制台和悬浮窗可读状态 | 统筹专家、发送门阀、专家日志摘要 | `pt028_gui_decision_state.v1`、GUI 状态 | 不暴露目标不可见日志到聊天框。 |
| F 发送门阀线程 | 校验窗口、用户确认、草稿 hash、平台预览、反馈计划 | 前端候选草稿、runner 材料、用户确认 | `send_gate_transfer_path`、受控发送命令或阻断提示 | 不修改专家结论。 |
| G 反馈回写线程 | 记录用户选择、执行结果、目标回应、关系变化 | 用户反馈、发送结果、目标回应 | 事件候选、关系候选、权重校准证据 | 不补写未经确认的人物事实。 |

多会话并发时，每个目标对象必须拥有独立的 `conversation_session_id`、`target_person_id`、`context_snapshot_id`、`expert_run_id` 和 `frontend_display_state_id`。共享层只能读取专家配置、schema、关系阶段目录和安全阈值；不得把 A 对象的 `ExpertContextPack`、草稿或风险日志复用到 B 对象。这样同时处理多个聊天窗口时，只有来源接入和 GUI 刷新是轻量共享，核心分析链路按目标对象隔离。

### 10.4 恋人关系分支与可复用部分

恋人关系分支当前专有内容：

- R0-R6/RX 阶段目录、阶段迁移规则、心理舒适梯度、亲密关系目标状态和同意准备度。
- 恋人关系统筹专家 `romantic_relationship_coordinator_expert.v1`。
- 目标侧反 PUA / 操控风险专家的范围定义。
- 自动发送阻断下的第三方提示卡和目标不可见日志。

可复用到商务、朋友、家庭、售后、冲突修复等关系图谱的内容：

- `ExpertContextPack` 的上下文组装方式。
- 四窗口目标对象上下文读取。
- 并行专家 lane 和 `parallel_expert_run_log`。
- 专家权重整合、理论探索层和独立审查层分离。
- 前端展示契约的“悬浮窗简显、控制台详显、发送窗口只收草稿候选”三分法。
- 发送门阀的用户确认、窗口校验、草稿 hash、平台预览、反馈回写计划。

后续新增系统目标时，应增加目标分支和统筹专家，而不是改掉通用主路径。例如商务关系可以新增 `business_relationship_coordinator_expert`，朋友关系可以新增 `friendship_coordinator_expert`，但都继续复用 `ExpertContextPack -> parallel_expert_run_log -> coordinator -> frontend_display_contract -> send_gate_transfer_path`。

### 10.5 GUI、悬浮窗和发送窗口展示边界

当前展示分三层：

| 界面 | 显示内容 | 不显示内容 | 目的 |
| --- | --- | --- | --- |
| 悬浮窗 | 一条短状态，如 `R2/O3/F0 · micro_progression · prompt-only`；可做水平滚动，悬停暂停，点击进入控制台 | 不显示专家完整日志、不显示隐私原文、不显示可被目标看见的风险判断 | 在微信旁边低打扰提醒关系阶段、线上/线下轨道、本轮意图和发送状态。 |
| GUI 控制台 | 类聊天模型窗口的详细日志：统筹摘要、专家证据、并行 lane 状态、发送门阀、第三方提示、草稿和分支记录 | 不自动注入目标聊天框 | 供用户审查“为什么这么建议、哪些证据不足、哪些专家有不同意见”。 |
| 发送窗口 | 只有用户确认后的草稿候选和受控发送检查清单 | 不显示专家内部日志和风险推理 | 防止把分析内容误发给目标对象。 |

悬浮窗位置接口应只依赖前端窗口层：默认吸附到微信或当前目标聊天软件窗口边缘，找不到目标窗口时落到屏幕右侧安全区域；它只读取 `frontend_display_contract.surfaces.dock` 和 `send_gate_transfer_path.current_mode`。控制台读取 `frontend_display_contract.surfaces.console`、`parallel_expert_run_log` 和 `expert_context_packs` 的摘要。发送窗口只读取 `send_gate_transfer_path.transfer_payload` 或阻断提示，不读取完整专家上下文。UI 显示原则参考 W3C WCAG 对移动、滚动或自动更新内容的要求：自动移动或自动更新信息应提供暂停、停止、隐藏或更新频率控制机制；对结构化信息也应保留关系和层级，避免把状态、证据和操作混在同一块文本里。参考：[WCAG 2.2.2 Pause, Stop, Hide](https://www.w3.org/WAI/WCAG22/Understanding/pause-stop-hide.html)、[WCAG 1.3.1 Info and Relationships](https://www.w3.org/WAI/WCAG22/Understanding/info-and-relationships.html)。

当前 GUI 已接入 `zhineng:decision-state:get` 的只读状态，并在控制台展示统筹日志、专家运行日志和发送门阀；悬浮窗已接收 runtime 短状态。仍未完成的是生产级低延迟事件流、真实多窗口压力测试、最终 UI 交互专项验收和真实反馈校准。

### 10.6 独立 UI/悬浮窗设计接口说明

本节给独立 UI 线程使用。UI 线程只需要遵守下面契约，不需要理解专家矩阵内部实现。

| Surface | 设计目标 | 运行时字段 | UI 可控项 | 边界 |
| --- | --- | --- | --- | --- |
| `dock` 悬浮窗 | 在微信或其他目标软件旁边低打扰显示当前运行状态 | `frontend_display_contract.surfaces.dock` | 位置、尺寸、吸附边、是否展开、是否暂停滚动、是否隐藏、透明度、点击打开控制台 | 只显示短状态；不得显示专家完整日志、原始隐私文本、目标不可见风险判断。 |
| `console` 控制台 | 以主流对话模型窗口风格展示详细可审查日志 | `frontend_display_contract.surfaces.console`、`parallel_expert_run_log`、`expert_context_packs` | 日志分组、折叠/展开、专家 lane 状态、证据引用、草稿预览、分支记录 | 展示用户可见的结构化推理摘要和证据日志，不展示模型隐含思维链。 |
| `send_window` 发送窗口 | 只承载确认后的草稿候选和发送门阀清单 | `send_gate_transfer_path.transfer_payload`、`send_gate_transfer_path.required_gates` | 草稿预览、目标窗口校验、用户确认按钮、发送阻断状态 | 不读取专家上下文，不显示风险推理，不自动发送。 |

悬浮窗放置接口：

| 字段 | 含义 | 当前默认 |
| --- | --- | --- |
| `placement_policy.preferred_anchor` | 首选吸附对象 | `target_application_window_edge` |
| `placement_policy.target_application` | 当前目标软件 | `wechat` 或草稿 channel |
| `placement_policy.fallback_anchor` | 找不到目标窗口时的位置 | `screen_right_safe_zone` |
| `placement_policy.avoid_target_chat_input_overlap` | 避免遮挡聊天输入框 | `true` |

| `placement_policy.persist_user_override` | 用户拖动后记住位置 | `true` |

上下文接口：

| Surface | 读取模型 | 允许字段 | 刷新策略 |
| --- | --- | --- | --- |
| 悬浮窗 | `read_only_runtime_decision_state` | `frontend_display_contract.surfaces.dock`、`send_gate_transfer_path.current_mode`、统筹专家摘要 | 默认 5 秒轮询；后续可替换为事件流。 |
| 控制台 | `read_only_runtime_decision_state` | 专家上下文包摘要、并行运行日志、统筹专家、前端展示契约、发送门阀、链路和分支 | 默认 5 秒轮询；详情区可手动刷新。 |
| 发送窗口 | `send_gate_transfer_payload_only` | `transfer_payload` 和 `required_gates` | 只在用户打开发送预览或门阀状态变化时刷新。 |

发送门阀开启后的传递路径：

| 状态 | 条件 | 传递内容 | 仍需门禁 |
| --- | --- | --- | --- |
| `blocked_prompt_only` | 默认；未显式授权受控预览，或身份/上下文/审查未通过 | 用户可见提示、草稿预览、悬浮窗短状态、日志入口 | 不进入目标窗口。 |
| `controlled_send_preview_candidate` | 显式 `controlledSendPreviewAuthorized=true`，且独立审查为 `actionable_draft`、身份已确认、无硬阻断 | 仅 `transfer_payload`：channel、目标 person id、目标显示名、草稿、对话动作、统筹专家要求 | 目标窗口校验、草稿 hash、操作者确认、平台预览阻断审计、反馈回写计划。 |

多人同时对话时，UI 层不得共用某个对象的状态。悬浮窗和控制台需要按 `conversation_session_id`、`target_person_id`、`context_snapshot_id`、`expert_run_id` 和 `frontend_display_state_id` 分桶展示；如果两个聊天窗口同时更新，悬浮窗只显示当前焦点窗口或用户固定窗口的短状态，控制台可用 tabs 或列表切换对象。不同对象之间只共享 schema、专家配置和视觉组件，不共享上下文包、草稿和风险日志。

### 10.7 恋爱专家梯度管理 V2：线上 / 线下双轨推进方案

本节回应“需要恋爱专家专门做梯度设计，并参考迷男方法等阶段管理经验，同时区分线上聊天和线下见面”的检查点。检查时确认：原代码已有 R0-R6/RX 阶段、四时间窗证据、心理舒适度模型和阶段迁移判断，但缺少线上聊天、线下见面、邀约转场、见面后反馈的独立进度轨。当前初版已在不推翻 `romantic_stage_gradient.v1` 的前提下，落地 `online_offline_progression_track.v1`、`date_transition_readiness.v1` 和 `romantic_progression_cadence.v1`。

#### 10.7.2 文献和方法来源到工程映射

| 来源 | 可吸收原则 | 工程映射 |
| --- | --- | --- |
| Sternberg 三角理论：亲密、激情、承诺三成分 | 恋爱目标不能只看热度，也要看亲密感和承诺信号 | `romantic_goal_analysis` 需要分别记录 `intimacy_signal`、`passion_signal`、`commitment_signal`。 |
| 社会渗透理论 / 自我披露深度与广度 | 关系推进依赖披露主题从浅到深、从窄到宽，并看互惠 | `target_context_windows` 按 today、7 天、30 天、历史评估披露广度、深度和互惠。 |
| Knapp 关系发展模型 | 初识、试探、升温、整合、绑定是可观察阶段 | R0-R6/RX 保持总阶段，新增线上 O 轨和线下 F 轨做细分。 |
| 亲密过程模型 / 伴侣回应性 | 情绪披露只有在被理解、确认、关心时才稳定转化为亲密 | 心理学专家重点评估“目标是否回应用户情绪”和“用户回复是否让对方感觉被看见”。 |
| 依恋理论 | 亲密推进会触发安全感、回避、焦虑等差异 | `attachment_psychology_expert` 给降档、修复和稳定建议，不把沉默简单等同拒绝。 |
| 投资模型和关系维护策略 | 关系稳定来自满意度、投入、替代选项和维护行为 | 博弈 / 关系维护专家评估互动投入、复访、共同计划、修复动作。 |
| Gottman bids / repair attempts | 小的连接请求和冲突修复会积累关系质量 | 每轮草稿要说明是“回应连接请求、发出连接请求、修复、邀约、边界确认”哪一种。 |
| CMC / hyperpersonal / 线上到线下转场研究 | 线上容易理想化、可编辑、缺少非语言反馈；第一次线下见面会重新校准吸引 | 线上推进以低压、可撤回、明确意图为主；线下轨单独记录见面前安全、见面中反馈和见面后复盘。 |
| 同意与边界资料 | 同意必须清晰、持续、无压力、可撤回；沉默或过去同意不能代表当前同意 | `consentReadinessForStage` 继续作为硬审查输入，身体亲密阶段只允许边界沟通和用户侧提醒，不输出施压策略。 |
| PUA / 迷男类方法 | 只能抽取“阶段意识、反馈校准、从线上到线下转场、节奏管理”的结构 | 禁止引入 negging、羞辱、制造嫉妒、虚假稀缺、隔离、内疚施压、隐私勒索、绕过同意等做法；这些进入风险审查词库。 |

#### 10.7.3 迷男方法的可用边界和禁用边界

本系统不实现操控型 PUA。所谓“可参考迷男方法”，只允许作为社交阶段管理的结构化启发：识别阶段、读取反馈、控制节奏、避免一下子跳到过深话题、把线上互动转为现实见面机会、见面后复盘。所有会削弱对方自主性、制造心理压力或绕过边界的内容，都进入风险审查或发送门阀阻断。

| 类别 | 可进入正向方法库 | 必须排除或进入风险审查 |
| --- | --- | --- |
| 开场 | 轻量、自然、可回应的开场；基于上下文，不强迫对方连续回应 | 侮辱式开场、贬低、冒犯测试、制造自卑。 |
| 升温 | 幽默、共情、共同兴趣、逐步自我披露、可逆小试探 | 羞辱、比较、故意冷落、制造嫉妒、虚假竞争者。 |
| 转场 | 明确、低压、可拒绝的邀约；给对方选择空间 | 封闭场景压迫、隔离、临时改变地点、让对方难以退出。 |
| 亲密边界 | 用语言确认舒适度、尊重迟疑、接受拒绝、见面后反馈 | 把“证明喜欢”绑定到亲密行为；利用礼物、关系身份、愧疚或威胁施压。 |
| 复盘 | 记录反馈、热度、对方边界、下次节奏 | 把对方的拒绝当作“可突破阻力”的技巧问题。 |

#### 10.7.4 线上 O 轨：聊天推进梯度

| 阶段 | 入口证据 | 本阶段目标 | 允许动作 | 升级条件 | 降档 / 停止条件 |
| --- | --- | --- | --- | --- | --- |
| O0 身份和窗口确认 | 目标对象、窗口、关系候选标签不明或上下文不足 | 只确认人物和当前消息，不推进关系 | `context_capture_hint`、补证据提示 | 人物和当前对话可读，且无风险例外 | 身份冲突、窗口不可读、公众号/服务号干扰。 |
| O1 低压开场 | 有当前消息，但关系热度低或历史不足 | 建立自然回应，不制造负担 | 简短回应、轻问句、承接对方信息 | 对方愿意继续、语气不排斥 | 冷淡、只读不回、明显忙碌。 |
| O2 互惠节奏 | 双方有来回、话题稳定 | 看回应速度、主动性、幽默和回问 | 轻调侃、共同兴趣、节奏匹配 | 对方主动展开、回问、情绪更松 | 只有单字回复、明显敷衍。 |
| O3 情绪披露和回应 | 出现生活、情绪、价值观或脆弱信息 | 让对方感觉被理解、确认、关心 | 共情回应、适度自我披露、记忆点回扣 | 对方继续披露或表达被理解 | 对方回避、转移、表达不适。 |
| O4 关系意图试探 | 暧昧、称呼、占有感、未来计划或亲密调侃增多 | 用可撤回方式测试对方对关系推进的接受度 | 低压表达好感、关系定义轻试探 | 对方正向回应或主动升级 | 对方明确否认、模糊但不愿继续、风险信号。 |
| O5 线下邀约准备 | O2-O4 稳定，且存在共同场景或见面理由 | 从线上转向现实见面，不把见面做成压力 | 具体但可拒绝的邀约、给 2 个低压选项 | 对方接受或提出替代时间 | 对方多次回避、地点不安全、意图不清。 |
| O6 见面前校准 | 已约定或接近约定见面 | 确认时间、地点、舒适度和期待 | 轻确认、给退出空间、说明安排 | 双方对时间地点清晰 | 临时改到不安全地点、对方焦虑或犹豫。 |
| O7 见面后复盘 | 已线下见面或有线下互动反馈 | 写回体验、热度、边界和下次建议 | 感谢、回扣细节、轻邀约下一次 | 对方积极复盘或继续互动 | 对方冷却、冲突、表达不舒服。 |

#### 10.7.5 线下 F 轨：见面推进梯度

| 阶段 | 入口证据 | 本阶段目标 | 允许动作 | 升级条件 | 降档 / 停止条件 |
| --- | --- | --- | --- | --- | --- |
| F0 公开安全见面 | 第一次或低熟悉度见面 | 让双方都安全、自然、可退出 | 公共地点、短时长、明确交通和结束点 | 对方放松、愿意延长或再次见 | 对方紧张、想离开、临时不适。 |
| F1 现场舒适建立 | 双方已见面，互动刚开始 | 降低尴尬，建立当下舒适感 | 轻松话题、观察节奏、避免查问式审问 | 对方主动展开、眼神/语气/身体朝向更开放 | 对方封闭、防御、频繁看手机或想结束。 |
| F2 连接请求和回应 | 互动稳定，有笑点、分享或共同体验 | 发出和回应小连接请求 | 共同体验、赞赏具体细节、自然照顾 | 对方接受连接并反向发起 | 对方不接、转移、表达不适。 |
| F3 非性亲密准备 | 明显互有好感，且关系已确认或接近确认 | 评估非性亲密舒适度 | 只做低强度、可拒绝、可停止的靠近建议；优先语言确认 | 对方明确舒适、主动靠近或语言许可 | 迟疑、僵住、沉默、后退、转移。 |
| F4 更强亲密边界复核 | 出现亲吻或更强亲密可能性 | 从“热度”转为“清晰边界和持续同意” | 边界确认、节奏确认、是否继续的开放问题 | 清晰、主动、持续的正向同意 | 含糊、沉默、酒精影响、压力情境。 |
| F5 亲密边界 / 健康 / 隐私沟通 | 双方开始讨论更深亲密 | 把安全、健康、隐私、排他性和期待说清楚 | 只允许沟通建议和用户侧提醒 | 双方都能清楚表达和接受边界 | 任何施压、羞辱、威胁、隐私索取。 |
| F6 目标状态事实确认 | 用户确认双方关系已进入生理亲密目标状态且可审计 | 写入关系事实和后续维护目标 | 不对外炫耀、不外发隐私；给关系健康、尊重和反馈建议 | 反馈稳定、双方愿继续关系 | 后悔、冲突、隐私风险、边界争议。 |
| F7 维护和修复 | 关系已稳定或进入波动 | 维持亲密、修复冲突、规划下一阶段 | 感谢、复盘、开放沟通、共同计划 | 冲突被修复，互动质量回升 | 反复拒绝修复、操控、威胁或伤害。 |

#### 10.7.6 梯度节奏机制

每一轮建议必须先给出“本轮意图”，再给具体草稿。意图枚举建议为：

- `observe_or_hold`：证据不足、对方冷淡、风险不明，只观察或补上下文。
- `warm_response`：回应对方情绪或信息，让对话继续自然流动。
- `small_reversible_probe`：小范围试探，如轻微好感、共同兴趣、低压约定。
- `micro_progression`：已有正向反馈时推进一小格，仍保持可撤回。
- `soft_invitation`：邀请线下或下一次互动，必须给替代选项和拒绝空间。
- `boundary_review`：涉及身体亲密、隐私、排他性、健康、安全时，优先边界沟通。
- `repair_or_downshift`：出现误读、冷却、冲突或压力时，先修复或降档。
- `maintenance`：关系已确认后，进入稳定、尊重、反馈、长期体验维护。

节奏不是按固定天数推进，而按“反馈密度 + 互惠质量 + 阶段证据 + 风险水平”推进。建议默认：

| 条件 | 节奏判断 | 输出 |
| --- | --- | --- |
| 四窗口都缺或只有当前一句 | 不允许阶段升级 | 生成 `context_capture_hint` 或低压回应。 |
| today 热、7 天冷 | 不升级关系事实 | 当前轮可微推进，但日志写入“短期热度，稳定性不足”。 |
| today 冷、历史热 | 降档或修复 | 先解释、关心、给空间，不硬推。 |
| 7 天和 30 天都有互惠、对方主动、无风险 | 可小步升级 | 输出明确本轮意图和可回写反馈点。 |
| 出现迟疑、沉默、回避、拒绝、未成年/酒精/隐私/威胁等风险 | 进入 RX 或边界复核 | 不给推进草稿，只给用户侧安全提示和补证据建议。 |

#### 10.7.8 前端和日志展示要求

V2 以后，GUI 控制台需要能显示：

- 当前总阶段：`R*`。
- 当前线上轨：`O*`，当前线下轨：`F*`。
- 本轮意图：如 `warm_response`、`small_reversible_probe`、`soft_invitation`、`repair_or_downshift`。
- 推进依据：引用 today、last_7_days、last_30_days、historical 中的证据摘要，不显示目标不可见风险推理。
- 降档依据：如果不推进，明确是“上下文不足、热度不稳、对方迟疑、风险、窗口不可读、身份未确认”哪一种。
- 悬浮窗只显示短状态，例如 `R2/O3/F0 · micro_progression · prompt-only`。

发送窗口仍然只接收可确认草稿和发送门阀清单。任何完整专家日志、风险推理、隐私原文、目标不可见分析都不得进入目标聊天框。

验收口径：

- `low_latency_event_stream_runtime=passed` 只证明低延迟事件流和桌面端订阅/回退路径存在。
- `feedback_bound_multi_window_event_stream=open` 表示最终专项验收仍缺与真实多窗口 operator feedback 绑定的低延迟事件流；这不同于低延迟工程契约本身。
- `multi_window_feedback_calibration_dry_run=partial` 证明多窗口隔离和校准逻辑存在，但不证明真实反馈学习完成。
- `real_feedback_calibration_evidence=passed` 只有在真实反馈批次覆盖全部目标窗口、至少两个唯一目标，并通过上述证据条件时成立。
- `final_special_acceptance_gate=passed` 证明验收门禁能正确阻断缺失真实证据的生产验收。
- `production_feedback_and_gui=open` 仍是正确状态，直到真实多窗口 operator feedback 全部写入且人工专项确认通过。

## 三条闭环样例

| 闭环 | 目标 | 关键链路 |
| --- | --- | --- |
| `loop_client_a_tech_review` | 推动张总进入技术评审 | 预算异议 -> 低承诺评审邀请 -> 对方确认评审 -> 回写技术评审事件 |

## 存储对象映射

每条闭环都必须包含：

- `Person`：目标对象和必要参与人。
- `RelationshipEdge`：用户与目标对象的关系边。
- `RawEvent`：原始互动、执行记录或反馈来源。
- `SemanticEvent`：从原始事件提取的可分析事件。
- `decision_id`：决策集群输出的行动建议标识。
- `trigger_id`：触发引擎生成的执行链路标识。
- `FeedbackRecord`：执行后反馈、目标推进、关系变化和新事件候选。
- `StorageAudit`：由存储运行时为每次写入自动记录。
- `EventIndex`：由源数据重建人物、关系、标签和时间索引。

## 流程连贯性回顾

当前流程是连贯的：

```text
用户目标
-> 人物和关系建模
-> 原始事件记录
-> 语义事件提取
-> 决策集群给出建议
-> 触发引擎生成执行预案
-> 自动化预览和本地试运行
-> 用户执行或确认
-> 反馈记录
-> 回写事件候选
-> 重建索引
-> 审计记录
-> 后续检索和下一轮决策
```

其中 `runtime/state/` 负责运行状态，`data/` 负责业务存储，二者通过 `run_id`、`decision_id`、`trigger_id`、`feedback_id` 和 `event_id` 关联。

## 与初始目标的一致性

初始目标是把系统构建为人类社交辅助系统，而不是单纯话术工具。当前样例满足这个方向：

- 它记录对象：通过 `Person` 和 `RelationshipEdge` 记录目标对象和关系状态。
- 它记录事件：通过 `RawEvent` 和 `SemanticEvent` 记录互动、线索、证据和影响。
- 它支持计划安排：通过 `decision_id` 和 `trigger_id` 把建议转为执行链路。
- 它支持反馈闭环：通过 `FeedbackRecord` 和回写事件判断行动效果。
- 它支持未来检索：通过人物、关系、标签和时间索引回放线索。
- 它支持外部检查：通过审计日志和状态笔记分工记录。

## 仍需注意

- 当前三条样例已达到每个客户 10 条近真实互动记录，但仍不等于用户真实聊天记录验证。
- 正式试点前应由用户提供或确认真实样本，替换近真实样例后重跑 `node scripts/run-mvp-loop.mjs --all`。
- 统一 MVP 脚本已能跑完“状态上报 + 存储写入 + 决策 + 触发 + 反馈 + 回写 + 索引 + 审计”。
- 自动化预览已接入 MVP 闭环并保持 dry-run，不应视为真实平台执行能力已完成。

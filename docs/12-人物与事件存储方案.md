# 人物与事件存储方案骨架

## 当前角色

本文件是第一阶段人物、关系、事件、反馈、索引和审计存储的落地依据。`docs/06-图谱数据结构与接口.md` 中的 `Event` 是图谱接口概念对象，本文件中的 `RawEvent` 和 `SemanticEvent` 是第一阶段实际存储对象。

执行 `docs/10-第一阶段执行方案.md` 时，所有人物和事件相关步骤必须同步遵守本文件；跨文档一致性检查见 `docs/13-文档互证审计与修复清单.md`。

## 一句话结论

人物存储和事件存储第一版采用 **本地文件 + JSON/JSONL + 可重建索引** 作为过渡实现，先验证“能记录、能关联、能检索、能回写、能审计”。正式扩展前必须迁移到 `graph_memory_database.v1` 的紧凑属性图数据库模式，避免人物、事件、专家建议和上下文快照继续扩散成大量临时文件或长表。

## 核心目标

- 人物存储：记录对象是谁、和用户是什么关系、当前关系状态如何。
- 事件存储：记录发生了什么、涉及谁、影响什么关系、证据和反馈是什么。

## 主要步骤

1. 定义人物存储 schema。
2. 定义关系边存储 schema。
3. 定义原始事件和语义事件 schema。
4. 建立本地目录结构。
5. 建立索引文件。
6. 建立事件写入、查询、回写规则。
7. 建立状态和审计记录。

## 每一步为什么必要

- 人物 schema：保证对象唯一，不混淆客户。
- 关系边 schema：让系统知道“这个人和用户是什么关系”。
- 事件 schema：让所有互动都能被记录和追踪。
- 本地目录：降低第一阶段实现成本。
- 索引文件：避免每次全量扫描。
- 写入/查询/回写规则：形成闭环。
- 状态和审计：方便外部检查运行状态和错误。

## 事实、推理、建议、猜测

| 类型 | 内容 | 处理方式 |
| --- | --- | --- |
| 事实 | 已有人际关系图谱和事件图谱 | 继续作为 schema 来源 |
| 事实 | 当前项目已有 Node.js 本地骨架 | 用现有结构扩展 |
| 事实 | 第一阶段是最小试点 | 不引入重型数据库 |
| 推理 | 本地 JSON/JSONL 足够做第一轮验证 | 用测试数据验证容量和检索速度 |
| 推理 | 事件需要 append-only 记录 | 检查是否方便追溯和重建 |
| 建议 | 人物、关系、事件、索引分开存储 | 看是否符合后续扩展 |
| 建议 | 事件用 JSONL，人物用 JSON | 便于追加和读取 |
| 猜测 | 第一阶段数据量较小 | 不能作为长期架构依据 |

### 2.1 图谱数据库目标模式 v1

当前 `data/**` 文件是第一阶段过渡层，只允许在样本量小、验收未完成时使用。正式扩展到更多人物、更多事件、网页端、其他聊天软件或业务 API 后，必须按 `schemas/graph-memory-database.schema.json` 迁移到 `graph_memory_database.v1`。

数据库不是按业务对象无限拆表，而是采用紧凑属性图：

| 逻辑集合 | 用途 | 主要键 |
| --- | --- | --- |
| `graph_nodes` | 保存 `person`、`source_identity`、`organization`、`conversation_thread`、`raw_event`、`semantic_event`、`goal`、`context_snapshot`、`expert_review`、`feedback`、`evidence` 等图节点 | `node_id`、`node_type` |
| `graph_edges` | 保存身份映射、参与关系、提及关系、事件派生、目标支持、状态更新、专家评审和证据引用等图边 | `edge_id`、`edge_type`、`from_node_id`、`to_node_id` |
| `graph_evidence` | 保存截图 hash、来源路径、引用摘要、内容指纹和证据可见性，不把大块原始私密内容复制进策略层 | `evidence_id`、`content_fingerprint` |
| `graph_snapshots` | 保存 `ContextSnapshot`、关系快照、事件时间线、专家评审引用和读取理由，供决策集群只读调用 | `snapshot_id`、`goal_id`、`person_id`、`thread_id` |
| `graph_audit_log` | 保存写入、合并、去重、确认、回写、迁移和发送阻断审计 | `audit_id`、`occurred_at` |

必备索引保持可重建：`node_type_id`、`edge_type_from_to`、`person_thread_time`、`event_time_type`、`source_identity_person`、`context_snapshot_goal_person`、`expert_review_context`、`content_fingerprint`。

统一上下游调用格式为 `GraphMemoryEnvelope`：

```json
{
  "schema_version": "graph_memory_envelope.v1",
  "input_refs": {
    "source_adapter_id": "sightflow_desktop.wechat",
    "observation_id": "intake_obs_xxx",
    "raw_event_ids": ["raw_event_xxx"],
    "semantic_event_ids": ["semantic_event_xxx"]
  },
  "upstream_boundary": "source_adapter_collects_only",
  "downstream_boundary": "decision_cluster_reads_context_snapshot_only",
  "relationship_snapshot_ref": "snapshot_rel_xxx",
  "event_timeline_ref": "snapshot_evt_xxx",
  "expert_review_refs": ["expert_review_psychology_xxx"],
  "retrieval_reasons": ["当前目标对象", "同线程最近事件", "专家复核建议"],
  "missing_data_gaps": [],
  "real_send_blocked": true
}
```

上下游边界规则：

- 上游接收软件只提交 `IntakeObservation`、来源能力和证据引用，不确认人物事实，不写专家判断，不真实发送。
- 标准化转译层把多来源输入转成 `RawEvent-ready`，并写入来源身份、内容指纹和发送阻断状态。
- 图谱记忆层负责去重、身份候选、人物/事件节点、图边、专家评审引用和上下文快照。
- 决策集群只读取 `ContextSnapshot` / `GraphMemoryEnvelope`，不能直接扫描原始私密文件，也不能把专家建议写回事实节点。
- 发送层只读取已确认草稿、目标窗口、会话匹配、草稿 hash 和人工确认，不从数据库推断“可以发送”。

缺数据处理规则：

- 当前证据足够完成本轮判断时，必须在当前步骤闭合，输出结论、依据和发送阻断状态。
- 当前证据不足但不影响理论探索时，保留 `missing_data_gaps`、`reserved_weight_inputs` 和后续接口权重，不阻断理论预测输出。
- 身份确认、高风险关系合并、P1/P2 事件事实写回和真实发送缺数据时，必须进入人工确认，不允许由模型补齐。
- 专家评审后的建议作为 `expert_review` 节点或 `expert_review_refs` 纳入后续上下文评审，但不能覆盖原始事实；再次评审时必须标注“专家建议来源”和“采纳/未采纳理由”。

单句简短输入的上下文编辑策略：

- 不把单句直接送入模型孤立判断。
- 先按 `thread_id`、`person_id`、当前目标、最近事件、关系状态、未读增量摘要和专家评审引用扩展成最小 `ContextSnapshot`。
- 如果只能识别到一句短消息但无法确认对象或线程，输出 `context_sufficiency_level=low`，只允许理论候选和澄清问题，不写事实、不触发发送。
- 如果对象、线程和目标已确认，短句作为最新增量，结合历史摘要和最近事件生成回复草稿，并在 `retrieval_reasons` 中说明引用了哪些上下文。

身份映射存储规则：

- `ChannelIdentity` 只记录来源渠道身份线索，不代表已确认人物事实。
- `PersonIdentityLink.status=confirmed` 且 `verified=true` 时，身份映射运行时可以把对应 `person_id` 回填到 `RawEvent.linked_person_ids`。
- 同名、低置信、仅窗口标题或缺少稳定 handle/hash/thread_key 的候选必须写入 `identity-confirmation-queue.jsonl`，等待用户确认后再写 `PersonIdentityLink`。
- 身份索引由 `packages/identity-resolution` 重建，不把业务存储写入 `runtime/state/`；运行证据写到 `runtime/identity-resolution-demos/**` 和 `runtime/identity-resolution-audits/**`。

### 3. 索引与查询阶段

- 按人物查事件。
- 按关系查事件。
- 按标签查事件。
- 按时间查事件。
- 按目标或决策 ID 查回写记录。

### 4. 回写与审计阶段

- 决策后生成事件候选。
- 执行后生成反馈记录。
- 反馈后更新关系摘要。
- 所有写入进入审计日志。
- 高风险事件只进入确认队列。

### 5. 最小闭环验证阶段

- 3 个客户。
- 每个客户 10 到 30 条记录。
- 全部转为事件。
- 能查到人物相关事件。
- 能生成下一步决策。
- 能写入反馈和回写事件。

## 关键风险

- 同一人物重复入库。
- 原始事件和语义事件边界混乱。
- 索引和原始数据不一致。
- 事件标签太粗，后续难检索。
- 本地文件并发写入损坏。
- 事件过多后检索变慢。

## 需要确认的信息

- 第一批人物字段是否够用。
- 是否需要支持一个人物多个身份。
- 事件是否必须保留原文。
- 原文是否只保留摘要。
- 事件标签是否先用固定枚举。
- 是否需要人工确认人物合并。
- 第一轮事件量预估。

## 最低成本验证方式

- 建 3 个客户。
- 写入 30 到 90 条原始事件。
- 自动生成语义事件。
- 建立 3 类索引：人物、标签、时间。
- 查询某客户的全部事件。
- 查询某标签下的全部事件。
- 写入一次反馈。
- 检查关系摘要是否能更新。

### 6. 试点样本导入阶段

- 用户或人工整理外部聊天、网页、手工记录为 `PilotImportBatch`。
- 进入导入或 MVP 前，先运行 `node scripts/validate-pilot-intake.mjs --input=<PilotImportBatch.json>` 生成 `pilot_intake_readiness.v1`。
- 前置门禁输出 `runtime/intake-validations/<import_id>/pilot-intake-readiness.json`，检查目标、人物、关系、样本量、语义覆盖、证据、反馈和单客户 1 小时时间盒。
- `required_failures` 不为空时停止导入或只做材料修正；要进入完整闭环时还需要至少 1 条 `feedback_records`，且 `runMvpLoopFromPilotImport` 会强制校验 `ready_for_closed_loop_mvp`。
- 系统生成 `RawEvent`，并按语义提示或关键词规则生成 `SemanticEvent`。
- 导入结果输出 `raw_event_count`、`semantic_event_count`、`semantic_coverage`、`feedback_count` 和 `ready_for_mvp_sample`。
- `semantic_coverage` 低于 70% 时不能作为扩大样本依据。
- 导入后必须重建人物、关系、标签和时间索引，并写入审计。
- 达到样本门槛后，可通过 `node scripts/run-mvp-loop.mjs --pilot-import=<file>` 直接驱动完整 MVP 闭环。

### 冷启动人物引入与候选图谱过渡

冷启动阶段允许人物先进入候选图谱，不直接写成稳定人物事实。系统可以从 `IntakeObservation.source_identity_hints`、`participants_hint`、网页账号、外部聊天导出和用户手工输入中生成 `CandidatePerson`，并把候选证据写入 `data/people/candidate-persons.json`。

候选人物支持多个角色位置：`PersonRoleBinding` 记录 `graph_id`、`scene`、`role`、`tags`、`confidence` 和证据引用；同一个候选或确认人物可以同时是客户、技术评审人、售后联系人、内部协同人等多个角色。关系权重不只按一条静态边判断，而是由 `SceneRelationshipWeight` 按远近层级、场景、近期互动、角色重要性、身份置信度和用户手工覆盖值计算；候选状态下权重有上限，避免未确认关系过早进入强事实。

用户确认候选后，`confirmColdStartPersonIntroduction` 必须完成四件事：

1. 写入或更新稳定 `Person` 与 `RelationshipEdge`。
2. 为相关渠道身份写入 confirmed + verified 的 `PersonIdentityLink`。
3. 将候选 `PersonRoleBinding` / `SceneRelationshipWeight` 切换到稳定 `person_id` 与正式 `relationship_id`。
4. 调用 `syncRelationshipGraphReferences` 同步 RawEvent/SemanticEvent 中的 `participants`、`linked_person_ids` 和 `linked_relationship_ids`，并把旧候选人物、旧候选关系和同步映射写入事件 `metadata.relationship_graph_reference_sync_history`。

这意味着即使人物最初只在候选图谱中出现，后续确认、合并或关系修改也不能丢失历史。事件索引必须随后重建，策略层只能读取同步后的稳定人物/关系摘要和保留的历史证据摘要。

### 可能性分支与主图谱隔离

当真实输入中出现“一个目标人物具有多重身份”“一句话牵出多个人物和多个事件”“事件之间存在先后或嵌套依赖”时，不允许直接把这些未确认解释写入 `data/people/**` 或 `data/events/**`。系统先生成独立的 `possibility_branch_analysis.v1`，保存到 `runtime/possibility-branches/<branch_id>/**`，用于发现当前人际关系图谱和事件图谱尚未完全覆盖的可能性。

可能性分支只保存四类内容：

- 身份假设：同一 `person_hint_id` 下可以挂多个 `possible_identity_bindings`，例如采购负责人、预算推进人、朋友介绍关系、私下沟通建议者等。
- 事件假设：同一输入中可以并列保存预算推进、接口延期、合同条款审查、合规材料、技术会议、价格沟通边界等事件。
- 嵌套依赖：事件可以通过 `parent_event_hypothesis_id` 和 `dependency_relation` 表示“先给合规材料 -> 才能约技术会”“合同条款审查影响预算推进”等依赖。
- 权重矩阵：`branch_weight_matrix.identity_event_edges` 只用于读取排序和上下文组装，不代表事实确认。

身份角色归因必须区分“转述者”和“被转述对象”：例如“周总提到李工担心接口延期”时，周总可以作为信息来源或会议协调者进入事件上下文，但不能因此继承李工的 `technical_reviewer` 身份。被转述对象的专业角色必须保留在自己的 `person_hint_id` 下。

可能性分支的权重机制采用人类权衡式的多因素评分：身份角色权重综合提及强度、来源置信度、角色适配、场景相关性、时间近因、证据具体度和当前目标相关性；事件侧重权重综合直接意图、目标相关性、证据密度、参与者中心性、时间近因、嵌套依赖、可行动性和风险关注度。权重只回答“后续模型和专家矩阵应该优先读取什么”，不回答“事实是否成立”。

分支提升规则：

- 默认动作是 `keep_in_possibility_branch`。
- 只有用户确认身份、来源证据足够稳定、或跨来源重复支持后，才允许进入 `CandidatePerson`、`PersonRoleBinding`、`SceneRelationshipWeight` 或 `SemanticEvent` 的既有确认流程。
- P1/P2 高风险事件、候选关系合并和真实发送仍必须进入人工确认；可能性分支输出不得直接触发发送。

```powershell

```

### ContextSnapshot

`ContextSnapshot` 是“当前可读上下文”的固化结果。系统默认不把全量原始聊天直接交给决策层，而是按任务、人物和线程缩小范围后，读取最近几轮、今日摘要、历史梯度摘要、高权重事件和风险信号。

当前工程已通过 `buildContextSnapshot` 和 `desktop_context_bridge.v1` 接入决策链路：桌面 `IntakeObservation` 先转生成版 `PilotImportBatch` / `RawEvent` / `SemanticEvent`，再组装 `context_snapshot.v1`。`context_snapshot.v1` 现在会按目标对象输出 `target_context_windows`，供并行专家矩阵读取人物关系、事件时间线、目标、预设偏好和 today / last_7_days / last_30_days / historical 阶段性上下文后，输出理论预测、独立审查、专家权重整合和 `message_draft`。

### 上下文读取原则

- 先确定任务、目标对象和候选人物，再读取该人物的 `ConversationThread`，避免在全量对话中盲目检索。
- 实时回复默认读取最近窗口、当前对话状态、今日对话状态、历史梯度摘要、高权重事件和风险事件。
- 只有出现摘要冲突、人物匹配冲突、关系阶段判断冲突、重要证据缺失、P1/P2 风险或用户要求复核时，才读取完整原始记录。
- 全量原始记录主要用于重建索引、重新摘要、审计和争议复核，不作为每轮回复建议的默认输入。
- 每次上下文快照必须记录 `read_policy`、读取范围、未读取范围和触发全量读取的原因。

### 本机上下文采集与检索候选方案

第一版优先使用低风险、可审计、可回退的本机方案。以下是候选方案，不代表已经接入：

| 能力 | 候选方案 | 作用 | 采用顺序 |
| --- | --- | --- | --- |
| 手工粘贴和文件导入 | 本地 JSON/JSONL、CSV、Markdown | 最小成本进入 `RawEvent` | 第一优先 |
| 浏览器网页读取 | Playwright | 读取网页 DOM、截图、页面快照和测试账号页面状态 | 第二优先 |
| 微信或聊天工具读取 | Wechaty、受控 PC 客户端导出、平台快照 | 读取测试账号或手工授权会话；稳定性和平台限制需单独验证 | 第三优先 |
| 截图 OCR | Tesseract OCR | 当 DOM 或导出不可用时，从截图提取文字 | 兜底 |
| 本地全文检索 | SQLite FTS5 | 按人物、线程、关键词和时间检索原始事件 | 第一版可选 |
| 向量检索 | Chroma 或同类本地向量库 | 按语义召回历史摘要和相似事件 | 后续优化 |

采集层必须先输出标准 `RawEvent`，再进入 `ContextSnapshot`。禁止让外部采集器直接绕过事件记录、人物匹配和审计。

## 执行约束

## 聊天拆分存储与准确回读 v1

聊天、网页或其他软件接口获取到的信息，在完成来源声明和身份解析后，统一以 `RawEvent` 进入本地业务存储。`packages/storage-runtime` 负责执行 `storeChatEventsWithSeparation`，将同一批聊天信息分离写入人物/关系存储、原始事件存储、语义事件存储和可重建索引。

- 人物与关系存储：确认后的 `Person` 与 `RelationshipEdge` 先写入 `data/people/**`。
- 原始事件存储：不丢失中间描述，把完整聊天文本、摘要、参与者、来源和时间写入 `data/events/raw-events.jsonl`。
- 语义事件存储：按预算报价、会议安排、决策信号、任务承诺、风险顾虑、偏好画像和高风险边界等规则生成 `SemanticEvent`，写入 `data/events/semantic-events.jsonl`。
- 可重建索引：写入后重建人物、关系、标签和时间索引，支持按人物、关系、标签、事件类型、关键词和关键事实回读。
- 完整性审计：`auditChatStorageCompleteness` 检查原始事件覆盖、语义事件覆盖、关键事实覆盖、人物/关系链接和索引一致性。

验收入口为 `npm run storage:chat:test`。该命令写出 `chat_storage_test_report.v1` 到 `runtime/chat-storage-tests/**`，作为聊天信息拆分存储、准确回读和代码审计的功能测试证据。

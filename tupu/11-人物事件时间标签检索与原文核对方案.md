## 0. 本文目标

本文回答四个问题：

1. 标签、人物、事件、时间轴之间的调取关系和层级表达如何设计，才能支撑标准化存储。

4. 如何通过人物、事件、时间、标签组装 `ContextSnapshot`，再交给模型回答或工具调用。

## 2. 核心对象层级

### 2.1 原文和证据层

```text
SourceArchive
-> CaptureEpisode / SourceEpisode
-> SignalFrame
-> RawObservation
```

职责：

- 保存完整原文、原图、原音频、原始 API 返回、附件和 hash。
- 给后续所有事件、标签、摘要提供证据根。
- 记录字符 offset、时间段、空间位置、来源线程和内容指纹。

边界：

- 原文只追加或标记删除，不被摘要覆盖。
- 原文不随身份修正和标签修正改写。
- 用户删除时保留删除审计，不再参与检索。

### 2.2 语义拆分层

```text
RawObservation
-> SemanticUnit
-> SemanticEvent
-> NestedEvent
```

职责：

- 把原文拆成语义单元。
- 从语义单元生成事件。
- 复杂内容按语义子事件拆为 `NestedEvent`。

边界：

- `SemanticUnit` 可以标准化表达，但必须有 `raw_observation_id`。
- `SemanticEvent` 记录结构化事实候选，不写关系状态。
- `NestedEvent` 可以独立打标签、独立赋权、独立检索。

### 2.3 人物身份层

```text
EntityMention
-> SourceIdentity
-> IdentityCandidate
-> Person
```

职责：

- `EntityMention` 保存原文里的称呼、昵称、账号显示名。
- `SourceIdentity` 保存来源侧身份，如微信昵称、手机号、系统账号。
- `IdentityCandidate` 保存候选映射。
- `Person` 只保存已确认人物。

边界：

- 未确认身份只能产生 `mention:*` 或 `source_identity:*` 标签。
- 已确认人物才能产生 `person:*` 标签。
- 身份修改应由传感器/身份链路产生 `IdentityRevisionEvent`。
- 图谱层接收身份投影后重建人物标签、人物索引和相关摘要。

### 2.4 标签层

```text
TagDefinition
-> TagAssignment
-> TagIndex
```

职责：

- `TagDefinition` 定义标签含义、适用对象、冲突策略和赋值方式。
- `TagAssignment` 记录某个标签被赋给哪个对象、证据是什么、置信度是多少。
- `TagIndex` 支撑 `tag -> event/summary/evidence/source` 的快速检索。

边界：

- 标签不是原文。
- 标签不是摘要。
- 标签不是关系状态。
- 标签是语义索引和事件签名。

### 2.5 时间轴层

```text
SourceTime
EventTime
TargetTime
TimelineIndex
```

需要区分三类时间：

| 类型 | 含义 | 示例 |
| --- | --- | --- |
| `source_captured_at` | 信息被采集的时间 | 微信窗口捕获时间 |
| `event_occurred_at` | 事件表达或发生时间 | 对方说这句话的时间 |
| `event_target_time` | 事件指向的未来/过去时间 | 明天下午评审 |

时间轴索引必须同时支持：

```text
精确时间点
时间区间
相对时间
业务时间窗
对话顺序
事件因果顺序
```

## 4. 原文保存方案

### 4.1 完整原文归档

每次来源输入必须先生成 `SourceArchive`：

```json
{
  "source_archive_id": "src_20260704_wechat_001",
  "source_type": "wechat_desktop",
  "platform": "wechat",
  "captured_at": "2026-07-04T10:30:00+08:00",
  "raw_text_ref": "runtime/graph-memory/source-archives/src_20260704_wechat_001/raw.txt",
  "raw_payload_ref": "runtime/graph-memory/source-archives/src_20260704_wechat_001/raw.json",
  "content_hash": "sha256:...",
  "delete_state": "active"
}
```

规则：

- `raw.txt` 保存可读原文。
- `raw.json` 保存来源元数据、原始字段、线程、参与者线索、附件引用。
- `checksums.json` 保存 hash，防止原文被悄悄改变。
- `manifest.json` 作为该源归档的入口。

### 4.2 长内容分块

内容爆炸时，不能把整段原文都交给抽取器。必须先分块：

```text
SourceArchive.raw.txt
-> SignalFrame(offset_start, offset_end, time_span, speaker_hint)
-> RawObservation(observed_content, source_fidelity)
```

分块原则：

- 聊天：按消息、发言人、时间戳拆。
- 长文：按段落、标题、语义边界拆。
- 图片/OCR：按区域、行、表格单元拆。
- 音频：按说话人、静音段、语义停顿拆。
- API/日志：按记录、字段、状态变化拆。

每个分块必须记录：

```text
source_archive_id
frame_id
offset_start / offset_end
time_span
content_hash
decode_quality
```

### 4.3 原文核对

每个事件和标签进入索引前必须通过原文核对：

```text
event.evidence_refs
-> RawObservation
-> SignalFrame
-> SourceArchive.raw_text_ref
-> offset substring
-> hash check
```

通过标准：

- 证据短片段必须是原文子串。
- offset 指向的内容必须和证据片段一致。
- 原文 hash 必须和归档 manifest 一致。
- 摘要不能作为唯一证据。
- 模型改写文本不能作为原始证据。

## 5. 标签提取核对方案

标签提取分三步：

```text
候选标签生成
-> 注册表核对
-> 证据反读核对
```

### 5.1 候选标签生成

候选来源可以并行：

- 规则抽取：来源、时间、模态、显式关键词、数值单位。
- 模型抽取：语义角色、情绪、意图、边界、冲突。
- 手动标注：用户补充标签。
- 传感器标注：设备状态、来源类型、截图区域、OCR 置信度。

### 5.2 注册表核对

所有标签必须生成 `TagAssignment`：

```json
{
  "tag_value": "event_family:budget_status",
  "subject_type": "NestedEvent",
  "subject_ref": "nested_event_001",
  "assigned_by": "rule",
  "evidence_refs": ["frame_001", "raw_obs_001"],
  "confidence": 0.9,
  "review_state": "accepted"
}
```

核对规则：

- `tag_value` 必须命中 `TagDefinition`。
- 标签值不能是长句。
- 标签不能包含敏感身份明文。
- 物理量必须有单位。
- 情绪推断必须有置信度和 `needs_review` 机制。
- 外部动作必须有 `external_action:blocked|allowed_with_confirmation`。

### 5.3 证据反读核对

每个重要标签必须能解释自己为什么存在：

```text
TagAssignment
-> subject_ref
-> event.evidence_refs
-> RawObservation
-> original substring
```

例如：

```text
emotion.appraisal:pressure
-> evt_010_avoid_repeated_prompting
-> evidence: "压力会变大"
-> raw_text: "她说最近不要一直催她回复，压力会变大。"
```

## 6. 多索引检索设计

内容爆炸后不能靠遍历事件。必须建立倒排索引和时间索引。

### 6.1 必备索引

| 索引 | 入口 | 返回 |
| --- | --- | --- |
| `PersonIndex` | `person_id` | events, summaries, relationships, source_identities |
| `SourceIdentityIndex` | `source_identity_id` | mentions, candidate persons, events |
| `MentionIndex` | `mention_id` | frames, observations, events |
| `EventIndex` | `event_family/event_type` | events, nested_events |
| `TagIndex` | `tag_value` | subjects, events, summaries, evidence_refs |
| `TimelineIndex` | time bucket / range | frames, events, summaries |
| `SourceIndex` | source/thread/platform | archives, episodes, events |
| `EvidenceIndex` | evidence_ref | raw_observation, frame, archive, offset |
| `SummaryIndex` | summary scope | summary_shards |
| `FingerprintIndex` | content hash | duplicates, archive refs |

### 6.2 查询计划

用户问题先转成 `QueryPlan`：

```json
{
  "query_id": "query_xxx",
  "intent": "answer_with_evidence",
  "constraints": {
    "persons": ["person:zhang_001"],
    "source_identities": [],
    "event_families": ["budget_status", "technical_review"],
    "tags": ["state:pending"],
    "time_range": {
      "mode": "recent",
      "start": null,
      "end": null
    },
    "goal_domain": "sales_customer_progress"
  },
  "must_include": ["evidence_ref", "source_archive_ref"],
  "cold_read_required": false
}
```

### 6.3 热路径和冷路径

热路径用于快速回答：

```text
QueryPlan
-> PersonIndex / TagIndex / TimelineIndex / EventIndex
-> candidate event ids
-> SummaryShard
-> evidence snippets
-> ContextSnapshot
```

冷路径用于核对和高风险场景：

```text
ContextSnapshot.evidence_refs
-> EvidenceIndex
-> RawObservation
-> SignalFrame
-> SourceArchive.raw.txt
-> original substring
```

进入冷路径的条件：

- 用户要求“看原文”或“核对来源”。
- 关系状态变更、身份合并、外部动作。
- 高风险标签，如 `risk:high`、`conflict:*`、`confirmation:requires_confirmation`。
- 热路径召回结果冲突或置信度不足。
- 摘要和原文证据不一致。

## 7. 精确检索流程

### 7.1 人物检索

```text
person:zhang_001
-> PersonIndex
-> related source_identity / mention
-> events
-> tags
-> summaries
-> evidence_refs
-> source_archive_refs
```

人物检索必须返回：

- 已确认人物相关事件。
- 该人物绑定的来源身份。
- 未确认但可能相关的候选 mention。
- 事件时间轴。
- 原文证据入口。

### 7.2 事件检索

```text
event_family:budget_status
event:budget_confirmation_pending
-> EventIndex / TagIndex
-> NestedEvent / SemanticEvent
-> SummaryShard
-> evidence_refs
```

事件检索必须区分：

- 事件族。
- 具体事件类型。
- 父事件和子事件。
- 候选事件和确认事件。
- 冲突事件组。

### 7.3 时间轴检索

```text
time_range=last_7_days
-> TimelineIndex
-> events sorted by event_occurred_at
-> source frames sorted by source_captured_at
-> target future actions sorted by event_target_time
```

时间轴必须能回答：

- 最近发生了什么。
- 某人某段时间说过什么。
- 哪些事件指向未来。
- 哪些事件在同一时间窗冲突。
- 某个关系或销售目标的推进顺序。

### 7.4 标签组合检索

```text
tag intersection:
person:zhang_001
AND event_family:technical_review
AND time.relative:tomorrow_afternoon
```

执行顺序：

1. 先用低基数硬约束缩小范围，例如人物、时间、目标域。
2. 再用事件族和标签求交集。
3. 再用摘要和语义检索补充相近内容。
4. 最后用证据核对确认可引用内容。

## 8. 排序和裁剪策略

候选事件过多时，不能全部进入上下文。建议评分：

```text
score =
  exact_person_match * 0.25
  + exact_event_match * 0.20
  + tag_overlap * 0.15
  + time_relevance * 0.15
  + evidence_quality * 0.10
  + goal_relevance * 0.10
  + risk_or_conflict_priority * 0.05
```

硬过滤优先于评分：

- 删除状态为 deleted 的源不参与。
- 没有 evidence_ref 的事件不进入可回答上下文。
- 关系状态写入、身份合并、外部动作必须走确认门。
- 用户指定人物或时间时，不符合硬约束的事件默认排除。

裁剪原则：

- 每个问题先取 Top K 事件。
- 每个事件只带摘要、关键标签、短证据、原文引用。
- 需要核对时再展开完整原文。
- 冲突事件不能裁掉其中一方，必须成组进入上下文。

## 9. ContextSnapshot 组装

`ContextSnapshot` 是模型回答和工具调用前的唯一推荐输入。

建议结构：

```json
{
  "schema_version": "context_snapshot.v1",
  "snapshot_id": "ctx_xxx",
  "query_plan": {},
  "matched_events": [],
  "summary_shards": [],
  "evidence_bundle": [],
  "source_archive_refs": [],
  "tag_explanations": [],
  "timeline_view": [],
  "identity_state": {
    "confirmed_persons": [],
    "candidate_identities": []
  },
  "risk_and_boundary": {
    "relationship_state_write_allowed": false,
    "identity_merge_allowed": false,
    "external_action_allowed": false,
    "confirmation_required": []
  },
  "missing_or_conflicting_information": [],
  "answer_scope": "summary_with_evidence_refs"
}
```

组装顺序：

```text
QueryPlan
-> Index intersection
-> CandidateEventSet
-> Rank and trim
-> SummaryShard read
-> EvidenceBundle read
-> Cold read if required
-> Boundary checks
-> ContextSnapshot
```

必须包含：

- 为什么召回这些内容。
- 命中了哪些人物、事件、时间、标签。
- 每个事件的证据短片段和原文引用。
- 哪些身份未确认。
- 哪些关系状态不可写。
- 哪些外部动作不可执行。
- 哪些内容冲突或缺证据。


## 11. 星云显示层级

星云是只读观察层，不是事实源。

建议显示层级：

| 层 | 节点 | 可见表达 |
| --- | --- | --- |
| L0 | SourceArchive | 小灰点或来源簇 |
| L1 | Person / SourceIdentity | 人物节点，确认身份更实，候选身份更虚 |
| L2 | EventThread | 事件线主节点 |
| L3 | SemanticEvent / NestedEvent | 事件颗粒节点 |
| L4 | Tag | 彩色标签环或边 |
| L5 | Evidence | 可点击证据锚点 |
| L6 | SummaryShard | 半透明摘要层 |
| L7 | ContextSnapshot | 查询时临时高亮组合 |

权重显示：

```text
V0 隐藏或极淡
V1 小点
V2 普通点
V3 加粗边或普通高亮
V4 明显高亮
V5 强高亮 + 需要确认标记
```

边界：

- 星云点击证据时必须回到 `evidence_ref`，再回到原文。
- 星云显示不能修改事实。
- 星云筛选和上下文组装共用索引，不单独维护一套事实。

## 12. 验收标准

失败即停止：

- 原文没有保存。
- 证据片段不能从原文反读。
- 标签未命中注册表。
- 人物标签指向未确认身份。
- 摘要成为唯一证据。
- 索引结果和原文证据不一致。
- 高风险事件没有确认门。

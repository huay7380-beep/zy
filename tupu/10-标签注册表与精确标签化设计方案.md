## 0. 本文目标

本文专门设计“标签注册表”和“精确标签化规则”。

目标不是让标签承载完整事实，而是让标签成为可控、可检索、可组合、可反读的语义索引层：

```text
传感器原始输入
-> 标准化事件 / 对象 / 人物线索
-> 自动或手动标签化
-> 标签注册表约束标签含义
-> tag/person/time/object/event query
-> 召回相关内容
-> 组装 ContextSnapshot
-> 交给模型回答或调用工具
```

设计目标：

1. 覆盖生活事件、物理世界、情感表达、抽象理解、关系互动、业务推进。
2. 标签本身保持精简，但含义必须精确。
3. 自动标签和手动标签使用同一套 `TagAssignment` 结构。
4. 标签能关联人物、时间、事件、物品、地点、情绪、目标、承诺、风险、证据。
5. 标签能反向召回原始内容，但不替代原文、事件对象或摘要。

## 1. 身份链路位置判断

用户补充的判断是正确的：身份获取和身份修正的入口更适合放在前端传感器链路，而不是由图谱层单独维护一套封闭身份系统。

建议边界：

| 层 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| 传感器层 | 捕获 `SourceIdentity`、人物线索、账号、昵称、通讯录、前端用户修正 | 不直接写关系状态 |
| 身份解析链路 | 生成 `IdentityCandidate`、`CanonicalPersonRef`、`IdentityRevisionEvent` | 不替代用户最终确认 |
| 图谱层 | 消费身份投影、按 `person_id` 建索引、发现冲突、生成修正请求 | 不私自确认身份合并 |
| 标签层 | 用 `person:<canonical_person_id>` 或 `source_identity:<id>` 做标签引用 | 不把未确认身份伪装成 confirmed person |

因此，身份修改的主链路建议是：

```text
用户在传感器/前端身份面板修改身份
-> 生成 IdentityRevisionEvent
-> 身份解析链路更新 CanonicalPerson 映射
-> 图谱层接收 identity projection
-> 重建 person tags / person indexes / summaries / nebula projection
```

图谱层仍可发起修正请求：

```text
GraphIdentityCorrectionRequest
```

但该请求应回到传感器身份链路处理，而不是图谱层绕过传感器直接改身份。

核心原则：

```text
原文不改
事件事实不随身份修正被删除
身份映射可以改
人物标签和索引随身份映射重建
每次身份修正保留审计记录
```

## 2. 外部表达基线

本方案借鉴以下稳定表达思路，不直接照搬为实现：

| 参考 | 借鉴点 |
| --- | --- |
| W3C PROV-O | 来源、活动、代理、派生关系，用于证据和来源追踪 |
| Schema.org Event / Person / Place / Thing | 常见世界对象的基础类型表达 |
| QUDT | 物理量、单位、维度、数值的规范表达 |
| W3C EmotionML | 情绪可用 category、dimension、appraisal、action tendency 表达 |
| 事件语义 / 语义角色 | actor、action、object、recipient、instrument、time、place、cause、result |

本系统的实现取向：

```text
标签用于索引和语义签名
事件对象用于结构化事实
证据对象用于原文与来源
注册表用于约束标签含义
```

## 3. 标签注册表

### 3.1 TagDefinition

每个标签必须先在注册表中定义，不能无限制生成自由字符串。

建议结构：

```json
{
  "schema_version": "tag_definition.v1",
  "tag_id": "event.type.budget_confirmation_pending",
  "namespace": "event",
  "dimension": "type",
  "value": "budget_confirmation_pending",
  "label_zh": "预算确认待定",
  "label_en": "budget confirmation pending",
  "description": "表示预算仍未完成确认，只能作为候选销售推进事件。",
  "applies_to": ["SemanticEvent", "NestedEvent", "SummaryShard"],
  "parent_tag_id": "event.family.budget_status",
  "aliases": ["预算待确认", "budget pending"],
  "value_type": "controlled_value",
  "precision_level": "semantic_signature",
  "privacy_level": "non_secret",
  "source_of_authority": "tag_registry",
  "allowed_assignment_methods": ["rule", "model", "manual"],
  "conflict_policy": "allow_conflict_group",
  "deprecated": false,
  "version": 1
}
```

### 3.2 TagAssignment

标签赋值必须记录来源、证据、方法和状态。

```json
{
  "schema_version": "tag_assignment.v1",
  "assignment_id": "tag_asg_001",
  "tag_id": "event.type.budget_confirmation_pending",
  "tag_value": "event:budget_confirmation_pending",
  "subject_type": "NestedEvent",
  "subject_ref": "nested_event_xxx",
  "assigned_by": "rule|model|manual|sensor",
  "assigned_at": "2026-07-04T14:30:00+08:00",
  "evidence_refs": ["frame_xxx", "raw_obs_xxx"],
  "confidence": 0.86,
  "status": "active",
  "review_state": "accepted|needs_review|rejected",
  "created_from": "prewrite_tag_signature_gate.v1"
}
```

手动标签不绕过结构，只是 `assigned_by=manual`，并记录用户、时间和原因。

## 4. 标签命名空间

### 4.1 路由标签

用于快速缩小范围。

```text
source:<source_type_or_platform>
modality:<text|image|audio|video|api|sensor|ocr|file>
domain:<sales_customer_progress|romance_relationship_maintenance|life|physical_world|work|health_note>
scene:<wechat_group|dm_chat|home_sensor|crm|invoice_ocr|manual_note>
time_bucket:<today|yesterday|this_week|future|history>
privacy:<public|private|sensitive|secret>
```

### 4.2 人物和身份标签

人物标签必须区分三层：

```text
mention:<mention_id>
source_identity:<source_identity_id>
person:<canonical_person_id>
```

规则：

- 未确认身份只能使用 `mention:*` 或 `source_identity:*`。
- 已确认身份才能使用 `person:*`。
- 人物标签不直接保存手机号、微信号、邮箱等敏感明文。
- 身份变更后，所有 `person:*` 派生标签和索引必须重建。

### 4.3 事件语义标签

用于反推事件签名。

```text
event_family:<budget_status|relationship_boundary|payment|physical_observation|object_location>
event:<specific_event_type>
intent:<request|promise|deny|accept|warn|prohibit_action|inform|ask|prefer>
polarity:<positive|negative|uncertain|conditional|conflict|prohibit|promise>
state:<pending|confirmed|blocked|changed|missing|detected>
```

### 4.4 语义角色标签

描述谁对什么做了什么。

```text
actor:<person_id_or_source_identity_or_system>
actor_role:<customer|user|romantic_target|manager|finance|sensor|system|family_member>
target_object:<object_or_concept_id>
object_type:<document|payment|budget|person_boundary|physical_object|place|task|emotion_state>
recipient:<person_id_or_group>
instrument:<tool_or_channel>
```

### 4.5 时间标签

时间标签应支持自然语言和标准化时间并存。

```text
time.absolute:<iso_datetime_ref>
time.relative:<tomorrow|tonight|next_week|after_friday>
time.duration:<duration_ref>
time.deadline:<deadline_ref>
time.sequence:<before|after|during|repeat>
```

标签里不直接放长时间描述；具体时间放事件字段，标签放引用或 bucket。

### 4.6 物理世界标签

用于物品、位置、状态、物理量。

```text
object.type:<invoice|phone|jacket|medicine|device|door|package>
object.instance:<object_id>
object.attribute:<color|size|material|owner|brand>
location:<place_id_or_area>
physical.quantity_kind:<temperature|amount|distance|heart_rate|weight|count>
physical.unit:<C|CNY|kg|m|bpm|piece>
physical.value_bucket:<high|low|mismatch|normal|unknown>
physical.state:<open|closed|missing|detected|moved|damaged>
```

具体数值不建议直接塞入标签；应保存在结构化字段：

```json
{
  "quantity_kind": "amount",
  "value": 12800,
  "unit": "CNY"
}
```

标签只放：

```text
physical.quantity_kind:amount
physical.unit:CNY
physical.value_bucket:mismatch
```

### 4.7 情绪和关系表达标签

情绪不只用一个词，需要区分类别、维度、评估和行动倾向。

```text
emotion.category:<anger|fear|sadness|joy|anxiety|warmth|insecurity>
emotion.valence:<positive|neutral|negative>
emotion.arousal:<low|medium|high>
emotion.appraisal:<uncertainty|loss_of_control|being_ignored|pressure>
emotion.action_tendency:<avoid|approach|repair|withdraw|seek_reassurance>
relationship_signal:<attention_insecurity|needs_space|boundary_warning|repair_intent>
```

规则：

- 明确表达的情绪可自动打标签。
- 推断出的心理动机必须降低置信度，必要时进入 `needs_review`。
- 不把一次情绪表达直接写成关系状态。

### 4.8 抽象理解标签

用于目标、承诺、义务、风险、偏好和冲突。

```text
goal:<sales_close|relationship_repair|reduce_pressure|collect_payment>
commitment:<promised|not_promised|conditional>
obligation:<must|should|must_not|optional>
permission:<allowed|blocked|needs_approval>
preference:<prefer_call|prefer_text|avoid_pressure>
risk:<low|medium|high|critical>
conflict:<budget_status|amount_mismatch|identity_conflict|instruction_conflict>
certainty:<observed|claimed|inferred|uncertain>
confirmation:<observed_candidate|requires_confirmation|confirmed|rejected>
```

### 4.9 证据和检索标签

用于从标签回到内容。

```text
retrieval:event
retrieval:evidence
retrieval:summary
evidence_ref:<frame_id>
raw_observation_ref:<raw_obs_id>
semantic_unit_ref:<sem_id>
source_archive_ref:<source_archive_id>
```

## 5. 标签如何组装上下文

用户提问后，系统应把问题解析成查询约束。

示例：

```text
用户问：张总最近关于预算和技术评审说了什么？
```

查询计划：

```json
{
  "person": "person:zhang_001",
  "tags": [
    "event_family:budget_status",
    "event_family:technical_review"
  ],
  "time_window": "recent",
  "must_include": ["evidence_ref", "source_archive_ref"],
  "output": "ContextSnapshot"
}
```

召回顺序：

```text
person tag
-> source identities / mentions
-> related events
-> event tags
-> evidence refs
-> summary shards
-> original source excerpts
-> ContextSnapshot
```

上下文组装必须说明：

- 为什么选入该内容。
- 关联了哪些标签。
- 哪些内容只是候选。
- 哪些关系或身份仍未确认。
- 是否存在冲突事实。

## 6. 标签验证规则

每个事件进入写入或索引前必须满足：

```text
有路由标签
有事件语义标签
有人物/主体标签或系统主体标签
有对象标签
有时间标签
有证据定位标签
有边界标签
标签数量在预算内
标签不承载完整原文
标签能召回事件、摘要、证据和源文本
```

强制失败条件：

- `person:*` 指向未确认身份。
- 标签值是长句或整段摘要。
- 情绪标签把推断当事实。
- 物理数值缺少单位。
- 事件标签没有证据。
- 外部动作未被标记为 blocked 或 allowed with confirmation。
- 身份变更后仍存在旧 `person:*` 标签指向。

### 7.2 详细测试

#### 事件 1：销售预算待确认

原文：

```text

```

```text
person:zhang_001

actor_role:customer_contact
target_object:budget
```

- 按 `person:zhang_001` 可召回。

## 8. 测试结论

十个事件覆盖了：

```text
人物
时间
事件
物品
地点
情绪
关系信号
抽象偏好
物理量
外部动作边界
风险
证据召回
```

当前标签方案可以实现用户描述的机制：

```text
自动/手动打标签
-> 标签关联所有相关内容
-> 按标签召回人物、事件、物品、时间、证据、摘要
-> 根据用户问题组装上下文
-> 交给模型解答或调用工具
```

但必须满足三个前提：

1. 标签必须先进入 `TagDefinition` 注册表。
2. 标签赋值必须走 `TagAssignment`，不能只散落在事件数组里。
3. 人物标签必须依赖传感器身份链路输出的 `CanonicalPersonRef`，身份修改后重建派生标签和索引。

## 9. 待用户确认的问题

1. 是否认可“身份修正入口在传感器链路，图谱层只消费身份投影和发起修正请求”。
2. 是否认可新增 `TagDefinition` 和 `TagAssignment` 作为标签注册表基础结构。
3. 是否认可人物标签分为 `mention:*`、`source_identity:*`、`person:*` 三层。
4. 是否认可物理数值不直接进入标签，而是进入结构化字段，标签只保存 quantity kind、unit、bucket。
5. 是否认可情绪标签采用 category、valence、arousal、appraisal、action tendency 的组合方式，而不是单一情绪词。

验证含义：

1. 每个 `TagAssignment.tag_value` 必须命中 `TagDefinition` 注册表。
2. 每个事件必须至少覆盖路由、事件语义、主体、对象/地点、时间、证据、边界/状态七类标签桶。
3. 每个证据短片段必须是原文子串，单片段不超过 42 字符，证据总压缩率不超过 0.72。
4. 标签必须能反推事件签名：目标域、来源、事件族、事件类型、主体、对象、时间、证据引用、源归档引用、边界状态。
5. 标签查询必须能找到指定人物、事件、物品、地点、情绪、风险和外部动作边界。
6. `ContextSnapshot` 必须能通过标签召回相关事件，并反读原始文本和证据片段。
7. 标签层不得产生关系状态写入、身份合并、真实外部动作。

### 11.7 下一步真实案例测试要求

用户后续给出真实事件时，测试应至少检查：

1. 是否生成独立 `TaggedEvent`，不把多事件压成一个粗事件。
2. 是否为每个事件生成 `TagAssignment`，且全部命中注册表。
3. 是否覆盖人物/身份、事件、时间、地点/物品、情绪/抽象意图、证据、边界状态。
4. 是否能通过标签查询找到指定人物和指定事件。
5. 是否能通过标签组合组装 `ContextSnapshot`。
6. 是否能从 `ContextSnapshot` 反读原始文本和证据短片段。
7. 是否保持证据精简，避免把标签或证据片段变成整段摘要。
8. 是否保持关系状态写入、身份合并、外部动作全部为 0。

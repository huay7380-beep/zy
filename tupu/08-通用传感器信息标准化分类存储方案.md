## 0. 本文定位

本文记录 `UniversalSensorObservationPacket.v1` 的第一版结构化方案，用于承接所有图谱相关的传感器信息接收、标准化拆分、分类存储、标签化读取、反向证据追溯和后续测试验证。

从本文开始，未来关于图谱相关的存储、分类、提取、标签、反读、验证、回测、星云展示、权重投影等内容，统一保存到 `D:\zhineng\tupu` 文件夹下。其他目录可以保存运行时产物、schema、测试脚本和数据文件，但理论方案、验证结论和图谱规则说明应回写到本目录。

本文不直接修改人际关系图谱规则，不确认任何关系状态，不进行身份合并，不启动真实外部动作。它只定义传感器信息进入图谱系统前的标准化数据结构和验证要求。

## 1. 设计目标

建立一套能承接多源传感器信息的统一格式，使任意输入都能被拆成可存储、可分类、可标签化、可检索、可反读、可验证的标准对象。

这里的传感器包括但不限于：

- 人类语言传感器：微信、聊天记录、语音转写、邮件、笔记、网页文本。
- 视觉传感器：截图、图片、视频帧、OCR、屏幕状态。
- 行为传感器：点击、浏览、打开文件、复制、发送、任务操作。
- 结构化传感器：CRM、表格、订单、日程、API、日志。
- 环境传感器：时间、地点、设备、网络、会话、运行状态。
- 人工传感器：用户手动输入、确认、修正、标注、删除。

目标不是把所有输入立即判断为事件，而是先把输入标准化为可验证的感知记录，再按规则提取人物、事件、标签、摘要和上下文。

## 2. 人类语言和思维习惯拆解

人类理解信息时通常按以下链路组织：

```text
来源 -> 场景 -> 时间 -> 人物/对象 -> 行为/状态 -> 意图/情绪/关系 -> 重要性 -> 可行动建议
```

因此，存储结构必须同时保留：

- 表达层：原文、原声、原图、原始字段。
- 感知层：一句话、一段音频、一块图像、一条 API 记录。
- 事实层：直接观察到的文本、数值、状态、动作。
- 语义层：承诺、请求、拒绝、计划、情绪、偏好、约束、风险。
- 事件层：谁在什么时间对什么对象做了什么，产生了什么影响。
- 关系层：只生成候选线索，不自动修改人际关系图谱。
- 检索层：按人物、事件、时间、来源、标签、目标域反读。

核心原则：

- 原始证据和模型理解必须分离。
- 直接观察和推断结论必须分离。
- 候选身份和确认身份必须分离。
- 摘要和原文必须分离。
- 星云展示和事实存储必须分离。

## 3. 现有语义理解技术拆解

现有语义理解技术在本系统中应被拆成以下可验证步骤：

1. 采集：获取原始文本、图片、音频、视频、API、日志或人工输入。
2. 解码：OCR、ASR、DOM 提取、表格解析、JSON 解析、日志解析。
3. 分帧：按时间、区域、消息、语义段、业务记录拆成 `SignalFrame`。
4. 直接观察：生成 `RawObservation`，只记录看见或听见的内容。
5. 语义单元：生成 `SemanticUnit`，识别语义角色和语言功能。
6. 实体提及：生成人物、组织、地点、产品、项目等 `EntityMention`。
7. 身份候选：生成 `IdentityCandidate`，但不自动合并身份。
8. 事件抽取：生成 `SemanticEvent` 和 `NestedEvent`。
9. 标签分类：生成多维 `TagProfile`。
10. 摘要分片：按人物、事件、时间、目标域、标签生成 `SummaryShard`。
11. 索引投影：生成面向检索和星云显示的 `IndexProjection`。
12. 证据回溯：所有语义对象必须能回到原始来源和证据片段。

## 4. 总体链路

```text
SourceArchive
-> CaptureEpisode
-> SignalFrame
-> RawObservation
-> SemanticUnit
-> EntityMention / IdentityCandidate
-> SemanticEvent / NestedEvent
-> TagProfile
-> SummaryShard
-> IndexProjection
-> ContextSnapshot
-> WeightProfile
-> NebulaProjection
```

新增关键层：

- `SignalFrame`：解决多源、多模态、时间对齐、局部区域定位问题。
- `RawObservation`：解决原始信号和语义理解之间缺少可验证中间层的问题。
- `SemanticUnit`：解决原文直接跳到事件导致误判的问题。

## 5. 标准交换格式

## 7. 对象定义

### 7.1 SourceArchive

原始证据层，只保存完整来源、原文、附件、截图、音频、视频、API 原始返回、哈希和隐私状态。

```json
{
  "source_archive_id": "src_xxx",
  "source_type": "wechat_desktop | audio | image | browser | api | manual | system_log",
  "modality": ["text", "image", "audio"],
  "platform": "wechat_desktop",
  "adapter_id": "desktop.wechat",
  "captured_at": "2026-07-04T10:30:00+08:00",
  "raw_ref": "runtime/source-archives/src_xxx/raw.txt",
  "artifact_refs": [],
  "raw_hash": "sha256:...",
  "privacy_level": "private",
  "delete_state": "active"
}
```

规则：

- 不写入解释性结论。
- 不写入确认关系状态。
- 不被摘要替代。
- 后续所有对象必须能追溯到 `source_archive_id`。

### 7.2 CaptureEpisode

一次采集行为，描述这次输入从哪里来、何时发生、如何采集、质量如何。

```json
{
  "episode_id": "ep_xxx",
  "source_archive_id": "src_xxx",
  "capture_method": "adapter | manual | import | api_polling",
  "operator": "system | user",
  "time_range": {
    "start": "2026-07-04T10:29:00+08:00",
    "end": "2026-07-04T10:31:00+08:00"
  },
  "participants_hint": ["客户A", "使用者"],
  "quality": {
    "completeness": 0.92,
    "ocr_confidence": null,
    "asr_confidence": null
  }
}
```

规则：

- 只说明采集行为，不判断事实成立。
- 可包含人物线索，但不能确认身份。
- 可包含质量评分，用于后续抽取可靠性判断。

### 7.3 SignalFrame

最小可定位信号片段，例如一句话、一段语音、一块图像区域、一条 API 记录、一次点击日志。

```json
{
  "frame_id": "frame_001",
  "episode_id": "ep_xxx",
  "frame_type": "utterance | image_region | audio_segment | api_record | screen_state | user_action",
  "modality": "text",
  "time_span": {
    "start": "2026-07-04T10:30:12+08:00",
    "end": "2026-07-04T10:30:15+08:00"
  },
  "content_ref": {
    "raw_text": "明天下午我可以和你确认合同细节",
    "offset_start": 120,
    "offset_end": 138
  },
  "spatial_ref": null,
  "speaker_hint": "客户A",
  "decode_quality": 0.98
}
```

规则：

- 用于对齐多模态和多时间片。
- 不直接写事件结论。
- 图像、视频、屏幕区域必须有 `spatial_ref` 或可替代定位信息。

### 7.4 RawObservation

直接观察层，只记录从 `SignalFrame` 里看见、听见或解析到的内容。

```json
{
  "raw_observation_id": "raw_obs_001",
  "frame_id": "frame_001",
  "observation_type": "text_statement | numeric_value | visual_object | status_change | api_field | user_action",
  "observed_content": "明天下午我可以和你确认合同细节",
  "observed_language": "zh-CN",
  "source_fidelity": "verbatim",
  "interpretation_allowed": false
}
```

规则：

- 忠于原始信号。
- 不改写，不概括，不推断。
- 允许记录解析结果，但必须标注解析质量。

### 7.5 SemanticUnit

语义单元层，把直接观察内容拆成语言学和认知意义上的最小可用单元。

```json
{
  "semantic_unit_id": "sem_001",
  "raw_observation_id": "raw_obs_001",
  "unit_type": "commitment | plan | request | question | refusal | emotion | preference | constraint | status | measurement | anomaly",
  "normalized_text": "客户A表示明天下午可以确认合同细节",
  "semantic_roles": {
    "agent": "客户A",
    "action": "确认",
    "object": "合同细节",
    "time": "明天下午",
    "beneficiary": "使用者",
    "condition": null,
    "cause": null,
    "result": null
  },
  "certainty": {
    "level": "extracted",
    "confidence": 0.86,
    "reason": "explicit_statement"
  }
}
```

规则：

- 可以进行标准化表达。
- 必须保留置信度。
- 必须保留证据回溯。
- 不能直接修改关系图谱。

### 7.6 EntityMention

实体提及层，记录文本、图像、日志或 API 中出现的人、组织、地点、产品、项目、物体等。

```json
{
  "mention_id": "mention_001",
  "surface_text": "客户A",
  "entity_type": "person | organization | product | place | object | project | account",
  "source_frame_id": "frame_001",
  "identity_status": "mention_only | candidate | confirmed",
  "candidate_entity_ids": ["person_candidate_001"],
  "requires_user_confirmation": true
}
```

规则：

- 未确认实体只能作为 `mention_only` 或 `candidate`。
- 身份合并必须经过用户确认。
- 同名、多账号、昵称、截图用户名必须进入候选身份流程。

### 7.7 SemanticEvent

事件层，记录结构化事件，不等同于关系状态写入。

```json
{
  "semantic_event_id": "evt_001",
  "event_family": "business_progress",
  "event_type_code": "contract_detail_confirmation_planned",
  "goal_domain": ["sales_customer_progress"],
  "occurred_at": "2026-07-04T10:30:12+08:00",
  "event_time_target": "2026-07-05T14:00:00+08:00",
  "participants": [
    {
      "role": "customer",
      "mention_id": "mention_001",
      "identity_status": "candidate"
    }
  ],
  "objects": ["合同细节"],
  "evidence_refs": ["frame_001", "raw_obs_001", "sem_001"],
  "confidence": 0.84,
  "confirmation_state": "candidate",
  "relationship_write_allowed": false
}
```

规则：

- 事件可以成为候选事实。
- 高风险事件、关系状态变化、身份合并必须进入确认流程。
- 事件必须能通过证据引用反读原文或原始片段。

### 7.8 NestedEvent

嵌套子事件层，用于复杂场景中一个大事件包含多个语义子事件的情况。

示例：

```text
客户说“价格可以接受，但要老板确认，最好下周给方案。”
```

可拆为：

- 价格接受倾向。
- 决策权不在当前对话者。
- 需要老板确认。
- 下周需要方案。
- 销售推进进入待方案节点。

规则：

- 第一版采用语义子事件粒度。
- 子事件可独立打标签、独立赋权、独立被检索。
- 子事件必须保留父事件和原始证据引用。

### 7.9 TagProfile

标签层，用于分类存储、快速检索、上下文组装和星云显示。

```json
{
  "target_id": "evt_001",
  "target_type": "semantic_event",
  "tags": [
    "source:wechat_desktop",
    "modality:text",
    "domain:sales_customer_progress",
    "event:contract",
    "intent:commitment",
    "time:future",
    "confirmation:candidate",
    "risk:medium",
    "retrieval:person",
    "retrieval:event"
  ]
}
```

标签必须多维并支持组合检索。

基础标签域：

```text
source:*        来源
modality:*      文本/语音/图片/视频/API/日志/人工输入
domain:*        销售/恋爱/工作/健康/财务/系统/通用关系
entity:*        人物/组织/地点/产品/项目/账号/物体
event:*         行为或事件类型
intent:*        请求/承诺/拒绝/计划/问题/解释/提醒
emotion:*       情绪、态度、关系温度
time:*          过去/现在/未来/长期/截止时间
risk:*          低/中/高/阻断
quality:*       OCR/ASR/缺失/冲突/不完整/低置信
confirmation:*  raw/candidate/confirmed/rejected/extraction_pending
privacy:*       私密/敏感/可共享/需隐藏
retrieval:*     person/event/source/time/tag/goal/evidence
weight:*        V0/V1/V2/V3/V4/V5
visual:*        nebula_visible/hidden/focus/faded
```

### 7.10 SummaryShard

摘要分片层，用于加速读取，不能替代原文。

摘要类型：

- `person_summary`
- `event_summary`
- `time_window_summary`
- `goal_domain_summary`
- `tag_summary`
- `source_thread_summary`
- `risk_summary`
- `confirmation_summary`

规则：

- 每个摘要必须列出覆盖范围。
- 每个摘要必须引用原始事件和证据。
- 摘要可以被重建。
- 摘要不能成为唯一事实源。

### 7.11 IndexProjection

索引投影层，用于快速读取和星云观察。

必须支持：

```text
source_id -> episodes / frames / observations
time_window -> frames / events / summaries
mention_id -> events / summaries / evidence
entity_candidate_id -> mentions / events
event_type_code -> events / nested_events
tag -> events / summaries / source_refs
goal_domain -> events / summaries / weight_profiles
content_hash -> source_archive / duplicate_candidates
confirmation_state -> candidates / confirmed / rejected / pending
```

## 8. 读取和反读规则

任何检索结果都必须支持两类读取：

1. 正向读取：从来源到事件。

```text
SourceArchive
-> CaptureEpisode
-> SignalFrame
-> RawObservation
-> SemanticUnit
-> SemanticEvent / NestedEvent
```

2. 反向读取：从标签、人物、事件或摘要回到原文。

```text
tag/person/event/time/goal
-> IndexProjection
-> SemanticEvent / SummaryShard
-> evidence_refs
-> RawObservation
-> SignalFrame
-> SourceArchive
```

验收要求：

- 通过任意核心标签必须能找到对应事件。
- 通过事件必须能找到所有标签。
- 通过事件必须能找到人物线索。
- 通过人物线索必须能找到来源证据。
- 通过摘要必须能回到原文或原始证据片段。
- 如果不能反读，必须标记为 `readback_failed`。

## 9. 分类验证要求

用户后续提供多种案例和场景时，每个案例都必须输出以下检查结果。

### 9.1 单独分类检查

检查输入是否被拆成独立对象：

```text
source_archive_created
capture_episode_created
signal_frames_created
raw_observations_created
semantic_units_created
entity_mentions_created
semantic_events_created
nested_events_created_when_needed
tag_profiles_created
summary_shards_created_when_needed
index_projection_created
```

任何缺失项必须说明原因：

```text
not_applicable
extraction_pending
insufficient_evidence
low_quality_source
requires_user_confirmation
schema_gap
```

### 9.2 标签覆盖检查

每个事件至少应覆盖以下标签维度：

```text
source
modality
domain
event
time
confirmation
retrieval
quality
```

如果事件涉及人物、情绪、风险、外部动作、关系变化、身份合并，还必须额外覆盖：

```text
entity
emotion
risk
privacy
weight
visual
```

标签覆盖不足时必须输出：

```json
{
  "tag_coverage_status": "failed",
  "missing_tag_dimensions": [],
  "reason": "",
  "repair_suggestion": ""
}
```

### 9.3 标签反读检查

每个测试案例都必须验证：

```text
tag -> event
tag -> summary
tag -> evidence
person_tag -> source evidence
event_tag -> nested events
time_tag -> source frames
goal_tag -> context snapshot candidates
confirmation_tag -> pending/confirmed/rejected queues
```

如果任一路径断裂，必须标记：

```json
{
  "readback_status": "failed",
  "broken_path": "tag -> evidence",
  "blocked_at": "semantic_event.evidence_refs",
  "required_fix": "add evidence_refs to event extraction result"
}
```

### 9.4 原文保留检查

必须验证：

- 原文是否完整保存。
- 摘要是否没有覆盖原文。
- 事件是否能回到原文片段。
- 删除状态是否可审计。
- 证据引用是否稳定。

### 9.5 关系和身份边界检查

必须验证：

- 未确认人物是否保持 `candidate`。
- 未确认身份是否没有合并进正式人物。
- 关系状态是否没有被自动写入。
- 高风险关系变化是否进入确认状态。
- 外部真实动作是否没有被自动执行。

```json
  "decision": "accepted | needs_schema_extension | needs_tag_extension | extraction_failed"
}
```

## 11. 通过标准

一个案例只有同时满足以下条件，才算该方案覆盖成功：

1. 输入被保存到原始来源层。
2. 输入被拆成可定位的 `SignalFrame`。
3. 原始观察和语义理解分离。
4. 人物、事件、标签、摘要都能独立存储。
5. 标签维度覆盖当前场景。
6. 能通过标签反读到事件、摘要和原始证据。
7. 未确认身份没有被自动合并。
8. 关系状态没有被自动写入。
9. 摘要没有替代原文。
10. 星云或可视化投影只读，不污染事实源。

若出现以下任一情况，必须判定为未通过或需要扩展：

- 无法保留原始来源。
- 无法定位证据片段。
- 标签无法反读到原文。
- 事件和人物分类混在一起。
- 摘要成为唯一事实源。
- 未确认关系被自动写入。
- 当前标签体系无法表达案例中的关键语义。

## 3. 事件到指标更新的代码对齐

理论公式需要落成纯函数，建议接口如下：

```js
export function applyRelationshipEvent(edge, event, rules) {
  const effectiveImpact = calculateEffectiveImpact(event, rules.global_formula_conventions);
  const metricDeltas = deriveMetricDeltas(event, effectiveImpact, rules.metric_update_formulas);
  const nextMetrics = clampMetrics(applyDeltas(edge.metrics, metricDeltas));
  const nextRiskTags = applyRiskTagDelta(edge.risk_tag_codes, event.risk_tag_delta);
  const nextPhase = derivePhase(edge.phase, nextMetrics, event, rules.phase_transition_rules);

  return {
    ...edge,
    metrics: nextMetrics,
    risk_tag_codes: nextRiskTags,
    phase: nextPhase,
    evidence: mergeEvidence(edge.evidence, event),
    audit: updateAudit(edge.audit)
  };
}
```

实现约束：

- 函数不能直接写文件。
- 输入和输出都要可序列化。
- 低证据置信度事件只能生成候选变化。
- 法律、安全、未成年人、医疗和暴力相关事件必须设置人工确认或专家复核。
- 任何指标变化都应能追溯到 `event_id`。

推荐测试样例：

| 事件 | 预期变化 |
| --- | --- |
| `promise_kept` | `trust` 上升。 |
| `promise_broken` | `trust` 下降，可能增加风险。 |
| `settlement_completed` | `economic_binding` 下降，`trust` 可能上升。 |
| `boundary_violation` | `boundary_clarity` 下降，`risk_level` 上升。 |
| `verified_support` | `trust`、`closeness` 或 `social_capital_value` 上升。 |
| `violence_or_threat` | `risk_level` 高优先级上升，并触发安全策略。 |

## 5. ContextSnapshot 对齐

决策层不应读取全量历史，而应读取上下文快照。

建议快照字段：

```json
{
  "context_snapshot_id": "ctx_<uuid>",
  "goal_id": "goal_<uuid>",
  "target_person_ids": ["person_<uuid>"],
  "relationship_edge_ids": ["rel_<uuid>"],
  "thread_ids": ["thread_<uuid>"],
  "current_relationship_summary": {},
  "recent_events": [],
  "high_weight_events": [],
  "active_risk_tags": [],
  "active_context_tags": [],
  "open_commitments": [],
  "evidence_gaps": [],
  "context_sufficiency_score": 0.72,
  "read_reasons": [
    "target_person_match",
    "recent_30d_events",
    "active_risk_tag"
  ]
}
```

与理论对齐：

- 目标状态树依赖 `goal_id` 和当前关系摘要。
- 记忆检索依赖 `thread_ids`、`edge_ids`、`recent_events`。
- 风险策略依赖 `active_risk_tags`。
- 是否能给建议依赖 `context_sufficiency_score`。
- 审计依赖 `read_reasons`。

## 7. 代码层工程回测对齐

本轮已经新增确定性代码回测层，入口为：

```powershell
npm run tupu:backtest
```

工程文件：

| 文件 | 对齐职责 |
| --- | --- |
| `schemas/goal-oriented-interaction-backtest.schema.json` | 定义理论/代码回测一致性产物契约。 |
| `packages/mvp-runtime/src/goal-oriented-interaction-backtest.mjs` | 解析 `tupu/05`，生成代码侧上下文、策略状态、决策、话术草稿、回写计划和一致性指标。 |
| `scripts/run-goal-oriented-backtest.mjs` | 命令入口，写出 runtime 审计产物和 `tupu/07` 汇总。 |
| `packages/mvp-runtime/tests/goal-oriented-interaction-backtest.test.mjs` | 验证理论解析、三场景十二轮回测、签名一致性和产物写出。 |
| `runtime/goal-oriented-backtests/**` | 每次工程回测生成的 JSON / Markdown 审计证据。 |
| `tupu/06-代码层工程实现与理论对齐说明.md` | 本轮代码层实现说明。 |
| `tupu/07-理论与代码工程回测一致性结果.md` | 最近一次工程回测的人类可读摘要。 |

代码侧对齐方式：

```text
tupu/05 理论逐轮回测
-> parseTheoryBacktestMarkdown
-> code ContextSnapshot / strategy_state / decision / writeback_plan
-> theory_signature 与 code_signature 对比
-> metrics 与 hard_exit_signals
-> runtime/goal-oriented-backtests/** 和 tupu/07
```

通过标准：

- `scenario_count=3`
- `turn_count=12`
- `required_theory_context_completion_rate=1`
- `full_code_context_completion_rate=1`
- `theory_code_match_rate=1`
- `strategy_evolution_rate=1`
- `message_draft_completion_rate=1`
- `writeback_completion_rate=1`
- `hard_exit_signals=[]`
- `gate_decision=goal_oriented_backtest_passed`

这一步解决的是“理论回测是否已经被代码读懂、复现并约束”的问题。它不等于完整实时自动对话系统；真实平台读取、低延迟上下文装配、生产级专家选择、指标更新纯函数和外部发送连接器仍按流程树问题台账继续推进。

## 8. DialogueAct 代码层状态

V3 图谱本身没有完整定义对话动作层，但问答记录明确提出这是下一步关键。

本轮回测层已经把每轮 `dynamic_strategy` 转为代码侧策略动作、具体 `message_draft`、人工确认和回写计划，因此“对话动作必须落成可执行草稿”已经进入工程回测。但生产级 `DialogueAct` schema 仍未单独发布，建议后续先定义最小枚举：

| 动作码 | 说明 | 可能影响 |
| --- | --- | --- |
| `acknowledge` | 承接对方表达和情绪。 | 降低抵触，维持亲密或信任。 |
| `empathize` | 共情和验证感受。 | 增加情绪安全，但可能增加情绪劳动。 |
| `clarify` | 澄清事实、角色、需求或时间。 | 增加边界清晰度和证据质量。 |
| `boundary_clarify` | 明确责任、权限、禁区。 | 增加边界清晰度，可能短期提高冲突。 |
| `offer_support` | 提供帮助。 | 增加信任和社会资本，也可能增加依赖。 |
| `offer_limited_support` | 限定范围的帮助。 | 保持关系同时控制风险。 |
| `request_commitment` | 索取承诺或确认。 | 增加法律、经济或执行确定性。 |
| `ask_for_evidence` | 索取证据或书面确认。 | 增加证据强度，可能提高对方防御。 |
| `deescalate` | 降温和暂停。 | 降低冲突升级。 |
| `exit_or_pause` | 安全退出或暂停互动。 | 降低风险，可能降低亲密或推进速度。 |
| `invite_next_step` | 邀约、约时间、进入下一步。 | 推进目标事件。 |

后续代码应把 `DialogueAct` 与事件和指标连接：

```text
DialogueAct
-> expected_event_type
-> expected_metric_effect
-> risk_constraints
-> message_draft_template
-> feedback_observation
```

## 9. 检索与分类对齐

最后一轮问答提出“应先精准分类还是构建检索算法”。代码对齐结论是：先稳定最小分类字段，再构建依赖这些字段的检索索引，两者同步推进。

最小分类字段：

- `primary_type_code`
- `secondary_type_codes`
- `event_type`
- `event_subtype`
- `risk_tag_codes`
- `context_tag_codes`
- `function_codes`
- `person_ids`
- `edge_ids`
- `thread_ids`
- `evidence_confidence`
- `occurred_at`

最小检索函数：

```js
export function buildContextSnapshot({
  goal,
  targetPersonId,
  threadId,
  maxRecentEvents = 20,
  includeHighRisk = true
}) {
  const edges = findRelationshipEdges({ targetPersonId });
  const recentEvents = findRecentEvents({ targetPersonId, threadId, limit: maxRecentEvents });
  const highRiskEvents = includeHighRisk ? findEventsByRisk({ targetPersonId }) : [];
  const openCommitments = findOpenCommitments({ targetPersonId });

  return scoreContextSufficiency({
    goal,
    edges,
    recentEvents,
    highRiskEvents,
    openCommitments
  });
}
```

实现时要把每次读取原因写入审计，避免模型“凭感觉”读取过多或过少记忆。

## 12. 对齐结论

理论到代码的核心对齐关系是：

```text
关系本体
-> relationship_types / RelationshipEdge

事件变化
-> relationship_event_schema / RawEvent / SemanticEvent

风险和场景
-> risk_tags / context_tag_codes / function_codes

目标规划
-> SocialGoalInput / ContextSnapshot / DecisionRecommendation

对话动作
-> DialogueAct / message_draft / TriggerPlan

执行反馈
-> FeedbackRecord / Writeback / Metric Update / Audit

理论工程回测
-> GoalOrientedInteractionBacktest / runtime/goal-oriented-backtests / tupu:backtest
```

后续只要每个新增字段、schema、运行时函数和测试都能放回这张对齐表，就能避免理论文档和代码实现分叉。

## 0. 核心判断

我认同“用公开案件做复杂事件验证”的方向，但必须给它明确边界：

```text
公开案件材料 = 复杂事件结构压力测试材料
不是法律判断材料
不是判决复刻材料
不是关系状态推断材料
```

使用公开案件的价值在于：

- 人物多，身份和角色会变化。
- 事件多，存在父子事件、并行事件、前因后果。
- 时间线复杂，有发生时间、披露时间、记录时间、判决时间。
- 证据层级复杂，有直接证据、转述、推断、冲突证据。
- 叙述来源多，容易出现信息冲突和缺口。
- 能测试标签、时间轴、证据反读、上下文组装是否扛得住。

使用公开案件的限制：

- 不把判决结果当作系统抽取的事实。
- 不评价法律责任。
- 不根据裁判观点推断人物动机。
- 不把媒体报道的总结直接当作原始证据。
- 不让系统生成现实法律建议。

## 1. 当前 11 号方案的下钻问题

11 号总方案已经定义了人物、事件、时间、标签、原文核对和上下文组装的总体链路，但复杂事件下仍有这些问题需要补齐。

### 1.1 粒度问题

风险：

```text
切得太粗：一个复杂事件吞掉多个可检索子事件。
切得太细：每句话都变成事件，导致事件爆炸。
```

解决方向：

```text
以“语义子事件”为粒度。
一个子事件必须有独立检索价值、证据引用、角色结构和标签签名。
```

### 1.2 父子关系问题

风险：

```text
父事件摘要覆盖子事件事实。
子事件失去父事件上下文。
父事件权重掩盖真正关键的子事件。
```

解决方向：

```text
CompositeEvent 只做聚合和导航。
NestedEvent 保存可核对事实。
父事件不作为唯一事实源。
```

### 1.3 时间问题

风险：

```text
来源记录时间、事件发生时间、被披露时间、目标时间混在一起。
时间线排序错误导致因果误判。
```

解决方向：

```text
每个事件必须区分 source_captured_at、event_occurred_at、event_reported_at、event_target_time。
因果顺序不得只靠文本顺序判断。
```

### 1.4 证据问题

风险：

```text
摘要当证据。
转述当原文。
推断当事实。
冲突证据被裁掉。
```

解决方向：

```text
证据分级。
每个事件必须带 evidence_ref。
冲突证据必须成组保留。
高风险事件必须冷读原文核对。
```

### 1.5 人物角色问题

风险：

```text
同一人物在不同事件中角色变化。
称呼、来源身份、确认人物混用。
组织角色和个人角色混用。
```

解决方向：

```text
Person 是稳定身份。
EventRoleBinding 是事件中的临时角色。
同一人可在不同事件中拥有不同角色。
```

### 1.6 标签问题

风险：

```text
标签只能检索，不能说明事件结构。
标签变成长句，失去精简价值。
标签覆盖不到冲突、条件、否定、时间目标、证据等级。
```

解决方向：

```text
标签分为路由标签、语义签名标签、状态边界标签、证据定位标签、时间标签、角色标签。
标签只做索引和签名，不承载完整事实。
```

## 2. 复杂事件对象层

建议在 11 号总方案基础上补充以下事件对象层。

```text
CaseCorpus
-> SourceArchive
-> SourceEpisode
-> SignalFrame
-> RawObservation
-> SemanticUnit
-> EventThread
-> CompositeEvent
-> NestedEvent
-> AtomicFact
-> EvidenceAnchor
```

### 2.1 CaseCorpus

`CaseCorpus` 表示一个复杂事件测试语料包。

用途：

- 聚合一个公开案件或复杂真实场景的全部公开材料。
- 保存来源清单、采集时间、材料类型、可信等级和使用边界。
- 不保存判决结论作为系统事实。

建议字段：

```json
{
  "schema_version": "case_corpus.v1",
  "case_corpus_id": "case_corpus_public_xxx",
  "case_name": "public_case_placeholder",
  "usage_boundary": "event_structure_validation_only",
  "source_archive_ids": [],
  "excluded_uses": [
    "legal_judgment",
    "relationship_state_write",
    "identity_merge",
    "external_action"
  ]
}
```

### 2.2 EventThread

`EventThread` 表示一条事件线，例如“资金流线”“沟通线”“行动线”“证据披露线”。

用途：

- 避免把所有子事件塞到一个扁平列表。
- 支撑按线索检索和星云分层展示。

常见事件线：

```text
communication_thread
action_thread
money_thread
location_thread
document_thread
relationship_thread
conflict_thread
evidence_disclosure_thread
system_process_thread
```

### 2.3 CompositeEvent

`CompositeEvent` 表示聚合事件，不直接替代子事件事实。

示例：

```text
一次争议事件
一次销售推进节点
一次关系边界冲突
一次多方沟通会议
一次公开案件中的关键阶段
```

规则：

- 可以有摘要。
- 可以聚合子事件。
- 可以有权重。
- 必须列出 child_event_ids。
- 不作为唯一证据源。

### 2.4 NestedEvent

`NestedEvent` 是第一版核心事件粒度。

一个 `NestedEvent` 必须满足至少两项：

```text
有独立行动、状态、承诺、拒绝、风险、冲突、情绪、边界、条件或证据变化。
能被人物、事件、时间或标签独立检索。
能影响目标分析、权重分析、时间线或上下文组装。
有可回溯证据片段。
```

禁止生成 `NestedEvent` 的情况：

```text
只是修饰语。
只是重复前文。
没有证据引用。
纯模型猜测。
不能独立被检索。
```

### 2.5 AtomicFact

`AtomicFact` 是比 `NestedEvent` 更小的事实断言，不一定独立成为事件。

用途：

- 保存数值、时间、地点、身份称谓、状态字段。
- 支撑事件核对和冲突检测。

示例：

```text
金额=12800 CNY
时间=2026-07-04T14:05:00+08:00
地点=厨房地面
状态=payment_status:unpaid
```

规则：

- 数值、单位、时间必须结构化。
- `AtomicFact` 可被多个事件引用。
- 冲突事实进入 `ConflictSet`。

### 2.6 EvidenceAnchor

`EvidenceAnchor` 是证据定位单元。

字段应包括：

```text
source_archive_id
frame_id
raw_observation_id
offset_start / offset_end
quote_snippet
evidence_type
evidence_strength
```

证据类型：

```text
direct_original
verbatim_quote
structured_record
ocr_text
asr_text
third_party_report
summary_only
model_inference
```

其中 `summary_only` 和 `model_inference` 不能作为高风险事件的唯一证据。

## 3. 嵌套事件拆分规则

### 3.1 拆分触发器

出现以下任一条件，应考虑拆出 `NestedEvent`：

| 触发器 | 说明 |
| --- | --- |
| 新行动 | 某人做了、说了、发送了、拒绝了、承诺了 |
| 新状态 | 预算待确认、付款未完成、关系未确认 |
| 新条件 | 如果、除非、等到、先后顺序 |
| 新边界 | 不要做、不能发、需要空间、不想被追问 |
| 新冲突 | 两个来源说法不一致 |
| 新证据 | 新文件、新截图、新 OCR、新转述 |
| 新人物角色 | 同一人角色变化或新人物加入 |
| 新时间目标 | 明天、周五、付款后、下次见面 |
| 新风险 | 高风险、阻断、确认门 |
| 新目标影响 | 影响销售推进、关系维护或其他目标 |

### 3.2 不拆分条件

出现以下情况不拆：

- 只是礼貌寒暄。
- 只是重复同一事实。
- 只是形容词，没有新的行为或状态。
- 没有独立证据。
- 只是模型推断出的动机。
- 只是为了凑标签。

### 3.3 父子关系类型

父子事件边不只一种：

```text
HAS_CHILD_EVENT
CAUSES
ENABLES
BLOCKS
CONTRADICTS
QUALIFIES
SUPERSEDES
EVIDENCES
MENTIONS
PARTICIPATES_IN
```

示例：

```text
付款未确认 BLOCKS 仓库发货
OCR 金额 12800 CONTRADICTS PO 金额 11800
“不是要分手” QUALIFIES “需要空间”
```

### 3.4 事件层级上限

第一版建议最多四层：

```text
CaseCorpus
-> EventThread
-> CompositeEvent
-> NestedEvent
```

`AtomicFact` 是事实支撑，不进入主事件层级深度。

超过四层时，不继续加深，而是改用：

```text
thread_ref
parent_event_ref
relation_edges
summary_shard
```

## 4. 复杂事件标签扩展

复杂事件需要在 `tupu/10` 标签基础上补充几个方向。

### 4.1 事件结构标签

```text
event_scope:<atomic|nested|composite|thread|case>
event_relation:<cause|effect|condition|conflict|qualification|sequence|evidence>
event_thread:<communication|action|money|location|document|relationship|conflict|evidence_disclosure>
```

### 4.2 证据等级标签

```text
evidence.type:<direct_original|verbatim_quote|structured_record|ocr_text|asr_text|third_party_report|summary_only|model_inference>
evidence.strength:<strong|medium|weak|insufficient>
evidence.conflict:<none|same_source_conflict|cross_source_conflict|time_conflict|identity_conflict>
```

### 4.3 时间关系标签

```text
time.role:<source_captured|event_occurred|event_reported|event_target>
time.relation:<before|after|during|overlap|unknown_order>
```

### 4.4 事实状态标签

```text
fact_status:<observed|claimed|disputed|contradicted|superseded|unverified>
inference_level:<direct|normalized|inferred|speculative>
```

### 4.5 验证门标签

```text
verification:<passed|needs_cold_read|needs_cross_source_check|needs_user_confirmation|readback_failed>
```

规则：

- `speculative` 不得进入事实结论。
- `summary_only` 不得作为高风险事件唯一证据。
- `contradicted` 必须进入 `ConflictSet`。

## 5. 冲突与不确定性处理

### 5.1 ConflictSet

复杂事件必须允许多种说法并存。

`ConflictSet` 用于保存冲突组：

```json
{
  "schema_version": "conflict_set.v1",
  "conflict_set_id": "conflict_xxx",
  "conflict_type": "amount_mismatch",
  "claims": [
    {
      "claim_id": "claim_invoice_total",
      "value": "12800 CNY",
      "event_id": "evt_invoice_ocr",
      "evidence_ref": "evidence_001"
    },
    {
      "claim_id": "claim_po_total",
      "value": "11800 CNY",
      "event_id": "evt_po_record",
      "evidence_ref": "evidence_002"
    }
  ],
  "resolution_state": "unresolved"
}
```

规则：

- 冲突未解决前，不做单一结论。
- 上下文组装必须把冲突双方一起带入。
- 星云显示冲突边，不隐藏任一方。

### 5.2 UncertaintyProfile

每个复杂事件建议带不确定性档案：

```text
source_uncertainty
identity_uncertainty
time_uncertainty
causal_uncertainty
semantic_uncertainty
evidence_uncertainty
```

用途：

- 告诉模型哪些内容不能当作事实。
- 告诉用户哪里需要补证据或确认。

## 6. 案件式验证方法

### 6.1 为什么适合用公开案件

公开案件材料适合作为复杂事件验证，因为它天然包含：

- 多人物。
- 多来源。
- 长时间线。
- 复杂因果。
- 证据冲突。
- 叙事摘要和原始材料混杂。
- 法律结论和事实描述混杂。

这正好能测试系统是否能做到：

```text
只抽事件事实
保留证据边界
保留冲突
不根据判决反推动机
不把摘要当原文
不把结论当原始事实
```

### 6.2 公开案件选择标准

建议选择满足以下条件的公开材料：

```text
公开来源稳定
材料结构清晰
人物不少于 3 个
事件不少于 20 个
时间跨度不少于 3 个时间点
至少包含 2 个来源类型
至少存在 1 个冲突或不确定点
不涉及需要隐藏的私人敏感原文
```

材料可以来自：

- 公开裁判文书。
- 官方通报。
- 新闻报道。
- 时间线整理。
- 公开访谈或公告。

执行真实案件验证时，应优先保存来源链接、发布时间、原文片段和来源类型。若需要准确引用具体公开案件，必须重新核对公开来源，不靠记忆复述。

### 6.3 不使用判决结论的规则

判决或通报中的结论可以作为一个来源文本，但不能绕过事件抽取规则。

允许：

```text
把判决书中的事实描述拆成来源片段。
把判决书中的时间、人物、行为作为 claimed fact。
把判决书作为 source_archive。
记录“该文书声称 X”。
```

禁止：

```text
直接把裁判结论写成系统事实。
直接推断人物动机。
直接评价法律责任。
把“法院认为”当成原始现场证据。
```

### 6.4 案件式验证流程

```text
CaseCorpus 建立
-> SourceArchive 保存公开材料
-> SignalFrame 分块
-> RawObservation 原文观察
-> SemanticUnit 拆语义单元
-> EventThread 分线
-> CompositeEvent 聚合阶段
-> NestedEvent 抽子事件
-> AtomicFact 抽事实断言
-> TagAssignment 打标签
-> ConflictSet 保存冲突
-> TimelineIndex 建时间线
-> EvidenceIndex 建证据索引
-> ContextSnapshot 组装问题上下文
-> ReadbackAudit 原文核对
```

### 6.5 案件式验证问题集

验证时不问“谁对谁错”，只问结构问题：

1. 这个复杂事件可以拆成哪些事件线？
2. 每条事件线有哪些 `CompositeEvent`？
3. 每个 `CompositeEvent` 下有哪些 `NestedEvent`？
4. 每个 `NestedEvent` 的人物、时间、行为、对象、证据是什么？
5. 哪些事实是直接观察，哪些是来源声称，哪些是推断？
6. 哪些事件互为条件、因果、阻断、冲突或限定？
7. 哪些证据可以原文反读，哪些只能摘要级引用？
8. 哪些内容需要冷读核对？
9. 哪些冲突必须成组进入上下文？
10. 如果用户按人物、时间、事件、标签查询，是否能准确召回？

## 8. 复杂事件通过标准

一个复杂事件测试只有同时满足以下条件，才算通过：

```text
source_archives_created=true
case_corpus_boundary_declared=true
event_threads_created=true
composite_events_created=true
nested_events_created=true
atomic_facts_created_when_needed=true
every_nested_event_has_evidence_ref=true
every_high_risk_event_cold_read_passed=true
conflict_sets_created_when_conflict_exists=true
timeline_roles_separated=true
tag_assignments_registered=true
query_by_person_passed=true
query_by_event_passed=true
query_by_time_passed=true
query_by_tag_passed=true
context_snapshot_contains_conflict_groups=true
legal_judgment_generated=false
relationship_state_writes=0
identity_merges_applied=0
external_actions_executed=0
```

失败条件：

- 父事件吞掉子事件。
- 子事件没有证据引用。
- 公开材料摘要被当作原文。
- 判决结论被当作系统事实。
- 时间线混淆采集时间、发生时间和目标时间。
- 冲突事实只保留一方。
- 查询只能模糊召回，不能按人物/时间/事件/标签精确缩小。

## 9. 全局统筹后的解决方案

### 9.1 先固定层级，不急着扩实现

当前只优化方案，因此先固定层级：

```text
CaseCorpus
SourceArchive
SignalFrame
RawObservation
SemanticUnit
EventThread
CompositeEvent
NestedEvent
AtomicFact
EvidenceAnchor
ConflictSet
TimelineIndex
TagAssignment
ContextSnapshot
ReadbackAudit
```

### 9.2 先定义拆分门槛，再定义抽取器

不要一开始就让模型自由抽取事件。先定义：

- 什么可以成为事件。
- 什么只能成为事实。
- 什么只能成为标签。
- 什么必须保留为证据。
- 什么必须进入冲突组。
- 什么必须进入确认门。

### 9.3 公开案件作为第三类测试集

建议后续测试集分三类：

| 测试集 | 用途 |
| --- | --- |
| 小样例事件 | 验证标签注册、查询、原文反读 |
| 真实用户案例 | 验证当前目标域：销售推进、恋爱维护 |
| 公开案件式复杂语料 | 验证复杂事件、多人、多源、冲突、长时间线 |

### 9.4 不把案件测试接入人际关系图谱

公开案件式测试只验证事件图谱能力，不进入用户的人际关系图谱记忆。

建议边界：

```text
case_corpus.is_sandbox=true
relationship_graph_write_allowed=false
identity_projection_scope=case_local_only
```

### 9.5 将复杂事件能力反哺真实目标域

案件式测试解决的是通用复杂事件能力，最终反哺：

- 销售客户推进中的多方决策链。
- 恋爱关系维护中的边界、否定、情绪、冲突和时间变化。
- 多来源证据下的事实核对。
- 长期关系时间线。

## 10. 下一步建议

## 0. 总目标确认

当前总目标不是“做很多对象”，而是建立一条稳定、可审计、可扩展的图谱处理链路：

```text
多源传感器/人工输入
-> 源完整数据本地归档
-> 原始事件和语义事件拆分
-> 人物、时间、事件、标签、证据锚点绑定
-> 复杂事件、冲突事件、长期叙事对象组织
-> 按人物/事件/时间/标签/来源/目标域精确检索
-> 通过 EvidenceAnchor 冷读原文核对
-> 组装 ContextSnapshot / NarrativeContextSnapshot
-> 结合已确认的人际关系框架和当前事件权重生成建议
```

这条链路必须满足：

- 能接收所有类别事件，而不是只服务销售或恋爱场景。
- 当前优先验证销售客户推进和恋爱关系维护。
- 每条信息先进入源归档，再进入事件抽取。
- 原文永久保存，摘要和标签只能加速检索，不能替代原文。
- 人物、事件、时间、标签、来源、证据之间必须能双向追溯。
- 标签既支持自动打标，也支持用户手动打标。
- 标签检索结果必须能反读到事件、人物、时间、来源和原文。
- 复杂事件按语义子事件拆分，不逐句机械切分。
- 长周期叙事通过轨迹、阶段、转折、模式、语境、来源视角、因果假设组织。
- 星云是只读投影，不写事实、不改关系、不执行动作。
- 关系状态写入、身份合并、真实外部动作、学习权重转正必须另行确认。

## 1. 总目标是否足以覆盖所有事件

方案覆盖所有事件的方式不是预先枚举世界上所有事件类型，而是使用分层承载：

```text
原文层承载所有输入。
RawEvent 承载尚未完全理解的原始事件。
SemanticEvent / NestedEvent 承载已结构化的语义事件。
TagRegistry 承载可扩展分类。
EvidenceAnchor 承载原文反读。
ConflictSet 承载互相矛盾的说法。
TrajectoryRecord / PhaseSegment 承载长周期发展。
ContextFrame / SourcePerspective 承载语境和来源立场。
CausalHypothesis 承载未定论因果解释。
NarrativeIndex 承载长期叙事检索。
```

因此，遇到未知事件时不得强行分类。正确降级路径是：

```text
SourceArchive
-> RawEvent
-> EvidenceAnchor
-> tags:unknown_or_pending
-> extraction_status:needs_review
```

等标签、身份、事件类型或语境确认后，再升级为 `SemanticEvent`、`NestedEvent` 或叙事层对象。

## 2. 不可漂移的主链路

后续任何方案、字段、schema、runtime 都必须服务这条主链路：

```text
SourceArchive
-> SourceEpisode
-> RawEvent
-> SemanticEvent
-> NestedEvent / AtomicFact / Signal
-> EvidenceAnchor
-> PersonIndex / EventIndex / TimelineIndex / SourceIndex / TagIndex / FeatureIndex / EvidenceIndex
-> ComplexEvent / ConflictSet
-> PhaseSegment / TurningPoint / PatternClaim
-> TrajectoryRecord
-> ContextFrame / SourcePerspective / CausalHypothesis
-> NarrativeIndex
-> ContextSnapshot / NarrativeContextSnapshot
-> WeightProfile
-> NebulaProjection
```

其中：

- `SourceArchive` 是事实源入口。
- `EvidenceAnchor` 是原文反读入口。
- `PersonIndex`、`EventIndex`、`TimelineIndex`、`SourceIndex`、`TagIndex`、`FeatureIndex` 是热路径召回入口。
- `NarrativeIndex` 是长期叙事问题入口。
- `ContextSnapshot` 是模型决策上下文入口。
- `NebulaProjection` 是只读观察入口。

## 3. 目标漂移检测规则

新增文件、新对象、新字段、新流程前必须回答：

```text
它服务哪个总目标？
它的上游输入是什么？
它的下游输出给谁？
它是否保留 EvidenceAnchor？
它是否能反读原文？
它是否会替代原文、事实或用户确认？
它是否会触发 relationship_state、identity_merge、external_action、learning_weight_promotion？
它是否只是解释层对象？
它的验证方法是什么？
它失败时如何降级？
```

只要无法回答其中任一项，就不能进入 JSON Schema 草案。

```text

让 schema 前只处理真正阻塞的缺口。
保留后续扩展入口，但不扩散主线。
```

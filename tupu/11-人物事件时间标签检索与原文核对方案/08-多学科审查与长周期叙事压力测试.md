## 0. 审查结论

当前方案已经能较好应对：

- 复杂事件的分层存储。
- 父事件、子事件、原子事实、证据锚点之间的映射。
- 多来源冲突事实保留。
- 人物、事件、时间、标签、证据的基础检索。
- 公开案件式材料的结构化压力测试。

但对于“长周期发展叙事”的有效存储和提取，目前还只能达到 `partial_pass`：

```text
能保存长期事件序列。
能按人物、事件、时间、标签召回。
能通过 SummaryShard 做阶段性摘要。
但还缺少专门表达“长期轨迹、阶段变化、转折点、反复模式、社会语境、来源视角”的对象层。
```

因此，方案可以进入下一轮优化，但不建议直接进入 schema/test fixture。必须先补充长期叙事层和多学科解释边界。

## 1. 文档级压力测试结果

本轮未使用真实公开案件，不进行联网来源核对。测试使用“公开案件式复杂结构”和“长周期关系/销售叙事”模拟，目的是检查方案承载能力。

| 测试项 | 场景 | 当前结果 | 主要原因 |
| --- | --- | --- | --- |
| T1 | 多人多源合同争议 | `pass` | `EventThread`、`NestedEvent`、`AtomicFact`、`ConflictSet` 能表达多线索和冲突 |
| T2 | 销售客户 6 个月推进叙事 | `partial_pass` | 能存事件和摘要，但缺少阶段、转折点、长期轨迹对象 |
| T3 | 恋爱关系长期维护叙事 | `partial_pass` | 能保存边界/情绪事件，但缺少关系节奏、反复模式、互动基线 |

## 2. 复杂事件拆解与映射审查

### 2.1 当前可行点

当前对象模型已经形成可用映射：

```text
CaseCorpus
-> EventThread
-> CompositeEvent
-> NestedEvent
-> AtomicFact
-> EvidenceAnchor
-> ConflictSet
-> ContextSnapshot
```

这能处理：

- 一个复杂事件包含多条线索。
- 一个父事件包含多个子事件。
- 一个子事件引用多个原子事实。
- 一个事实或事件有多个证据锚点。
- 多个事实构成冲突组。
- 查询结果回到原文。

### 2.2 当前风险

仍需注意三个风险：

1. `EventThread` 目前偏“线索分类”，还不能表达长期阶段变化。
2. `CompositeEvent` 可以聚合阶段，但缺少阶段边界标准。
3. `NestedEventSplitGate` 能判断“拆不拆”，但还没定义“长期重复性事件如何聚合”。

### 2.3 解决方案

新增长期叙事对象：

```text
TrajectoryRecord.v1
PhaseSegment.v1
TurningPoint.v1
PatternClaim.v1
NarrativeIndex.v1
```

这些对象不替代 `NestedEvent`，只负责把长期事件组织成可解释叙事。

## 3. 长周期发展叙事测试

### 3.1 测试场景 A：销售客户 6 个月推进

输入结构：

```text
1 月：首次接触，客户表达兴趣。
2 月：技术评审，提出接口和白名单要求。
3 月：预算待确认，决策人未出现。
4 月：竞品报价出现，客户压价。
5 月：试点通过，但合同条款卡在付款周期。
6 月：客户仍未签约，内部采购流程延后。
```

当前方案能做：

- 每个事件拆为 `NestedEvent`。
- 技术、预算、合同、付款可以进入不同 `EventThread`。
- 冲突和阻断可进入 `ConflictSet`。
- 通过人物、事件、时间、标签召回相关内容。

当前不足：

- 缺少“销售推进轨迹”对象，无法直接表达“从兴趣到卡在付款周期”。
- 缺少“阶段变化”对象，无法稳定区分探索期、评审期、谈判期、阻断期。
- 缺少“转折点”对象，无法把竞品报价、付款条款变更标记为叙事转折。

结论：

```text
事件级存储=pass
长期叙事提取=partial_pass
```

### 3.2 测试场景 B：恋爱关系 4 个月互动变化

输入结构：

```text
第 1 月：互动频繁，主动邀约。
第 2 月：回复变慢，但仍保持见面。
第 3 月：出现“需要空间”“不要一直追问”。
第 4 月：否定分手，但沟通频率下降。
```

当前方案能做：

- 每个关系边界和情绪表达拆为 `NestedEvent`。
- 否定分手和需要空间可通过 `QUALIFIES` 连接。
- 压力、边界、偏好标签可召回。
- 高风险关系写入保持 blocked。

当前不足：

- 缺少互动基线对象，无法表达“回复变慢”相对什么变慢。
- 缺少长期节奏对象，无法表达“频繁 -> 变慢 -> 边界 -> 低频”的轨迹。
- 缺少重复模式对象，无法表达“多次催促都引发压力”。
- 缺少关系语境对象，无法区分玩笑、试探、边界、稳定关系规则。

结论：

```text
边界事件存储=pass
关系长期叙事=partial_pass
```

### 3.3 测试场景 C：公开案件式复杂材料

输入结构：

```text
官方通报：给出阶段性事实描述。
新闻报道：补充人物背景和时间线。
公开文书：包含事实描述和裁判观点。
当事人公开发言：包含主观叙述。
时间线整理：将不同来源重排。
```

当前方案能做：

- 所有来源进入 `SourceArchive`。
- 事实描述拆成 `NestedEvent` 或 `AtomicFact`。
- 冲突进入 `ConflictSet`。
- 判决结论不得直接写为系统事实。

当前不足：

- 缺少 `SourcePerspective` 表达来源立场、位置和利益关系。
- 缺少 `StatementChain` 表达“谁转述了谁”。
- 缺少 `NarrativeLayer` 区分事实叙述、解释叙述、法律叙述、媒体叙述。

结论：

```text
公开案件式结构验证=partial_pass
```

## 4. 多学科专家视角审查

### 4.1 人类学家视角

关注点：

- 同一行为在不同文化、关系、场景中的意义不同。
- 当事人的本土分类和系统外部分类要区分。
- 需要保存“语境”，而不只是事件。

当前方案优点：

- 原文保存和证据锚点能保留第一手材料。
- `EventRoleBinding` 能避免把人物身份和事件角色混在一起。

问题：

- 缺少 `ContextFrame` 保存情境、场景规则、互动习俗、当事人自己的分类。
- 标签体系偏分析者视角，缺少 emic/etic 双层分类。

建议新增：

```text
ContextFrame.v1
emic_label
etic_label
local_norm
interaction_setting
interpretation_scope
```

### 4.2 社会学家视角

关注点：

- 社会角色、权力关系、组织位置、网络结构影响事件意义。
- 长期模式比单次事件更重要。

当前方案优点：

- 人物、事件、时间、标签和冲突都能独立建模。
- `EventThread` 适合保存多条线索。

问题：

- `EventRoleBinding` 仍偏单事件角色，缺少稳定的社会位置和关系结构。
- 缺少群体、组织、制度、权力差异、依赖关系的层。

建议新增：

```text
SocialPositionBinding.v1
GroupContext.v1
InstitutionalContext.v1
PowerRelationMarker.v1
```

### 4.3 历史学家视角

关注点：

- 年代顺序不等于因果。
- 来源出处、版本、修订、缺失材料必须保留。
- 长周期叙事需要分期和转折。

当前方案优点：

- 多时间角色已经区分。
- `EvidenceAnchor` 和 hash 规则能保护来源。

问题：

- 缺少分期对象。
- 缺少版本化叙事。
- 缺少“资料空白”的显式对象。

建议新增：

```text
PhaseSegment.v1
TurningPoint.v1
SourceGap.v1
NarrativeVersion.v1
```

### 4.4 叙事学/话语分析视角

关注点：

- 谁在讲述。
- 用什么叙述框架。
- 什么被省略。
- 事件顺序如何被重排。

当前方案优点：

- 公开案件式验证已经意识到“判决结论不能当系统事实”。
- `ConflictSet` 能保留多说法。

问题：

- 缺少叙述者、叙述立场、话语类型。
- 缺少对“事实叙述”和“解释叙述”的区分。

建议新增：

```text
SourcePerspective.v1
StatementChain.v1
DiscourseType.v1
NarrativeClaim.v1
```

### 4.5 法证/证据审查视角

关注点：

- 证据链完整性。
- 直接证据、转述、推断必须严格区分。
- 高风险结论需要证据等级和交叉验证。

当前方案优点：

- `EvidenceAnchor` 很关键，方向正确。
- 冷读触发规则合理。

问题：

- 证据强度仍偏粗，缺少 chain-of-custody、提取方法、二次加工记录。
- OCR/ASR/PDF 提取物需要保存提取版本和校验状态。

建议新增：

```text
EvidenceDerivationRecord.v1
ExtractionMethodProfile.v1
EvidenceCustodyLog.v1
```

### 4.6 信息科学/检索专家视角

关注点：

- 大规模检索不能只靠事件列表。
- 长周期问题需要专门索引，不只是时间排序。
- 摘要层必须可重建、可失效、可追溯。

当前方案优点：

- `PersonIndex`、`TagIndex`、`TimelineIndex`、`EvidenceIndex` 方向正确。

问题：

- 缺少 `NarrativeIndex`。
- 缺少摘要版本和失效策略。
- 缺少查询类型分类：事实查询、叙事查询、冲突查询、趋势查询。

建议新增：

```text
NarrativeIndex.v1
SummaryInvalidationPolicy.v1
QueryIntentType.v1
```

### 4.7 认知科学/事件分割视角

关注点：

- 人类理解事件依赖边界、目标变化、角色变化和预测误差。
- 事件拆分不能只靠句子边界。

当前方案优点：

- `NestedEventSplitGate` 明确反对逐句机械切分。
- 拆分触发器包含行动、状态、条件、冲突、人物角色等。

问题：

- 缺少“目标状态变化”和“预期违背”的显式触发器。
- 缺少“事件边界置信度”。

建议新增：

```text
event_boundary_confidence
goal_state_change
expectation_violation
```

## 5. 关键缺口列表

### P1 缺口：长周期叙事对象缺失

影响：

- 能存长期事件，但不能稳定回答“这段关系/客户推进是如何发展到现在的”。

建议：

```text
TrajectoryRecord.v1
PhaseSegment.v1
TurningPoint.v1
PatternClaim.v1
NarrativeIndex.v1
```

### P1 缺口：语境层不足

影响：

- 对人际关系、社会互动、公开案件材料的解释可能过度事件化。

建议：

```text
ContextFrame.v1
SocialPositionBinding.v1
SourcePerspective.v1
```

### P1 缺口：因果关系和时间顺序仍需分离

影响：

- `CAUSES` 边可能被误用，时间先后可能被解释为因果。

建议：

```text
CausalHypothesis.v1
causal_status:<hypothesis|supported|disputed|rejected>
```

### P2 缺口：长期模式提取没有边界

影响：

- “经常”“逐渐”“反复”“越来越”这类模式可能被模型随意总结。

建议：

```text
PatternClaim.v1
evidence_window
occurrence_count
counter_examples
confidence
```

### P2 缺口：来源视角和转述链不足

影响：

- 公开案件和多方叙述中，容易把“某来源说”误当成“事实如此”。

建议：

```text
SourcePerspective.v1
StatementChain.v1
claim_voice:<direct|reported|interpreted|quoted>
```

## 6. 解决方案总纲

在现有 5 个方案件后，建议新增第 2 批方案件：

1. `TrajectoryRecord.v1`：长期轨迹对象。
2. `PhaseSegment.v1`：阶段划分对象。
3. `TurningPoint.v1`：转折点对象。
4. `PatternClaim.v1`：长期反复模式声明。
5. `ContextFrame.v1`：语境和本土分类。
6. `SourcePerspective.v1`：来源视角和立场。
7. `CausalHypothesis.v1`：因果假设，不直接等于事实。
8. `NarrativeIndex.v1`：长期叙事检索索引。

这些对象不替代当前 `NestedEvent`，而是在长期叙事层组织事件。

## 7. 推荐的长周期叙事链路

```text
NestedEvent / AtomicFact / EvidenceAnchor
-> EventThread
-> PhaseSegment
-> TurningPoint
-> PatternClaim
-> TrajectoryRecord
-> NarrativeIndex
-> NarrativeContextSnapshot
```

普通事件问题仍走：

```text
PersonIndex / TagIndex / TimelineIndex / EvidenceIndex
-> ContextSnapshot
```

长期叙事问题走：

```text
NarrativeQueryPlan
-> NarrativeIndex

-> PhaseSegment
-> TurningPoint
-> representative NestedEvents
-> EvidenceAnchors
-> NarrativeContextSnapshot
```

本篇定位：在 V3 人际关系图谱理论基础上，完善“目标导向人际交互辅助系统”的理论框架。它不只说明关系类型如何建模，还说明系统如何逐轮理解对话、设置目标、动态调整策略、处理事件、保存记忆、快速检索，并在心理学、逻辑学、社会学和人类学视角下检查遗漏。

## 1. 一句话结论

当前 V3 人际关系图谱已经完成关系本体、风险标签、群体节点、关系路径、生命周期和指标体系的基础覆盖；真正缺口不在“继续无限增加关系类型”，而在把图谱变成一个能围绕目标对象逐轮感知、推理、决策、输出和回写的动态系统。

优化后的理论核心是：

```text
目标导向人际交互 = 目标对象 + 关系状态 + 当日主题 + 历史阶段 + 当前话轮焦点 + 对方意图/情绪 + 事件证据 + 风险边界 + 动态策略 + 可执行话术 + 反馈回写
```

系统不应只回答“对方是谁”，而要回答：

1. 使用者现在想对这个目标对象达成什么。
2. 目标对象当前处于什么关系阶段和心理状态。
3. 当前这句话在推进、阻碍、试探、转移、求助、交换、冲突或风险上意味着什么。
4. 本轮策略应如何在保护使用者利益、尊重边界、保留证据和推进目标之间取平衡。
5. 生成的话术执行后，哪些事件、指标、关系阶段和策略需要回写。

## 2. 当前遗漏项检查

基于问答记录和 V3 图谱，原理论篇已经覆盖关系图谱本体，但对目标导向交互系统仍有以下遗漏。此版已将这些遗漏纳入理论框架。

| 遗漏项 | 原问题 | 本版补齐位置 |
| --- | --- | --- |
| 目标对象模型不够明确 | 只描述关系类型，未明确“对某个目标对象达成什么” | 第 7 节 |
| 逐轮上下文结构不完整 | 需要包含当日主题、历史关系阶段、当前聊天焦点、对方侧重和情绪 | 第 8 节 |
| 动态策略演化不足 | 策略应随话轮、反馈、风险和目标推进变化 | 第 9 节 |
| 对话动作层缺失 | V3 有图谱字段，但缺少语言动作到事件/指标的桥 | 第 10 节 |
| 通用事件处理流程不足 | 需要覆盖各种事件的处理流程和逻辑编排 | 第 11 节 |
| 跨学科审视不足 | 需要心理学、逻辑学、社会学、人类学共同检查 | 第 4-6 节 |
| 存储体系不够集中且可区分目标对象 | 需要集中但区分对象、标签化且快速检索 | 第 12 节 |
| 回测标准不明确 | 需要逐轮对话回测并验证上下文与策略演化 | 第 13 节和回测结果文件 |

## 4. 跨学科总框架

目标导向人际交互辅助系统必须同时满足四类学科视角。

### 4.1 心理学视角

心理学负责解释个体内部状态和互动动机。

系统每轮至少要观察：

- 情绪：焦虑、愤怒、恐惧、兴奋、防御、羞耻、内疚、冷淡、好奇。
- 意图：求助、试探、确认、推责、交换、邀约、拒绝、索取承诺、争取资源。
- 需要：安全、控制、认可、归属、收益、减少损失、恢复自尊。
- 边界：是否侵犯使用者边界，是否测试承诺上限。
- 依恋和依赖：是否出现过度依赖、回避、焦虑索取、反复确认。
- 修复能力：对方是否能道歉、补偿、改正、承认事实。

心理学不能被简化为“操控对方弱点”。在本系统中，它的用途是识别动机、减少误判、保护使用者边界，并让话术更符合对方真实状态。系统不应生成欺骗、胁迫或绕过同意的建议。

### 4.2 逻辑学视角

逻辑学负责区分事实、假设、推断、证据和结论。

系统每轮要区分：

- 事实：对方明确说了什么、做了什么。
- 证据：聊天记录、合同、截图、邮件、网页、第三方证明。
- 假设：系统推测出的潜在意图或风险。
- 推理链：为什么从当前话语推导出某个风险、事件或策略。
- 冲突：当前说法是否与历史记录、合同、承诺或前文矛盾。
- 反事实路径：如果顺从、拒绝、澄清、拖延、换条件，会分别导致什么。
- 结论等级：确定、较可能、低置信度、需要补信息、必须人工确认。

逻辑层的基本原则是：低证据不能支撑高强度结论，高风险不能靠单轮推断自动执行。

### 4.3 社会学视角

社会学负责解释角色、制度、权力、规范和资源。

系统要观察：

- 角色关系：客户、供应商、上级、下属、同事、朋友、家人、平台方、监管者。
- 权力结构：谁有评价权、审批权、资源分配权、合同解释权、组织话语权。
- 互惠结构：谁长期索取，谁长期付出，是否形成单向消耗。
- 社会资本：对方能带来信息、机会、声誉、资源、引荐或背书吗。
- 群体压力：公司、家庭、社群、圈层、平台规则是否影响行为。
- 制度边界：合同、流程、职业伦理、平台规则、法律义务。

社会学视角让系统不会把所有互动都当成私人心理问题。很多对话实际是组织角色、资源分配和制度权力的结果。

### 4.4 人类学视角

人类学负责解释文化、仪式、亲属、礼物交换和生命周期。

系统要观察：

- 文化语境：地域、行业、家庭、组织、宗教、圈层、平台文化。
- 礼物和人情：请客、送礼、帮忙、引荐、红包、资源互换背后的义务。
- 仪式节点：节日、婚丧、生日、入职、升迁、离职、见家长、签约、复盘。
- 亲属结构：血缘、姻亲、拟制亲属、照护关系、重组家庭、监护关系。
- 生命周期：儿童、学生、成人、育儿、养老、临终、身后事务。
- 身份归属：同乡、校友、战友、行业圈、社群成员、粉丝、平台用户。

人类学视角补足“同一句话在不同文化和仪式场景中含义不同”的问题。例如商业宴请、亲友借钱、客户节日问候和家族红白事不能只按普通聊天处理。

## 5. 系统理论分层

优化后的系统分为九层。

| 层级 | 名称 | 核心问题 | 输出 |
| --- | --- | --- | --- |
| L1 | 输入感知层 | 发生了什么，谁说了什么 | `RawEvent` |
| L2 | 语义解析层 | 这句话/动作是什么意思 | `SemanticEvent`、意图、情绪、风险候选 |
| L3 | 对象锚定层 | 涉及哪个人、组织、群体、合同或平台 | `entity_id`、`edge_id`、置信度 |
| L4 | 关系状态层 | 当前关系是什么阶段、什么风险和指标 | `RelationshipEdge`、`metrics`、`phase` |
| L5 | 上下文快照层 | 本轮决策需要读哪些记忆 | `ContextSnapshot` |
| L6 | 目标规划层 | 对这个目标对象要达成什么 | `GoalState`、目标阈值、停止条件 |
| L7 | 策略推演层 | 可选路径、成本、风险、收益是什么 | `StrategyState`、`ranked_options` |
| L8 | 对话执行层 | 这一轮应该怎么说、怎么做 | `DialogueAct`、`message_draft` |
| L9 | 反馈回写层 | 执行后发生了什么，系统如何更新 | `FeedbackRecord`、新事件、指标变化 |

每一层都必须可审计：输入是什么、输出是什么、依据是什么、置信度是多少、是否需要人工确认。

## 6. 总闭环逻辑

系统的闭环流程如下：

```text
用户目标
-> 目标对象识别
-> 历史关系读取
-> 当日主题压缩
-> 当前话轮解析
-> 对方意图/情绪/侧重分析
-> 事件候选和风险候选
-> 关系指标和阶段更新候选
-> 目标状态差距分析
-> 策略路径推演
-> 对话动作选择
-> 生成 message_draft
-> 人工确认
-> 执行反馈
-> 事件和关系回写
-> 索引刷新
-> 下一轮上下文更新
```

这个闭环保证系统不会只做静态分析，也不会只生成一次性话术。每一轮都要更新状态，并让下一轮策略继承当前结果。

## 7. 目标对象模型

目标导向系统必须区分全局目标和目标对象目标。

### 7.1 目标层级

```text
TotalGoal
└─ ScenarioGoal
   └─ TargetObjectGoal
      └─ ConversationRoundGoal
```

| 目标层 | 说明 | 示例 |
| --- | --- | --- |
| `TotalGoal` | 使用者长期目标 | 建立可验证的人际交互辅助闭环 |
| `ScenarioGoal` | 当前场景目标 | 推进某客户进入试点 |
| `TargetObjectGoal` | 针对目标对象的目标 | 让张总确认技术评审会议 |
| `ConversationRoundGoal` | 本轮话语目标 | 承接预算顾虑并争取半小时会议 |

### 7.2 目标对象字段

每个目标对象应有独立目标档案：

| 字段 | 说明 |
| --- | --- |
| `target_id` | 目标对象 ID。 |
| `target_kind` | 人、组织、群体、合同、平台、案件。 |
| `relationship_edge_ids` | 与使用者相关的关系边。 |
| `current_phase` | 当前关系阶段。 |
| `current_metrics` | 当前关系指标。 |
| `active_goal` | 当前针对该对象的目标。 |
| `minimum_success` | 最小成功条件。 |
| `ideal_success` | 理想成功条件。 |
| `hard_constraints` | 不可触碰限制。 |
| `stop_conditions` | 何时暂停或退出。 |
| `open_loops` | 未完成承诺、待确认事项、未解决冲突。 |
| `preferred_strategy` | 当前首选策略。 |
| `strategy_history` | 策略演化历史。 |

### 7.3 目标设定原则

目标必须具体、可验证、可回写。

不合格目标：

- “让对方喜欢我。”
- “让客户更信任我。”
- “把关系搞好。”

合格目标：

- “让客户在本周内确认 30 分钟技术评审会议。”
- “让对方明确延期原因并书面确认责任边界。”
- “在不借钱的前提下保留朋友关系，并提供非金钱帮助。”

## 8. 逐轮上下文模型

用户提出的上下文建议是正确的，但还需要补足。完整的逐轮上下文应包含 16 个部分。

| 模块 | 必要性 | 内容 |
| --- | --- | --- |
| `daily_topic_summary` | 必须 | 当日主题内容的精炼部分，只保留与本轮决策有关的信息。 |
| `historical_relationship_stage` | 必须 | 从历史对话、事件和指标中分析出的关系节点阶段。 |
| `current_chat_focus` | 必须 | 当前聊天正在围绕什么问题展开。 |
| `counterpart_focus` | 必须 | 对方当前最在意什么。 |
| `counterpart_emotion` | 必须 | 对方情绪和强度。 |
| `target_object_goal` | 必须 | 针对目标对象设立的目标。 |
| `dynamic_strategy` | 必须 | 根据上下文和目标构建的当前策略。 |
| `role_and_power_context` | 必须 | 双方角色、权力距离、资源控制和制度约束。 |
| `risk_boundary_state` | 必须 | 当前风险标签、边界状态、人工确认要求。 |
| `evidence_state` | 必须 | 事实、证据、假设和证据缺口。 |
| `open_commitments` | 必须 | 未完成承诺、待确认事项、对方未回应点。 |
| `conversation_momentum` | 推荐 | 对话推进、停滞、升级、降温或转移状态。 |
| `retrieved_memory_refs` | 必须 | 本轮读取了哪些记忆和原因。 |
| `context_sufficiency_score` | 必须 | 是否足以给出具体建议。 |
| `next_best_action` | 必须 | 本轮最合适的行动或话术方向。 |
| `writeback_plan` | 必须 | 本轮后需要记录什么反馈和事件。 |

### 8.1 上下文压缩规则

上下文不能简单拼接全部聊天记录。应按以下顺序压缩：

```text
目标对象 -> 当前线程 -> 当日主题 -> 最近互动 -> 高权重历史事件 -> 活跃风险 -> 未完成承诺 -> 当前话轮
```

只在以下情况读取完整原始记录：

- 摘要与当前话轮冲突。
- 证据不足但风险高。
- 用户要求查看原始记录。
- 法律、合同、安全或声誉事件需要复核。
- 系统准备改变关系阶段或高风险标签。

## 9. 动态策略演化模型

策略不是固定模板，而是随对话进行动态演化。

### 9.1 策略状态

| 策略状态 | 适用场景 | 目标 |
| --- | --- | --- |
| `observe_and_clarify` | 信息不足、关系早期、意图不明 | 收集信息，降低误判。 |
| `acknowledge_and_bridge` | 对方有情绪或跑题 | 承接对方，同时桥接回目标。 |
| `value_first` | 需要建立信任或推进合作 | 先给低风险价值。 |
| `evidence_and_boundary` | 出现责任、合同、金钱、承诺争议 | 留痕并明确边界。 |
| `conditional_exchange` | 对方索取资源或让步 | 以条件交换维护互惠。 |
| `commitment_request` | 目标进入可确认阶段 | 索取时间、会议、书面确认或下一步承诺。 |
| `deescalate_and_pause` | 情绪升级或冲突升高 | 降温、暂停、延后。 |
| `safety_exit` | 出现安全、胁迫、骚扰、暴力、勒索 | 停止普通策略，进入安全优先。 |

### 9.2 策略演化触发器

| 触发器 | 策略变化 |
| --- | --- |
| 对方给出明确信息 | 从澄清转入价值或承诺。 |
| 对方推迟或模糊回应 | 从推进转入障碍识别和低承诺请求。 |
| 对方情绪升高 | 从推进转入承接和降温。 |
| 对方索取资源但不对等 | 转入条件交换和边界澄清。 |
| 对方责任转嫁 | 转入证据和边界策略。 |
| 对方确认下一步 | 转入承诺记录和触发计划。 |
| 风险标签升高 | 转入人工确认或停止。 |

### 9.3 策略输出要求

每次策略输出都必须包含：

- 当前策略名。
- 选择原因。
- 目标状态差距。
- 本轮语言动作。
- 预期事件。
- 风险和停止条件。
- 反馈观察点。

## 10. 对话动作层

V3 图谱提供关系和指标，但要让 LLM 在逐句对话中行动，必须有对话动作层。

| 对话动作 | 作用 | 可能影响 |
| --- | --- | --- |
| `acknowledge` | 承认对方表达 | 降低防御，维持对话 |
| `empathize` | 情绪共情 | 提升情绪安全，也可能增加情绪劳动 |
| `clarify` | 澄清事实、需求、时间、角色 | 增强证据和边界 |
| `reframe` | 重构问题角度 | 把抱怨转成目标或选项 |
| `bridge_to_goal` | 桥接回使用者目标 | 防止跑题 |
| `offer_value` | 提供低风险价值 | 提升信任和社会资本 |
| `ask_for_commitment` | 索取承诺 | 推进目标状态 |
| `set_boundary` | 设置边界 | 防止责任、资源或情绪过载 |
| `conditional_exchange` | 条件交换 | 维护互惠 |
| `document_and_confirm` | 留痕确认 | 增强法律和证据稳定性 |
| `deescalate` | 降温 | 降低冲突 |
| `pause_or_exit` | 暂停或退出 | 控制风险 |

对话动作必须映射到事件和指标。例如 `ask_for_commitment` 预期触发 `promise_kept`、`meeting_scheduled` 或 `contract_next_step_confirmed`；`set_boundary` 预期提升 `boundary_clarity`，但可能短期增加 `conflict`。

## 11. 通用事件处理流程

目标是覆盖大多数事件，而不是为每个事件写孤立流程。

### 11.1 事件处理总流程

```text
事件进入
-> 来源可信度判断
-> 涉及对象和关系边定位
-> 事件类型识别
-> 情绪/意图/权力/风险解析
-> 证据充分度判断
-> 指标变化候选
-> 生命周期变化候选
-> 策略路径推演
-> 人工确认门槛判断
-> 话术或行动计划
-> 执行反馈
-> 回写事件和索引
```

### 11.2 事件类型与优先逻辑

| 事件类型 | 处理优先级 | 核心逻辑 |
| --- | --- | --- |
| 普通互动 | 正常 | 更新亲密、频率和主题。 |
| 信息支持 | 正常 | 更新信任、社会资本和互惠。 |
| 实际帮助 | 正常/中 | 更新信任、依赖、资源流。 |
| 承诺给出 | 中 | 写入 open commitment。 |
| 承诺兑现 | 中 | 提升信任和修复能力。 |
| 承诺违背 | 中/高 | 降低信任，检查风险。 |
| 边界试探 | 中 | 检查情绪劳动、资源索取、责任转移。 |
| 责任转嫁 | 高 | 留痕、澄清、拒绝接盘。 |
| 财务事件 | 高 | 记录金额、方向、凭证、期限。 |
| 合同/法律事件 | 高 | 证据优先、人工确认、避免口头承诺。 |
| 冲突升级 | 高 | 降温、延后、第三方或书面化。 |
| 隐私/名誉风险 | 高 | 限制信息、保存证据、必要时停止。 |
| 安全威胁 | 最高 | 停止普通互动策略，进入安全计划。 |
| 关系修复 | 中 | 验证道歉、补偿、非重复期。 |
| 关系终止 | 高 | 检查未结义务、资产、合同、安全风险。 |

### 11.3 决策逻辑顺序

系统必须按以下优先级编排逻辑：

```text
安全和法律风险
> 用户明确限制
> 证据充分度
> 目标对象关系阶段
> 当前话轮情绪
> 用户目标推进
> 沟通风格优化
```

这可以防止系统为了推进目标而忽略安全、边界和证据。

## 12. 集中但区分目标对象的标签化检索存储体系

### 12.2 集中与分离原则

集中：

- 所有关系类型、风险标签、对话动作、事件类型和检索索引统一维护。
- 所有事件都进入统一事件库，便于审计和重建索引。
- 所有读取原因和决策依据进入统一审计链。

分离：

- 每个目标对象有独立目标、阶段、策略和开放事项。
- 每个线程有独立当日主题、上下文快照和话轮记录。
- 隐私等级高的证据只保留引用和摘要，不进入普通策略层。
- 组织、群体、合同、案件和平台节点不混入个人关系边。

### 12.4 快速检索策略

检索应按“任务先行、目标对象优先、风险优先、证据优先”的顺序。

```text
输入当前话轮
-> 识别 target_id 和 thread_id
-> 读取 active_goal 和 strategy_state
-> 读取最近 N 轮 daily_topic_summary
-> 读取 active risk tags
-> 读取 open commitments
-> 读取高证据强度历史事件
-> 生成 ContextSnapshot
```

检索结果必须带 `read_reason`：

- `target_goal_active`
- `recent_turns`
- `open_commitment`
- `active_risk`
- `contract_or_money_related`
- `emotion_shift`
- `user_requested`
- `evidence_conflict`

没有读取原因的记忆不应进入决策。

## 15. 理论结论

目标导向人际交互辅助系统的本质不是“帮用户想一句漂亮话”，而是把人际互动拆成可持续运行的闭环：

```text
目标 -> 对象 -> 上下文 -> 事件 -> 关系状态 -> 策略 -> 话术 -> 反馈 -> 回写 -> 下一轮目标
```

V3 人际关系图谱提供了关系世界的地图；本理论篇补足了目标、上下文、策略、事件处理、存储检索和回测方法。后续工程实现应严格围绕这条闭环推进：先保证每条信息可落盘、可检索、可解释，再逐步提升策略质量和专家 Agent 能力。

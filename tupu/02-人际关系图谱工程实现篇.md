本篇目标：把理论篇中的人际关系图谱整理为可落地的工程架构。它重点回答数据如何进入系统、如何落盘、如何更新关系边、如何检索记忆、如何生成建议、如何回写反馈。

## 1. 工程边界

当前项目第一阶段聚焦 B2B 商务沟通和客户跟进，原则是“确定性工作流 + LLM 节点预留”，不是直接做全自动 agent。

因此工程实现必须满足以下边界：

- 外部平台交互默认 dry-run，不真实发送消息。
- 高风险事件和 P1 事件不得自动确认。
- 推荐动作必须尽量落成具体 `message_draft`，但需要人工确认。
- 真实聊天、网页或手工记录先组织为 `PilotImportBatch`，再进入导入和 MVP 闭环。
- 人物关系图谱负责对象和关系，事件图谱负责事件和线索，策略层只能读取脱敏摘要和约束。
- 每个新增能力先补 schema、状态记录和测试样例，再扩展复杂逻辑。

## 2. 总体工程架构

建议将 V3 图谱落成以下运行链路：

```text
外部输入
-> Intake Adapter
-> RawEvent
-> SemanticEvent
-> Entity Resolution
-> RelationshipEdge Instance
-> Metric and Risk Update
-> ContextSnapshot
-> Decision Cluster
-> TriggerPlan and MessageDraft
-> Manual Confirmation
-> FeedbackRecord
-> Writeback
-> Index Rebuild
-> Audit
```

这个链路与当前项目流程树一致：用户目标、人物关系、事件记录、决策建议、触发计划、平台快照验证、反馈、回写、索引、审计。

## 3. 输入层：Intake Adapter

输入层负责把外部材料转成可审计原始事件。

第一阶段可支持：

- 手工粘贴聊天片段。
- 用户整理后的 `PilotImportBatch`。
- 网页或平台页面快照。
- 截图/OCR 的后续扩展入口。
- 聊天工具导出文件的后续扩展入口。

每个输入源都必须转成 `RawEvent`，而不是直接送给决策层。原因是系统需要保留原始证据，后续才能解释为什么判断某个意图、风险或关系变化。

最小输入字段：

| 字段 | 说明 |
| --- | --- |
| `source_type` | 手工、聊天导出、网页、截图、平台快照等。 |
| `source_ref` | 原始来源路径、消息 ID、页面路径或用户标记。 |
| `occurred_at` | 事件发生时间。 |
| `captured_at` | 系统捕获时间。 |
| `participants` | 涉及人物或组织。 |
| `raw_text` | 原始文本。 |
| `privacy_level` | 隐私级别。 |
| `evidence_confidence` | 证据置信度。 |

## 4. 事件层：RawEvent 到 SemanticEvent

`RawEvent` 是原始证据，`SemanticEvent` 是语义化事件。

语义事件要回答：

1. 发生了什么。
2. 谁发起，谁受影响。
3. 属于哪类事件。
4. 对哪些关系边有影响。
5. 触发哪些风险标签。
6. 影响哪些指标。
7. 证据强度如何。

V3 的 `relationship_event_schema` 已提供可对齐结构：

```json
{
  "event_id": "evt_<uuid>",
  "edge_id": "rel_<uuid>",
  "event_type": "interaction | support | conflict | promise_kept | promise_broken | boundary | disclosure | financial | legal | caregiving | medical | romantic | safety | transition | ritual | platform | posthumous",
  "event_subtype": "free enum with canonical mapping",
  "occurred_at": "ISO-8601 or date",
  "initiator_entity_id": "person/org/group/platform or null",
  "target_entity_ids": ["ids"],
  "valence": "positive | neutral | negative | mixed",
  "impact_score": "0.0-1.0",
  "evidence_confidence": "0.0-1.0",
  "privacy_level": "public | limited | private | sensitive | legal_medical",
  "risk_tag_delta": ["tag codes to add/remove"],
  "notes": "short human-readable event summary"
}
```

工程上不应让 LLM 直接改关系边。LLM 或规则节点只能提出事件候选、风险候选和指标影响建议，再由确定性运行时根据 schema 校验和更新规则写入。

## 5. 实体解析层：Entity Resolution

实体解析负责把自然语言中的“他、客户、我妈的邻居、张总、那个供应商”锚定到已有或新建实体。

最小流程：

```text
提取候选实体
-> 匹配已有人物、组织、群体、合同、项目或平台
-> 判断是否需要新建 candidate entity
-> 记录别名、来源、置信度和待确认状态
-> 关联到 RawEvent 和 SemanticEvent
```

对“我妈的邻居”这类间接对象，不应丢弃。应通过关系路径模板建模为：

```text
user -> parent
parent -> neighbor
derived target -> situational_acquaintance 或 introduced_contact
```

实体解析的输出必须保留置信度。低置信度对象不应直接进入高风险建议，只能进入补信息或人工确认。

## 6. 关系边运行时实例

V3 的模板关系是 `relationship_types`，真实运行时需要 `relationship_edge_instance`。

关键字段包括：

| 字段 | 工程作用 |
| --- | --- |
| `edge_id` | 关系边唯一 ID。 |
| `subject_entity_id` | 通常是用户或用户代表的组织。 |
| `object_entity_id` | 目标人、组织、群体、合同、案件或平台。 |
| `primary_type_code` | 主关系类型。 |
| `secondary_type_codes` | 次关系类型。 |
| `risk_tag_codes` | 风险标签。 |
| `context_tag_codes` | 场景标签。 |
| `group_ids` | 所属群体、项目、合同、组织或案件。 |
| `relationship_path` | 介绍链、亲属链、平台链或间接关系路径。 |
| `roles` | 双方角色和角色置信度。 |
| `phase` | 生命周期阶段。 |
| `metrics` | 亲密、信任、依赖、权力、经济、法律、风险、边界等指标。 |
| `evidence` | 证据引用、证据置信度、最后验证时间。 |
| `time` | 生效、首次看到、最后互动、重大事件时间。 |
| `maintenance_policy` | 维护频率、下次复核时间、用户优先级。 |
| `audit` | 创建来源、创建时间、更新时间、schema 版本。 |

运行时关系边的职责是保存当前状态，不应直接保存所有原始对话。完整原始记录应在事件层和证据层保存，关系边只保存摘要、指标和引用。

## 7. 指标更新层

V3 的 `v3_event_driven_metric_update_rules` 给出事件驱动更新原则：

- 所有核心指标默认 clamp 到 `[0,1]`。
- 事件影响需要考虑时间衰减。
- 有证据权重：`effective_impact = impact_score * evidence_confidence * recency_factor`。
- 用户明确标记、法律文件、医疗/咨询/合同正式记录优先于系统推断。

典型公式包括：

```text
trust_new = clamp(trust_old
  + 0.06*promise_kept
  + 0.08*verified_support
  - 0.16*promise_broken
  - 0.25*privacy_violation
  - 0.35*betrayal
  - 0.60*violence_or_threat)
```

```text
boundary_clarity_new = clamp(boundary_clarity_old
  + 0.10*explicit_agreement
  + 0.08*role_clarification
  - 0.12*boundary_violation
  - 0.10*ambiguous_commitment)
```

工程实现建议：

1. 指标更新必须是纯函数，输入旧边、语义事件和更新规则，输出候选新边。
2. 高风险指标变化需要写审计记录。
3. 用户确认或正式证据可以覆盖模型推断。
4. 不要把低置信度事件直接用于重大关系类型转移。

## 8. 风险与人工确认层

V3 的复核门槛可整理为：

- `risk_level >= 0.70` 需要专家或人工复核。
- `legal_binding >= 0.70` 需要人工复核。
- 涉及未成年人、弱势对象、安全风险、法律医疗信息时需要人工确认。
- 涉及配偶、子女、监护、患者、治疗师、遗产、暴力相关边的关系类型转移需要人工确认。
- 安全风险优先于普通关系维护建议。

工程含义：

```text
如果触发高风险标签
-> 决策层只能输出安全、留痕、求助、停止或人工确认建议
-> 不得自动生成推进性强的话术
-> 不得自动执行外部平台动作
```

## 9. 上下文快照与记忆检索

系统不应每次都读取全部历史。应先按任务和对象构造 `ContextSnapshot`。

推荐检索顺序：

```text
任务识别
-> 目标对象匹配
-> ConversationThread 定位
-> RelationshipEdge 读取
-> 近期 RawEvent / SemanticEvent 摘要
-> 高权重事件和风险信号
-> 当前目标和限制
-> context_sufficiency_score
```

最小索引键：

| 索引 | 用途 |
| --- | --- |
| `person_id` | 找到目标对象所有关系和事件。 |
| `edge_id` | 找到某条关系边相关事件和指标历史。 |
| `thread_id` | 找到某个聊天、项目或客户跟进线程。 |
| `event_type` | 找到承诺、冲突、财务、边界、支持等事件。 |
| `risk_tag_code` | 找到风险相关事件。 |
| `context_tag_code` | 按工作、商业、法律、线上等场景过滤。 |
| `time_range` | 控制近期和长期记忆窗口。 |
| `evidence_confidence` | 优先读取高证据强度记录。 |

检索算法应服务于可解释决策。每次读取记忆都应能说明读取原因，例如：最近 30 天互动、未解决冲突、高风险标签、用户显式要求、证据不足复核。

## 10. 决策层：从目标到行动选项

决策层不直接读取全部原始数据，而读取脱敏摘要、关系状态、事件线索、风险约束和用户目标。

推荐流程：

```text
输入：SocialGoalInput + ContextSnapshot + RelationshipEdge Summary
-> 目标解析
-> 当前关系状态评估
-> 风险和边界检查
-> 目标状态设定
-> 事件路径推导
-> 行动选项生成
-> ROI 和风险比较
-> 输出 recommended_option.message_draft
```

输出至少包括：

- `agent_opinions`
- `deliberation`
- `ranked_options`
- `recommended_option`
- `message_draft`
- `skill_plan`
- `evidence_pack`
- `feedback_plan`
- `validation_plan`
- `context_sufficiency_score`

若上下文不足，决策层应输出补信息问题或停止条件，而不是硬给建议。

## 11. 触发与行动层

触发层把推荐转成可人工确认的执行计划。第一版只生成计划，不真实调用微信、日历或提醒连接器。

触发计划必须包含：

- 触发来源：用户主动、定时、职业例行、偏好例行、图谱信号。
- `message_draft`：具体草稿。
- `manual_execution_checklist`：目标对象、草稿、平台预览、确认闸门、行动后反馈。
- `confirmation_gates`：人工确认节点。
- `dry_run_preview`：平台预览。
- `feedback_plan`：执行后应收集什么反馈。

触发层应阻断：

- 高风险自动发送。
- 未确认收件人。
- 无草稿或草稿不含目标对象。
- 平台快照未证明发送被阻断。
- 用户未确认。

## 12. 反馈回写层

反馈是闭环的关键。执行后要记录：

- 用户是否执行。
- 对方是否回复。
- 回复内容和情绪。
- 关系是否推进或降温。
- 是否出现新风险。
- 使用者评分。
- 是否需要二次优化。

反馈可以生成：

- 新 `RawEvent`。
- 新 `SemanticEvent`。
- 关系指标变化。
- 风险标签增加或移除。
- 下一次触发计划。
- 决策策略校准数据。

没有反馈，系统只能做一次性建议；有反馈，系统才会变成可学习、可验证、可审计的闭环。

## 14. 推荐的实现顺序

### 第一阶段：稳定落盘契约

- 固定 `RawEvent`、`SemanticEvent`、`RelationshipEdge`、`ContextSnapshot` 的最小字段。
- 明确每个外部输入都先成为 `RawEvent`。
- 确保事件可追溯到原始证据。
- 建立人物、关系边、事件、风险和时间索引。

### 第二阶段：建立分类与检索闭环

- 实现实体锚定和关系边匹配。
- 实现风险标签和场景标签的候选提取。
- 实现 `context_sufficiency_score`。
- 决策层只读取快照和摘要，不直接扫全量原始记录。

### 第三阶段：接入指标更新和生命周期

- 把 V3 公式整理为可测试的纯函数。
- 对承诺、支持、冲突、边界、财务、法律等事件写测试样例。
- 高风险变化进入人工确认和审计。

### 第四阶段：接入对话动作层

- 定义 `DialogueAct`。
- 定义自然语言意图到事件和动作的映射。
- 让每个推荐选项都生成具体 `message_draft`。
- 在反馈后验证话术效果。

### 第五阶段：扩大真实样本

- 使用真实 `PilotImportBatch`。
- 运行 `pilot:validate`。
- 运行 `mvp:import`。
- 运行报告页、状态看板、完成度审计和流程树校验。

## 15. 工程结论

V3 图谱已经具备工程化基础，但不能直接等同于代码实现。落地关键是把图谱模板、运行时边、事件、指标、检索、决策、触发、反馈和审计串起来。

当前最优路线不是继续增加关系类型，而是先完成以下三项：

1. 每条信息可落盘：原始事件、语义事件、证据、人物、关系边、标签和置信度。
2. 每次建议可解释：读取哪些上下文、满足哪些目标、触发哪些风险、为什么生成这句草稿。
3. 每次执行可回写：执行结果能更新事件、指标、风险、关系阶段和下一次计划。
